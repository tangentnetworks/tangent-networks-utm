The Router

/var/www/htdocs/tn/cgi-bin/router.pl
The entry point for all CGI requests. Runs under taint mode. Sets a clean PATH and removes dangerous environment variables. Constructs the lib path from $RealBin and validates it against a strict character set before adding it to @INC. Contains one gate check: if DEVEL mode is active and the request arrived on a non-loopback address, it returns 503 and exits without routing the request. On a clean gate check it calls TNWAF::route_request() and exits. Router.pl has no pledge or unveil of its own. The sandbox belongs to the CGI process that TNWAF dispatches to.

Security Levels

Every CGI script calls TNSecurityCheck::security_check() as its first action after loading modules. Three levels are in use across the 24 scripts:

standard
Validates the session cookie if present, performs an origin check on state-changing methods. Does not require CSRF token or active session. Used for scripts that serve data to any authenticated browser session.
protected
Requires a valid session cookie and a CSRF token in the request body. Used for scripts that perform state changes on behalf of a logged-in user.
restricted
Requires a valid session cookie, a CSRF token, and admin role. Used for scripts that modify system configuration, queue privileged operations, or access sensitive data stores. Provides a full audit trail.

Pledge and Unveil Model

Every CGI script wraps its sandbox initialisation in eval { require OpenBSD::Pledge; 1 } and eval { require OpenBSD::Unveil; 1 }. Both are optional: if the modules are absent (non-OpenBSD systems), the script degrades gracefully rather than failing. The unveil lock call unveil() with no arguments is always present to seal the filesystem view once all paths have been declared.

All existence checks on optional paths are performed before the first unveil() call. Once any unveil is issued, the kernel begins restricting filesystem visibility. A -d check on an as-yet-unveiled path returns false even if the directory exists, so probing must happen while the filesystem is still fully visible.

All canonical paths that will be passed to unveil are resolved with File::Spec->rel2abs() before the sandbox is locked. After unveil, rel2abs() calls getcwd() internally, which requires rpath on directories that may not be unveiled. All path resolution therefore happens in a pre-pledge block.

Two pledge promise strings are used across all 24 scripts:

stdio rpath
Read-only scripts that serve data and write nothing. Used by: e2g_feeds.pl, e2g_read_log.pl, services.pl.
stdio rpath wpath cpath flock
All other 21 scripts. Required when a script writes queue trigger files, appends to log files, or performs atomic temp-to-rename rewrites.

firewall.pl is the only script with rx permission on /bin and /usr/bin in its unveil configuration. It requires execute access to read pf log data via external programs.

The universal unveil paths present on every script are:

data/lib (r)
Perl module source files.
data/config (r)
Reads security.conf.
data/keys (r)
Session and HMAC key files. Present on most scripts; absent only where key operations are not needed.
Perl runtime paths (r)
/usr/lib/perl5, /usr/libdata/perl5, /usr/local/lib/perl5, /usr/local/libdata/perl5, /usr/local/lib, /usr/lib. Each is probed for existence before unveiling. The layout varies by OpenBSD version and chroot setup. Only directories that exist are unveiled.

XS Module Pre-loading

Scripts that use DBD::SQLite load it with a top-level use DBD::SQLite statement before the pledge block runs. DBI loads its driver lazily on the first connect() call. If that call happens after pledge is active, dlopen() requires prot_exec, which is not granted. Pre-loading ensures the shared object is mapped before the sandbox is locked down.

STDIN Pre-read Pattern

Scripts at the protected and restricted security levels that accept JSON POST bodies include a pre-read block in their BEGIN:

perl 
if ($ENV{CONTENT_TYPE} && $ENV{CONTENT_TYPE} =~ /application\/json/ && $ENV{CONTENT_LENGTH}) {
		    read(STDIN, my $json_body, $ENV{CONTENT_LENGTH});
		    $ENV{POSTDATA} = $json_body;
		}

TNSecurityCheck::security_check() creates a CGI->new object internally to read the CSRF token, which drains STDIN. If the JSON body is not pre-read into $ENV{POSTDATA} before that call, it is lost and the script receives an empty request body. This pattern is referenced in comments as STDIN-PREREAD-001.

Script Reference

control.pl

Authentication and session controller. Handles user registration, login, logout, password reset, CSRF token issuance, session validation, DEVEL mode management, and user deletion. This is the only script with a multi-stage password reset flow using HMAC-signed, single-use reset tokens with a 10-minute TTL.

Security level: routes dynamically by path. Auth and API routes use standard or no session requirement. DEVEL and admin routes enforce session and CSRF. The handle_user_delete handler additionally checks that the session role is explicitly admin after require_session('admin') returns.

Pledge: stdio rpath wpath cpath flock. Unveil includes data/db (rwc), /tmp (rwc), /dev/urandom (r), and the standard lib and config paths. Optional paths including data/keys and data/logs are probed for existence before unveiling. DBD::SQLite is pre-loaded before the sandbox block.

Log: /tmp/control-YYYY-MM-DD.log inside the chroot (host path: /var/www/tmp/control-YYYY-MM-DD.log). Date-stamped per day. Multiple slowcgi worker processes share the file; the BEGIN block uses flock(LOCK_EX) before duping to STDERR to prevent interleaved startup writes.

logs.pl

Serves log file content and archive availability data for the Logs view. Covers 25 log sources across system, security, proxy, mail, and web service categories. Two actions are available via the action query parameter: fetch returns log lines from a source, and available_dates returns the archive date inventory for a source with gap analysis.

The 25 configured log sources are: system/messages, system/daemon, bootlog/rc.local.log, doas/doas.log, dhcpd/dhcpd.log, unbound/unbound.log, rad/rad.log, collectd/collectd.log, sslproxy/sslproxy.log, sslproxy/sslproxy_connect.log, sockd/sockd.log, p3scan/p3scan.log, smtp-gated/smtp-gated.log, spamd/spamd.log, snort/alert.log, snort/snort.log, snort/snortinline.log, snortsentry/snortsentry.log, e2guardian/access.log, e2guardian/e2guardian.log, httpd/httpd_access.log, httpd/httpd_error.log, imspector/imspector.log, and ftp-proxy/ftp-proxy.log. Each source defines a live path, an archive pattern using a {date} placeholder, a type (live or boot), a line limit, and an optional flowbits filter flag.

For fetch requests, the date parameter accepts live for the current log file or a signed integer day offset (for example -1 for yesterday). Live logs use tail to return the last N lines; archive logs use head because archives are written in reverse chronological order during rotation. Boot logs have no line limit and read the entire file. Line counts for metadata are obtained via wc -l. The path is canonicalised before any file operation and checked against the log base directory to reject path traversal attempts.

Server-side line filtering is applied before the response is built, reducing bandwidth to mobile clients. The filter parameter accepts: all, error, warning, auth, blocked, or connection. Three log timestamp formats are recognised and extracted: bracketed ISO format ([YYYY-MM-DD HH:MM:SS]), standard syslog (Mon DD HH:MM:SS), and ISO 8601 with timezone offset. Each line is assigned a level of info, warning, or error based on keyword matching.

Sources with filter_flowbits enabled (system/messages, system/daemon, snort/snort.log, snort/snortinline.log) strip Snort flowbit noise lines before processing. This removes both flowbits key lines and syslog entries from Snort processes that leak into the system message log.

The available_dates action scans the last seven days for archive files and returns both available dates with file sizes and gaps with classified reasons. Gap reasons are: future_date, system_offline (rotation did not occur, system was likely powered off), empty_not_rotated (service was running but produced no output), or missing_archive (file should exist but does not). Consecutive gaps sharing the same reason are consolidated into date ranges in the response. Rotation state is read from a metadata file at data/logs/.rotation_meta with a fallback to the rotation stamp at data/logs/bootlog/.rotation_stamp.

When a requested file does not exist, a structured error response is returned with a typed error_type field and diagnostic details rather than a generic failure, giving the UI enough context to display a meaningful message.

Security level: standard. Pledge: stdio rpath wpath cpath flock. Unveil covers the log base directory tree and standard lib, config, and key paths. Log source names are validated against the configuration inventory before any file path is constructed; unknown source names are rejected immediately.

Queue Architecture

Several scripts use a synchronous queue pattern to execute privileged operations without running as root inside the chroot. The CGI script writes a request file, then polls an outcome directory until the root daemon writes a result or the timeout expires. This keeps the CGI layer unprivileged while allowing it to trigger operations that require root.

The queue directories and their associated scripts are:

data/queue/integrity/
Used by integrity_request.pl and integrity_status.pl. Request files contain action and check name. Outcome files contain the raw JSON output from TNAudit.pl. The integrity runner processes requests within one second. Maximum CGI poll wait: 120 seconds.
data/queue/powerstate/
Used by power_mgmt.pl. Request files contain a shell command string. Maximum CGI poll wait: 30 seconds. Timeout is treated as likely success for shutdown and restart.
data/services/queue/
Used by manage_services.pl. Request files contain action and service name. Outcome files are matched by job ID prefix. Maximum poll wait varies by service (15 to 120 seconds).
data/services/queue/unbound/
Used by unbound_control.pl. Request files are JSON. Maximum poll wait: 10 seconds at 0.2-second intervals.
data/services/queue/e2gfilters/
Used by e2g_mode_switch.pl (request and outcome subdirectories) and e2g_write_input.pl (userlist subdirectory). Mode switch polls for up to 25 minutes to accommodate full filter rebuild time.
data/services/queue/pf-rules/
Used by pf_write_input.pl, pf_delete_input.pl, pf_read_input.pl, and pf_trigger.pl. The user-input subdirectory holds the persisted queue files. The triggers subdirectory holds signal files for the validation daemon.

Queue directories that must exist before the pledge block is applied are created at script startup if absent. After pledge and unveil are active the script cannot create directories outside the unveiled paths, so this initialisation must happen in the pre-pledge section.

All queue outcome files are consumed (read and deleted) by the CGI script that polls for them. If a CGI request times out before the daemon writes the outcome, the outcome file will remain in the directory. The integrity runner cleans outcome files older than five minutes. Other queue processors have their own cleanup intervals. Do not manually delete active request files while a CGI is polling for their outcome.

Summary

  • All 24 CGI scripts are reachable only through router.pl via TNWAF. Direct path access is not possible. httpd hard-blocks secrets directories at the server level before any CGI runs.
  • Three security levels are in use: standard (origin check), protected (session and CSRF), restricted (admin session and CSRF with audit trail). Every script calls security_check() before any business logic.
  • All scripts run under Perl taint mode and apply OpenBSD pledge and unveil sandboxing. Both are optional and degrade gracefully on non-OpenBSD systems. Two pledge strings are used: stdio rpath for read-only scripts, stdio rpath wpath cpath flock for all others. firewall.pl is the only script with rx permission on system binary directories.
  • Scripts that use DBD::SQLite load it before the pledge block to prevent dlopen failure after the sandbox is active. Scripts that accept JSON POST bodies at protected or restricted levels pre-read STDIN into $ENV{POSTDATA} in the BEGIN block before TNSecurityCheck drains it.
  • All canonical paths passed to unveil are resolved with File::Spec->rel2abs() before the first unveil call. All filesystem existence checks on optional paths are performed before the first unveil call.
  • Several scripts use a synchronous queue pattern: the CGI writes a request file, a privileged root daemon processes it, and the CGI polls an outcome directory. This keeps all CGI processes unprivileged while allowing them to trigger root-level operations.
