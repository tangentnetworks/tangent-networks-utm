Foreword

TNSecurity is the cryptographic and security utility layer. It owns password hashing, session ID signing, CSRF token generation and validation, SRI hash computation, HMAC operations, key management, and security event logging. Every other module calls into TNSecurity for these operations. This document covers each subsystem, the algorithm choices, and the file paths involved.

Password Hashing

Passwords are hashed with PBKDF2-HMAC-SHA256 via the Crypt::PBKDF2 module (p5-Crypt-PBKDF2-0.161520). The algorithm parameters are fixed constants in the module:

  • Hash function: HMAC-SHA256
  • Iterations: 100,000
  • Output length: 32 bytes (one SHA-256 block)
  • Salt: 32-character alphanumeric string from /dev/urandom

The Crypt::PBKDF2 object is constructed once at module load time and reused on every call. The derived key is stored in hex encoding, prefixed with pbkdf2: so the verification path can identify the algorithm without a separate metadata column.

perl 
# Hash a password (returns hash string and salt)
my ($hash, $salt) = TNSecurity::hash_password($plaintext);
# $hash: "pbkdf2:a3f8c2..."
# $salt: "rXk2mNpQ..."

# Verify
if (TNSecurity::verify_password($plaintext, $hash, $salt)) { ... }

Legacy Algorithm

Prior to the PBKDF2 migration, passwords were stored as the result of 10,000 iterations of SHA-256: sha256_hex(sha256_hex(... password . salt ...)). These hashes are plain 64-character hex strings with no prefix.

verify_password() detects the legacy format by checking for the absence of the pbkdf2: prefix and uses the legacy verification path. needs_rehash() returns true for any hash without the prefix. TNAuth calls needs_rehash() on every successful login and immediately re-hashes the password with PBKDF2 while the plaintext is available, upgrading the stored hash transparently without any user interaction.

Recovery Code Hashing

Recovery codes use a simpler scheme: sha256_hex($code) with no salt. No salt is needed because recovery codes are long random strings that are not susceptible to dictionary attack. The contract is:

  • hash_recovery_code($code) returns sha256_hex($code).
  • verify_recovery_hash($code, $stored_hash) computes the hash and compares with timing_safe_compare().

Session ID Signing

Raw session IDs are 64-character hex strings stored in the database. Before being set as a cookie, the session ID is signed with HMAC-SHA256 using the session key:

text 
cookie_value = session_id + "." + hmac_sha256_hex(session_id, session.key)

On every incoming request, verify_session_id() splits the cookie value on the first dot, recomputes the expected signature, and compares with timing_safe_compare(). Only the raw session ID (before the dot) is used for the database lookup. A tampered or forged cookie value fails signature verification and returns undef before any database access occurs.

The session key file: data/keys/session.key. A missing key file is always a fatal error -- the process dies with a message directing the operator to run init_db.pl. Silent auto-generation is intentionally not implemented: generating a new key would invalidate all existing sessions without any operator alert.

CSRF Tokens

CSRF tokens are stateless. There is no database table. A token is an HMAC-SHA256 of a time-window string using the HMAC key:

text 
token = hmac_sha256_hex("csrf:" + floor(time / TOKEN_LIFETIME), hmac.key)

Validation accepts the current window and the previous window:

perl 
return 1 if timing_safe_compare($token, hmac_sha256_hex("csrf:$window",   $key));
return 1 if timing_safe_compare($token, hmac_sha256_hex("csrf:$window-1", $key));

Accepting the previous window handles the case where the browser fetched a token just before a boundary and submitted the form just after it. TOKEN_LIFETIME is read from security.conf [csrf], floored at 60 seconds. The default is 3600 seconds (1 hour).

The HMAC key file: data/keys/hmac.key. Same fatal-on-missing behaviour as the session key.

In DEVEL mode, generate_csrf_token() returns a token derived from the string 'devel' as the key rather than the real HMAC key, and validate_csrf_token() returns 1 unconditionally.

Reset Tokens

Password reset tokens are short-lived proof that a user completed the identity challenge (security questions or recovery code). They are generated and verified by control.pl, not by TNSecurity directly, but they use TNSecurity::hmac_hex() internally.

The token format is:

text 
reset_token = base64("username:expiry") + "." + hmac_sha256_hex(payload + ":reset", hmac.key)

The :reset purpose tag in the HMAC input prevents a reset token from being replayed as a session signature and vice versa, even though both use the same HMAC key file. Reset tokens expire after 600 seconds (10 minutes).

hmac_hex($data, $key_name) maps key names to files: 'session' maps to session.key, any other name maps to hmac.key.

Key Management

Two key files are used at runtime:

data/keys/session.key
Used for HMAC-SHA256 session ID signing.
data/keys/hmac.key
Used for CSRF tokens and reset tokens.

Both files contain a 64-character alphanumeric string generated by generate_random_string(64) from /dev/urandom. Permissions are 0600, owned www:www. They are created by init_db.pl at install time via TNSecurity::generate_key().

read_key() untaints the file path, verifies the file exists, and reads the key. A missing key file is a fatal error that crashes the process immediately with a message directing the operator to run init_db.pl. This is deliberate: a missing key in production means something has gone wrong and the operator must be notified. Silent auto-generation would invalidate all existing sessions without any alert.

Do not rotate the session key or HMAC key while users have active sessions. Rotating the session key invalidates all signed cookies immediately -- every logged-in user is forced to re-authenticate. Rotating the HMAC key invalidates all CSRF tokens, which causes every in-flight form submission to fail until the user fetches a new token.

SRI Hash Generation

generate_sri_hash($file_path) reads a file, computes SHA-384, base64-encodes the result, and returns an integrity= attribute value:

text 
sha384-<base64-padded-hash>

update_all_sri_hashes() iterates over the known JS and CSS asset files and generates hashes for each. It returns a count and list of processed files. In DEVEL mode both functions return empty string and 1 respectively, skipping all I/O.

The SRI hash table in TNWAF is the authoritative store of known-good hashes. The table is populated by running the POST /devel/regenerate_sri endpoint after any change to the asset files. See the TNWAF document for the server-side tamper detection that uses this table.

Input Validation

validate_username($username)
Returns true if the username matches /^[a-zA-Z0-9_-]{3,32}$/.
validate_email($email)
Returns true if the email matches a basic RFC 5322 pattern.
sanitize_input($input)
Strips control characters (\x00-\x1F\x7F). Returns empty string for undef input.
untaint_filepath($path)
Rejects paths containing .., then allows only [-/\w.]. Returns undef on failure.

Random Generation

All random values come from /dev/urandom, read in binary mode. generate_random_string($length) reads $length raw bytes and maps each byte modulo 62 to the alphanumeric character set. generate_token($length) calls generate_random_string and then takes sha256_hex of the result concatenated with the current time, returning a 64-character hex string.

Security Event Logging

log_security_event($level, $event, $details) formats a log line and writes it to data/logs/security.log if LOG_TO_FILE = 1 in the config. Each line includes a timestamp, level, client IP from $ENV{REMOTE_ADDR}, the event name, and details.

text 
[Mon Mar  9 14:22:01 2026] [warning] [10.10.10.42] ACCOUNT_LOCKED - alice locked after 5 failed attempts

File writes use flock(LOCK_EX) to prevent interleaved lines from concurrent slowcgi workers. The log directory is created if it does not exist.

Timing-Safe Comparison

timing_safe_compare($a, $b) compares two strings in constant time by XOR-ing each byte pair and ORing the results. It returns true only if all bytes match and both strings are the same length. Used for all HMAC comparisons, password verification, and CSRF validation. Never use eq for cryptographic comparisons.

HTTP Security Headers

print_secure_headers($content_type, $cache) emits the standard security header set for CGI JSON API responses. It is the single point of truth for API response headers; TNWAF's print_headers() covers routed file responses separately.

Headers emitted:

Content-Security-Policy
default-src 'none'; connect-src 'self' -- maximum restriction for JSON API endpoints that serve no HTML, CSS, or JS.
Strict-Transport-Security
max-age=31536000; includeSubDomains -- effective only over HTTPS; harmless over HTTP.
Referrer-Policy
strict-origin-when-cross-origin
Permissions-Policy
Denies geolocation, microphone, camera, and payment APIs.
X-Frame-Options
DENY -- belt-and-suspenders alongside CSP frame-ancestors 'none'.
X-Content-Type-Options
nosniff
Cache-Control
no-cache, no-store, must-revalidate unless $cache is true.

Summary

  • Passwords use PBKDF2-HMAC-SHA256, 100,000 iterations, 32-byte output, prefixed pbkdf2:. Legacy SHA-256 hashes are detected by the absent prefix and upgraded transparently on next login.
  • Session IDs are HMAC-signed before being set as cookies. A missing or tampered signature returns undef before any database lookup. A missing key file is always fatal -- no silent key generation.
  • CSRF tokens are stateless. No database table. Tokens are HMAC-SHA256 of a time-window string. Both the current and previous window are accepted to handle boundary edge cases.
  • Reset tokens embed username and expiry, signed with HMAC using a :reset purpose tag to prevent cross-use with session signatures.
  • All cryptographic comparisons use timing_safe_compare(). Never use eq for HMAC or hash comparisons.
  • Security log writes are flock-protected. The log file is never truncated by the application.
