Foreword

This document covers the content inspection stack -- the chain of daemons that PF diverts traffic through before forwarding it to its destination. It covers SSLproxy, e2guardian, Snort IDS and IPS, snortsentry, p3scan, smtp-gated, imspector, spamd, and the SOCKS proxy. It also covers the e2guardian filter management system: the three filter profiles, the feed download scripts, the queue processor, and the WebUI status pipeline.

The firewall document covers the PF divert rules that feed traffic into this stack. This document covers what happens after diversion. Read the firewall document first if you have not already.

Compilation details and custom patches for the programs in this stack are documented separately. That documentation is not covered here.

Content Filtering

Inspection Stack Overview

The inspection stack is a chain of user-space daemons connected to the traffic stream through PF divert rules. No traffic leaves the machine uninspected. The chain for HTTP and HTTPS traffic is the deepest -- SSLproxy decrypts TLS, e2guardian filters content, and Snort IPS performs deep packet inspection on the decrypted stream. Other protocols have shallower chains appropriate to their threat profile.

SSLproxy
Transparent TLS interception proxy. Decrypts HTTPS connections using a locally installed CA, forwards decrypted content to e2guardian, re-encrypts outbound. Also handles POP3, POP3S, SMTP, SMTPS, and submission protocol proxying.
e2guardian
Content filter. Receives decrypted HTTP traffic from SSLproxy on port 8080. Applies phrase matching, domain blacklists, URL blacklists, MIME type filtering, and ClamAV virus scanning. Three switchable filter profiles.
Snort IDS
Network intrusion detection. Monitors the LAN interface (%%INT_IF%%) passively. Writes alerts in fast format. Does not block traffic.
Snort IPS
Network intrusion prevention. Runs inline via the PF divert-packet mechanism on loopback ports 8080, 8110, and 9199 -- the post-decryption paths from SSLproxy. Receives decrypted content and can drop packets.
snortsentry
Tangent-authored Perl daemon. Watches the Snort IDS alert file and adds offending source IPs to the PF <snort_block> table. Supports IPv4 and IPv6. Escalating block durations.
p3scan
Transparent POP3 and POP3S proxy. Scans mail attachments with ClamAV before delivery to the mail client.
smtp-gated
Transparent SMTP proxy. Scans outbound mail with ClamAV and SpamAssassin before forwarding to the destination mail server.
imspector
Transparent instant messaging proxy. Logs and optionally filters conversations on MSN, AIM, Yahoo, and IRC protocols.
spamd
OpenBSD spamd greylisting daemon. Delays connections from unknown senders to reduce spam volume reaching smtp-gated.
Dante sockd
SOCKS5 proxy. Provides LAN clients with a proxied outbound connection path for applications that support SOCKS.

SSLproxy

SSLproxy performs transparent TLS interception. LAN clients connect to external HTTPS destinations normally. PF intercepts the connection and diverts it to SSLproxy, which presents a forged certificate signed by the local CA. SSLproxy then opens its own TLS connection to the real destination, giving it access to the plaintext stream in both directions.

SSLproxy runs as _sslproxy under the PF user isolation rules. The CA certificate and key are at /usr/local/etc/sslproxy/ca.crt and /usr/local/etc/sslproxy/ca.key. LAN clients must have the CA certificate installed as a trusted root for HTTPS inspection to work transparently -- without it, browsers will show certificate warnings for every HTTPS site.

ProxySpecs

Each ProxySpec defines a listening address and port, and where to forward the decrypted stream. The up: parameter is the upstream port where the next inspection daemon listens.

conf 

### 1. Web Inspection Layer

 http-v4-8081  -> ProxySpec http     127.0.0.1 8081 up:8080
 http-v6-8081  -> ProxySpec http     ::1       8081 up:8080
 https-v4-8443 -> ProxySpec https    127.0.0.1 8443 up:8080
 https-v6-8443 -> ProxySpec https    ::1       8443 up:8080

### 2. POP3 / POP3S Mail Retrieval Layer

 pop3-v4-8994  -> ProxySpec pop3     127.0.0.1 8994 up:8110
 pop3-v6-8994  -> ProxySpec pop3     ::1       8994 up:8110
 pop3s-v4-8995 -> ProxySpec pop3s    127.0.0.1 8995 up:8110
 pop3s-v6-8995 -> ProxySpec pop3s    ::1       8995 up:8110
 pop3s-v4-8993 -> ProxySpec pop3s    127.0.0.1 8993 up:8110
 pop3s-v6-8993 -> ProxySpec pop3s    ::1       8993 up:8110

### 3. SMTP / SMTPS Mail Submission Layer

 smtp-v4-8464  -> ProxySpec smtp     127.0.0.1 8464 up:9199
 smtp-v6-8464  -> ProxySpec smtp     ::1       8464 up:9199
 smtps-v4-8465 -> ProxySpec smtps    127.0.0.1 8465 up:9199
 smtps-v6-8465 -> ProxySpec smtps    ::1       8465 up:9199

### 4. Dynamic AutoSSL Engine Layer

 autossl-v4-8466 -> ProxySpec autossl 127.0.0.1 8466 up:9199
 autossl-v6-8466 -> ProxySpec autossl ::1       8466 up:9199

HTTP and HTTPS both forward to port 8080 where e2guardian listens. POP3 and POP3S forward to port 8110, SMTP variants to port 9199 -- these are the Snort IPS divert ports for non-web protocols. SSLproxy handles the protocol-level wrapping; the upstream daemon sees plaintext in all cases.

e2guardian

e2guardian is the content filter. It receives decrypted HTTP traffic from SSLproxy on loopback port 8080. It does not perform TLS -- SSLproxy handles all encryption and decryption. enablessl = off and sslmitm = off in the e2guardian configuration are correct and intentional: e2guardian sees plaintext because SSLproxy has already unwrapped it.

e2guardian runs as _e2guardian under PF user isolation rules, which prevents it from opening direct connections to LAN or WAN addresses. All its network communication happens through loopback, through SSLproxy.

Main Configuration

filterip = 127.0.0.1 / filterports = 8080
e2guardian listens on loopback port 8080. Only SSLproxy sends traffic here -- no direct client connections reach this port.
filtergroups = 1
Single filter group. All LAN clients are assigned to the same group and subject to the same active filter profile. Group assignment is via /etc/e2guardian/lists/filtergroupslist.
httpworkers = 50
Maximum concurrent HTTP worker threads. Sufficient for a home or small office deployment. Increase if e2guardian logs worker exhaustion under heavy concurrent load.
weightedphrasemode = 2 / phrasefiltermode = 2
Full phrase weighted scanning on both URL and body content. Mode 2 scans both the URL and the page body. The naughtiness score from phrase matching is compared against the naughtynesslimit in the active filter profile (175 for all three profiles).
maxcontentfiltersize = 2048
Maximum content body size in KB that e2guardian will filter. Content larger than 2 MB is passed without filtering. Increase cautiously -- larger values increase memory usage per worker thread.
contentscanner = clamdscan.conf
ClamAV daemon scanner. e2guardian passes downloaded content to clamd for virus scanning. The clamd socket is at /var/www/htdocs/tn/data/tmp/clamd.socket.
authplugin = ip.conf
IP-based client identification. e2guardian identifies clients by source IP address for logging and group assignment purposes. No password authentication.
loglevel = 3
Logs all requests including exceptions. The access log at /var/www/htdocs/tn/data/logs/e2guardian/access.log receives one line per request. The daemon log at /var/www/htdocs/tn/data/logs/e2guardian/e2guardian.log receives operational messages.

e2guardian Filter Profiles

Three filter profile configuration files exist in /etc/e2guardian/. Only one is active at a time. The active profile is e2guardianf1.conf -- the file e2guardian actually loads. Each filter management script copies its profile template over this file when activated, then reloads e2guardian.

The template file extensions used here (such as e2guardianf1.conf.a, *.c, and *.u) are intentionally chosen configuration suffixes. Be aware that some graphical GUI text editors may misinterpret these extensions as source code language files (like C or Unshared source) and apply incorrect syntax highlighting. Tangent Networks is aware of this interface behavior, and a standardized extension schema will be implemented in the next release to resolve this confusion.

e2guardianf1.conf.c -- Childsafe
Strict filtering. Blacklists point to lists/blacklists/childsafe/domains and lists/blacklists/childsafe/urls. Includes the childsafe phrase lists for bad words, profanity, and spam phrases. Blocks adult content, malware domains, ad networks, and general threat categories. Updated by e2g_childsafe_filter.sh.
e2guardianf1.conf.a -- Adult (General)
Permissive profile. Blacklists point to lists/blacklists/adult/domains and lists/blacklists/adult/urls. Blocks malware domains and ad networks but does not include porn filtering categories. Updated by e2g_adult_filter.sh.
e2guardianf1.conf.u -- User (Custom)
User-defined profile. Blacklists point to lists/blacklists/webuser/domains and lists/blacklists/webuser/urls. Feed sources are defined through the WebUI and written to /var/www/htdocs/tn/data/services/queue/e2gfilters/userlist/userfeeds.txt. Updated by e2g_user_filter.sh.

All three profiles share the same storyboard (e2guardianf1.story), the same phrase list base, the same exception and grey list paths, and the same MIME and file extension lists. The only difference between profiles is which blacklist directory the banned site and URL lists point to, and which additional phrase lists are loaded.

Only one filter script should be in crontab at a time. The filter scripts self-manage the crontab entry -- when a filter is activated via the WebUI queue processor, it removes any existing e2g_*_filter.sh entry and adds its own. The crontab at any given time reflects which profile is scheduled for daily feed updates. The current active profile is also written to the status JSON file read by the WebUI dashboard.

Feed Management Scripts

Each filter profile has a corresponding feed management script. All three follow the same structure: download configured feeds into an isolated work directory, extract and deduplicate domains and URLs, write the final lists to the e2guardian blacklist directory, copy the profile template over e2guardianf1.conf, and reload e2guardian. The adult, childsafe, and user scripts all support the Capitole archive inline -- no shared library is used. Capitole category routing is determined at runtime by scanning each category's usage file with find and grep: categories whose usage starts with black are merged into the e2guardian blacklist; categories whose usage starts with white are appended to /etc/e2guardian/lists/localexceptionsitelist and deduplicated. The user filter supports Capitole if a CAPITOLE: entry is present in userfeeds.txt; without one it processes only the configured feed URLs.

Feed source categories

Feed sources are declared in a per-profile configuration file using category prefixes. The childsafe and adult scripts read from /etc/e2guardian/feeds/. The user script reads from the WebUI-managed userfeeds.txt.

conf 
HOSTFILE:  https://example.com/hosts.txt       # hosts-format blocklist
DOMAIN:    https://example.com/domains.txt     # plain domain list
PORN:      https://example.com/porn.txt        # adult domain list (childsafe only)
MIXED:     https://example.com/mixed.txt       # mixed domains and IPs
ADVERTISER: https://example.com/ads.txt        # ad network hosts format
CAPITOLE:  https://example.com/blacklists.tar.gz  # Capitole-Toulouse archive

The Capitole archive from Toulouse University is a comprehensive categorised blacklist/allowlist. Each category directory contains a usage file whose first word is either black or white. The scripts discover categories dynamically at runtime using find ... -name usage -exec grep -l -- no category names are hardcoded, so archive layout changes are handled automatically. Categories whose usage starts with black are merged into the e2guardian blacklist; categories whose usage starts with white are appended to /etc/e2guardian/lists/localexceptionsitelist and sorted deduplicated. Categories with any other usage value are skipped and logged. This prevents legitimate domains in child-safe allowlist categories (such as child, educational, radio) from being incorrectly added to the blocklist. Other feed types are processed by extraction method -- hostfile format extracts the last field of each line, URL format extracts the hostname from each URL, mixed format filters out IP addresses and keeps domain entries.

Work directories

Each filter type has an isolated work directory under /etc/feeds/:

  ads/                   # downloaded advertiser feeds
  mix/                   # downloaded mixed list feeds
  porn/                  # downloaded porn domain feeds (childsafe)
  new_feeds/             # staging area before merge

  blockdomains           # final merged domain list
  blacklists/            # extracted Capitole archive (per-run, overwritten each update)

/etc/e2guardian/feeds/
  general.txt            # feed source URLs for adult/general filter
  childsafe.txt          # feed source URLs for childsafe filter
  # (user filter reads from WebUI-managed userfeeds.txt -- see above)

/etc/e2guardian/lists/

Reload behaviour

After updating the blacklist files and copying the profile template, the scripts call e2guardian -r to trigger a configuration reload. e2guardian keeps the old configuration in memory if the reload fails -- there is no service interruption on a bad config. If e2guardian is not running when the script fires, it starts it instead of reloading.

sh 
# Check which filter is currently active
cat /var/www/htdocs/tn/data/services/queue/e2gfilters/status/active_mode.json

# Run feed update manually (as root)
/usr/local/sbin/e2g_childsafe_filter.sh
/usr/local/sbin/e2g_adult_filter.sh
/usr/local/sbin/e2g_user_filter.sh

# Check last feed update log
ls -lt /var/www/htdocs/tn/data/logs/cron/ | grep e2g
tail -50 /var/www/htdocs/tn/data/logs/cron/e2guardian-childsafe-*.log

# Count current blacklist entries
wc -l /etc/e2guardian/lists/blacklists/childsafe/domains
wc -l /etc/e2guardian/lists/blacklists/childsafe/urls

WebUI Filter Management Pipeline

Four daemons handle the bridge between the WebUI and the e2guardian filter system. They run continuously from rc.local and communicate through a queue directory structure.

e2g_status_writer.sh via e2g_status_writer_runner.sh
Polls /etc/crontab every 2 seconds. Determines which filter script is currently scheduled, parses its cron time, and writes a status JSON to /var/www/htdocs/tn/data/services/queue/e2gfilters/status/active_mode.json. The WebUI filter panel reads this file to display the current mode and next scheduled update time. Modes reported: general (adult), childsafe, custom (user), or none.
e2g_queue_processor.sh via e2g_queue_processor_runner.sh
Processes filter switch requests from the WebUI. Reads request files from /var/www/htdocs/tn/data/services/queue/e2gfilters/request/, validates the requested mode, runs the appropriate filter script, and writes the outcome to /var/www/htdocs/tn/data/services/queue/e2gfilters/outcome/. The queue processor runner uses exec rather than a loop -- it starts the processor once and exits when it completes, relying on rc.local to restart it if needed.
e2g_user_filter_detector.sh via e2g_user_filter_runner.sh
Polls userfeeds.txt every 2 seconds using a checksum comparison. When the file changes (WebUI adds or removes a user feed URL), the detector launches e2g_user_filter.sh in the background and updates the stored checksum. This gives near-immediate feed list updates without waiting for the daily cron run.
sh 
# Check all four pipeline daemons are running
pgrep -f e2g_status_writer_runner
pgrep -f e2g_queue_processor_runner
pgrep -f e2g_user_filter_runner

# Check PID files
ls /var/www/htdocs/tn/data/run/webui/e2g_*.pid

# View status JSON
cat /var/www/htdocs/tn/data/services/queue/e2gfilters/status/active_mode.json

# View detector log
tail -20 /var/www/htdocs/tn/data/services/queue/e2gfilters/outcome/detector.log

Snort IDS and IPS

Two Snort instances run simultaneously serving different roles. Both are started from rc.local and log through OpenBSD syslog using the logger utility -- Snort's stderr output is piped to logger with distinct tags, and syslog routes each tag to a separate file.

IDS instance -- passive monitoring
Runs on %%INT_IF%% with snort -i %%INT_IF%% -d -c /etc/snort/snort.conf. Monitors all LAN traffic passively. Does not touch packets. Writes alerts in fast format to /var/www/htdocs/tn/data/logs/snort/alert.log (via output alert_fast: alert.log in snort.conf). Also writes binary capture logs. Syslog tag: snort_ids, log file: /var/www/htdocs/tn/data/logs/snort/snort.log. PID file: /var/www/htdocs/tn/data/run/snort/snort_%%INT_IF%%.pid.
IPS instance -- inline prevention
Runs with snort -d -Q -c /etc/snort/snortinline.conf. Uses the ipfw DAQ in inline mode on divert port 700. Receives decrypted traffic from the SSLproxy loopback chain. Can drop packets. No interface flag -- the divert port in snortinline.conf provides the traffic source. Syslog tag: snort_ips, log file: /var/www/htdocs/tn/data/logs/snort/snortinline.log. PID file: /var/www/htdocs/tn/data/run/snort/snort_.pid.

Syslog routing for Snort. OpenBSD syslog is intentionally minimal and does not support per-program log routing without explicit facility/tag configuration. Both Snort instances pipe stderr through logger -- IDS as logger -t snort_ids -p daemon.info, IPS as logger -t snort_ips -p daemon.info. The syslog configuration routes these tags to their respective log files. snortsentry gets its own facility tag snortsentry routed to /var/www/htdocs/tn/data/logs/snortsentry/snortsentry.log. This is the standard approach for multi-instance daemons on OpenBSD.

IDS configuration highlights

/etc/snort/snort.conf uses the ac-split search method with optimisation enabled. The DAQ directory is set to /usr/local/lib/daq/. Output goes to both syslog (output alert_syslog: LOG_DAEMON LOG_ALERT LOG_AUTH) and the fast alert file. The null log output (output log_null) suppresses packet logging to disk -- alerts only. Rules are loaded from /etc/snort/rules/.

IPS configuration highlights

/etc/snort/snortinline.conf adds the inline-specific directives to the base configuration. The DAQ is set to ipfw in inline mode on divert port 700 (config daq: ipfw, config daq_mode: inline, config daq_var: port=700). Policy mode is inline (config policy_mode: inline) and snaplen is set to 65535 to capture full packets. This instance sees only the traffic that PF has diverted to loopback ports 8080, 8110, and 9199 -- decrypted HTTP, POP3, and SMTP streams.

snortsentry

snortsentry is a Tangent-authored Perl daemon that bridges the Snort IDS alert file and the PF firewall. It watches /var/www/htdocs/tn/data/logs/snort/alert.log for new alerts and adds the source IP addresses of offending hosts to the PF <snort_block> table. It was written specifically to handle IPv6 addresses, which tools available at the time rejected silently.

Key configuration

CheckInterval 1 / BatchInterval 5
Checks the alert file every second. Applies pending blocks to PF every 5 seconds. The batch interval prevents excessive pfctl calls under sustained attack traffic.
Priority 1
Only blocks alerts at Snort priority 1 (highest severity). Lower priority alerts are logged but do not result in PF blocks. Adjust upward to reduce false-positive blocks.
BlockDuration 60 / MaxBlockDuration 86400
Initial block is 60 seconds. If the same IP triggers further alerts, the block escalates through the tiers below up to a maximum of 24 hours.
EscalationTiers 60, 300, 1800, 3600
Block duration escalation: 1 minute, 5 minutes, 30 minutes, 1 hour. Each subsequent offence from the same IP moves to the next tier.
UseAnchor no
Table-only mode. snortsentry manages the <snort_block> table directly via pfctl -t snort_block -T add. It does not create or use the PF anchor named in PfAnchor -- that setting is available for operators who prefer anchor-based management. The blocking rule is defined in pf.conf: block drop quick from <snort_block>.
AutoWhitelistRFC1918 yes
Automatically whitelists all RFC 1918 private ranges. Prevents LAN clients from being blocked by a misbehaving or misconfigured local device triggering a Snort rule.
Whitelist 192.168.122.0/24 %%INT_NET4%% %%INT_NET4%%
Additional explicit whitelists beyond RFC 1918 auto-whitelist. Includes the VM management network and the LAN subnet.
UnblockOnExit no
Blocks persist in the PF table when snortsentry stops or restarts. State is saved to /var/www/htdocs/tn/data/db/snortsentry/snortsentry.state and restored on restart.
MaxBlocks 10000
Maximum concurrent blocked IPs. Prevents the <snort_block> table from growing without bound under a sustained scanning attack.
sh 
# Check snortsentry is running
pgrep -l snortsentry

# View current snort_block table
pfctl -t snort_block -T show

# Count blocked IPs
pfctl -t snort_block -T show | wc -l

# Flush all snortsentry blocks
pfctl -t snort_block -T flush

# Run in foreground for debugging
/usr/local/sbin/snortsentry --no-daemon --log-level debug

p3scan

p3scan is a transparent POP3 proxy that scans mail attachments for malware before they reach the mail client. PF diverts POP3 (port 110) to loopback port 8994 and POP3S (port 995) to loopback port 8995. SSLproxy handles TLS for POP3S and forwards the decrypted stream to p3scan via its ProxySpec. p3scan passes attachments to ClamAV for scanning and holds the message if a virus is detected.

p3scan runs under the _p3scan user and is blocked from direct network access by PF user isolation rules. Upstream POP3 connections are made through the router's own address via the NAT chain, not by p3scan directly opening connections.

smtp-gated

smtp-gated intercepts outbound mail on SMTP (25), SMTPS (465), and submission (587) ports. It scans messages with both ClamAV and SpamAssassin before forwarding to the destination mail server. Messages containing viruses or scoring above the spam threshold are held in the spool.

Key configuration

mode pf
PF transparent proxy mode. smtp-gated recovers the original destination from the PF state table.
bind_address 127.0.0.1
Listens on loopback only. PF diverts to this address.
antivirus_type clamd
ClamAV daemon. Socket at /var/www/htdocs/tn/data/tmp/clamd.socket.
antispam_type spamassassin / antispam_path 127.0.0.1:783
SpamAssassin spamd listening on loopback port 783.
spam_threshold 5
Messages scoring 5 or above are treated as spam. SpamAssassin default threshold. Adjust based on observed false positive rate.
lock_on virus,spam,maxhost
Locks (temporarily blocks further mail from) a sender on virus detection, spam detection, or exceeding the per-host connection limit.
max_connections 8 / max_per_host 4
Global and per-host connection limits. Prevents a single compromised LAN host from consuming all smtp-gated capacity.
spool_leave_on error,spam
Leaves messages in the spool directory on error or spam detection for manual review. Spool path: /var/www/htdocs/tn/data/spool/smtp-gated/msg/.

imspector

imspector is a transparent instant messaging proxy. PF diverts IM protocol ports (MSN 1863, AIM 5190, Yahoo 5050, IRC 6667) to loopback port 16667 where imspector listens. imspector logs conversation content and can apply filtering rules. It runs as _imspector.

SSL support for IM protocols is not nonexistsent. Program logs plaintext IM traffic only.

Program binds an IPv4 socket only and has no IPv6 listener. PF passes IPv6 IM traffic with an explicit quick rule to make this gap visible in the ruleset rather than letting it fall through silently. IPv6 IM traffic exits without logging or filtering.

spamd

OpenBSD spamd provides greylisting for inbound SMTP connections. It delays connections from unknown senders -- legitimate mail servers retry and eventually deliver, while many spam sources do not retry. The active list in /etc/mail/spamd.conf is the nixspam list of recent spam sources. spamd-setup downloads and installs the list. Greylisted senders are temporarily deferred; listed senders are trapped.

spamd complements smtp-gated -- spamd reduces the volume of spam that reaches smtp-gated for content scanning, reducing ClamAV and SpamAssassin load.

SOCKS Proxy -- Dante sockd

Dante provides a SOCKS5 proxy for LAN clients. Applications that support SOCKS can route connections through the proxy rather than using direct TCP connections. sockd listens on %%INT_IF%% port 1080 and exits on %%EXT_IF%%.

Key configuration

internal: %%INT_IF%% port = 1080
Accepts SOCKS connections from the LAN interface only.
external: %%EXT_IF%%
Outbound connections exit on the WAN interface.
socksmethod: none
No SOCKS authentication required. LAN access is the access control.
user.notprivileged: _sockd
Drops privileges to _sockd after binding.

The access rules permit all LAN clients (%%INT_NET4%%), block connections targeting loopback (prevents the proxy from being used to reach local services), block bind operations below port 1023, and allow all other outbound connections. Log is at /var/www/htdocs/tn/data/logs/sockd/sockd.log.

Summary

  • All HTTP and HTTPS traffic passes through SSLproxy for TLS decryption, then e2guardian for content filtering, then Snort IPS for deep packet inspection. e2guardian does not handle TLS itself -- SSLproxy provides plaintext. This is correct by design.
  • Three e2guardian filter profiles are available: childsafe (full block including adult content), adult/general (malware and ads, no porn filtering), and user/custom (WebUI-defined feeds). Only one is active at a time. Profile switching is managed through the WebUI queue processor and reflected in the status JSON within seconds.
  • Two Snort instances run simultaneously. The IDS monitors %%INT_IF%% passively and feeds alerts to snortsentry, which blocks offending IPs in the PF snort_block table with escalating durations. The IPS runs inline on the post-decryption loopback chain and can drop packets directly.
  • snortsentry uses table-only mode (UseAnchor no). The blocking rule is in pf.conf. The PfAnchor setting is dormant -- available for operators who prefer anchor-based management.
  • SSLproxy user authentication is disabled (UserAuth no). The users database exists and is ready for future integration with TNAuth when per-user inspection policies are needed.
  • p3scan, smtp-gated, imspector, spamd, and sockd cover mail scanning, IM logging, spam greylisting, and SOCKS proxying respectively. All run under dedicated system users and are subject to PF user isolation rules.
