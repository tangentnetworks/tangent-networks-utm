Foreword

This document covers the PF firewall configuration of Tangent Networks: the rule structure, packet logging architecture, table management, the full divert chain that routes traffic through the inspection stack, NAT strategy, the blocklist automation that keeps threat intelligence current, and the WebUI-driven addons anchor system that allows live rule changes without reloading the base ruleset. It assumes you have read the Network Overview document. Interface names, addresses, NAT configuration, and the BPF buffer requirement are all covered there and are not repeated here.

The firewall document is the bridge between the network layer and the content inspection layer. Every divert rule documented here has a corresponding service documented in the Content Filtering fragment. This document explains what PF does with a packet; the Content Filtering document explains what happens to it after diversion.

All addresses shown are for the VM-based development and testing environment and are for illustration purposes only. Replace them with your actual interface addresses before deploying on hardware.

Firewall

Architecture Overview

PF is the sole packet filter on the system. It handles stateful filtering, NAT, traffic normalisation, and the diversion of traffic into the content inspection stack. The inspection stack itself (SSLproxy, Snort, e2guardian, p3scan, smtp-gated, imspector) runs as a set of user-space daemons. PF connects them to the traffic stream through divert rules without needing to understand application-layer content itself.

The firewall configuration is split across two files. The base ruleset lives in /etc/pf.conf and is loaded at boot. User-managed rules (IPs blocked or passed via the WebUI, custom rules, GeoIP policies) are applied through a named anchor and managed separately without touching the base ruleset.

IPv6 Architecture

The firewall operates in full dual-stack mode. Every IPv6 LAN flow receives the same inspection coverage as IPv4 flows, despite several proxy daemons (p3scan, smtp-gated, imspector, ClamAV) being IPv4-only codebases.

SSLproxy acts as the IPv6/IPv4 protocol boundary. IPv6 flows from LAN clients are diverted to SSLproxy's ::1 listeners via divert-to lo with an inet6 qualifier. SSLproxy decrypts, inspects, and re-originates traffic as IPv4 on loopback. All downstream daemons see only IPv4 loopback connections regardless of whether the originating client used IPv4 or IPv6.

Each protocol divert uses two rules, one inet and one inet6, both with divert-to lo. The inet/inet6 qualifier is required: without it PF resolves lo to ::1 only, generating only IPv6 divert rules and silently leaving IPv4 clients unintercepted. With the qualifier, PF resolves lo to 127.0.0.1 for inet rules and ::1 for inet6 rules.

The path for a LAN client (IPv4 or IPv6) connecting to an HTTPS destination:

text 
LAN client (%%INT_NET4%% or %%INT_NET6%%)
      |
      | PF divert-to lo port 8443  [R]
      v
SSLproxy (::1:8443 / 127.0.0.1:8443) -- TLS decryption
      |
      | forwards decrypted stream to lo:8080 (e2guardian)
      v
PF divert-packet port 700  [D] -- intercepts traffic to port 8080
      v
Snort IPS -- deep packet inspection, rejects or reinjects
      |
      v
e2guardian (127.0.0.1:8080) -- content filtering
      |
      v
NAT44 / NAT66 / NAT64 match out $ext_if
      |
      v
pass out $ext_if

Packet Traversal Order

The ruleset is structured in labelled sections. Understanding this order is essential when diagnosing why a packet was passed, blocked, or diverted.

text 
[A] Performance limits, variables, tables
[B] Logging -- real interfaces only, NOT loopback
[C] Default deny
[D] Loopback -- Snort IPS divert-packet rule
[E] FTP proxy anchor
[F] Scrub -- per-interface, per-family
[G] NDP/MLD exemptions (MUST precede bogon blocks)
[H] Loopback source spoofing blocks
[H2] LAN antispoofing
[I] Bogon blocks -- IPv4 and IPv6 on WAN ingress
[J] Additional spoofing blocks (link-local, Teredo, 6to4)
[K] Dynamic blocklists (snort_block, blocklist, bogons)
[L] Addons anchor
[M] User and daemon isolation
[N] ICMPv4 and ICMPv6 (RFC 4890)
[O] DHCPv4 and DHCPv6
[P] SSH
[Q] DNS
[R] Application proxy diverts -- MUST precede broad LAN pass
[S] Broad LAN pass -- after diverts so quick divert rules fire first
[T] WAN ping -- inbound echo to ($ext_if), any source
[U] NAT44, NAT66, NAT64
[V] Outbound general
[W] Application pass rules

Dual Logging Architecture

Two pflog interfaces serve distinct purposes and are managed entirely differently. Understanding this split is essential before reading the rest of the firewall configuration.

pflog0 -- standard block log
Managed by pflogd, started by rc.d. Receives PF block events in binary pcap format. Written to /var/log/pflog with a 160-byte snaplen (pflogd_flags=-s 160 -i pflog0 -f /var/log/pflog in /etc/rc.conf.local). Tangent tooling does not touch this interface or its log file.
pflog1 -- all-traffic sovereign interface
Not managed by pflogd. Brought up by /etc/hostname.pflog1 (contains only up) during netstart. The rule match log (to pflog1) on { %%INT_IF%% %%EXT_IF%% } directs a copy of every packet on the real interfaces to pflog1. Loopback is deliberately excluded: proxy relay traffic on loopback produces enormous volume with no security value. The pf-tcpdump pipeline reads from pflog1 continuously, writing to the MFS rolling window for real-time WebUI display and to a 7-day persistent archive. Full pipeline details are in the System Orchestration document.

pflog1 requires /etc/hostname.pflog1 to exist. If the file is absent, pflog1 is never created and set loginterface pflog1 fails silently at PF load time. The pf-tcpdump pipeline will start but find nothing to read. The WebUI PF log panel will be permanently empty. After a sysupgrade, verify it is still present: stat /etc/hostname.pflog1

sh 
# Verify both pflog interfaces are up
ifconfig pflog0
ifconfig pflog1

# Check pflogd is running on pflog0 only
ps aux | grep pflogd

# Check the pf-tcpdump pipeline is reading pflog1
ps aux | grep tcpdump

Performance Limits

PF's in-kernel limits are set before any rules are evaluated. These are hard ceilings: when a limit is reached, new connections are rejected. The values are sized for sustained traffic on an 8 GB system running the full service stack.

set limit states 200000
Maximum concurrent tracked connections. Each stateful pass rule creates an entry. 200,000 is comfortable for a home or small office gateway; reduce on low-RAM hardware.
set limit src-nodes 100000
Maximum tracked source nodes. Used when sticky-address or source-hash options are in effect.
set limit frags 8192
Maximum fragment reassembly entries. The scrub rule handles reassembly; this limit caps memory used for in-flight fragments. Loopback traffic is local and unfragmented, so this limit is sized conservatively.
set limit table-entries 1000000
Maximum total entries across all PF tables. The blocklist and bogon tables can grow large. One million entries accommodates a full Spamhaus DROP list plus bogon ranges plus per-user blocks with room to spare.
sh 
# Check current state table usage against limits
pfctl -s info | grep -A5 'State Table'

# Check table entry counts
pfctl -s Tables
pfctl -t blocklist -T show | wc -l
pfctl -t bogons    -T show | wc -l

TCP Timeout Tuning

PF maintains state for every tracked connection. Timeouts control how long state is kept after a connection reaches each TCP phase. A router that accumulates tens of thousands of connections needs aggressive timeout values to keep the state table from filling with dead entries.

pf 
set timeout tcp.established 86400   # keep established state for 24 hours
set timeout tcp.closing      60     # wait 60s for FIN/ACK exchange
set timeout tcp.finwait      30     # wait 30s after FIN before cleanup
set timeout tcp.closed       10     # remove closed state after 10 seconds

The 24-hour tcp.established value is intentionally long: it preserves state for idle but legitimate long-lived connections (SSH sessions, persistent WebSocket connections, overnight downloads). The tcp.closed value of 10 seconds is aggressive; it reclaims state table entries promptly after clean TCP teardown, complementing the kernel-level keepalive tuning in sysctl.conf.

Variables and Tables

Variables declared at the top of pf.conf are expanded wherever they appear in rules. All interface references use these variables; updating the variable updates every rule that references it.

pf 
ext_if = "%%EXT_IF%%"            # WAN interface
int_if = "%%INT_IF%%"            # LAN interface

int_net4 = "%%INT_NET4%%"        # LAN IPv4 CIDR
int_net6 = "%%INT_NET6%%"        # LAN IPv6 ULA prefix

ssh_port  = 22
im_ports  = "{ 1863, 5190, 5050, 6667 }"   # Instant Messaging ports
fb_ports  = "{ 5228, 5229, 5230 }"         # Firebase Cloud Messaging
nat64_pfx = "%%NAT64_PFX%%"                 # NAT64 well-known prefix (RFC 6052)
nat64_int_ip4 = "%%NAT64_INT_IP4%%"        # NAT64 form of %%INT_IP4%% (int_if)

Tables are kernel-resident address sets that can be updated at runtime without reloading pf.conf. They are the mechanism through which pfblock.sh, snortsentry, and the WebUI firewall panel add and remove addresses from the active ruleset atomically.

pf 
table <lan_nets>     { %%INT_NET4%%, %%INT_NET6%% }
table <blocklist>    persist
table <bogons>       persist
table <snort_block>  persist
<lan_nets>
Both LAN prefixes (IPv4 and IPv6) in a single table. Used in the broad LAN pass rule at section [S] to allow traffic from either address family without writing two separate rules.
<blocklist> and <bogons>
Runtime-populated threat intelligence tables. Start empty at PF load. Populated by pfblock.sh using pfctl -T replace, which swaps the entire table contents in a single kernel operation with no gap in enforcement. The persist flag keeps the table allocated in kernel memory even when empty.
<snort_block>
Populated by snortsentry when Snort triggers a block alert. Both IPv4 and IPv6 addresses are supported. Entries persist until PF is reloaded or the table is explicitly flushed.

Base Policy and Rule Ordering

PF evaluates rules in order and the last matching rule wins, except where quick is specified, which stops evaluation immediately. The opening directives establish the foundation everything else builds on.

pf 
set loginterface pflog1
match log (to pflog1) on { $ext_if $int_if }
block return on { $ext_if $int_if } all

The match rule is not a pass or block decision: it applies the log attribute to every packet on the real interfaces before any pass or block rule is evaluated. Because match rules do not terminate evaluation, the log attribute sticks through all subsequent rule processing. Every packet that PF touches on $ext_if or $int_if is logged to pflog1 regardless of the eventual pass or block decision.

The default deny uses block return on real interfaces, which sends TCP RST or ICMP unreachable to blocked sources. Spoofed and bogon sources use block drop selectively (see sections [H] through [K]) because sending a response to a known-bad source is wasteful and reveals the firewall's presence.

Loopback is deliberately not included in the default deny and has no set skip on lo. PF must see loopback traffic so the Snort IPS divert-packet rule fires on proxy re-originated flows. A global block on loopback would catch reinjected packets and break the inspection chain. After Snort reinjects a diverted packet on loopback it must pass freely.

Why loopback is not skipped. The standard OpenBSD pattern is set skip on lo, which bypasses all PF processing on loopback. This ruleset omits it because the Snort IPS divert-packet rule at section [D] must fire on loopback traffic. Without PF processing on loopback, the entire post-decryption inspection chain is bypassed. This follows the UTMFW reference implementation. If Snort inline inspection stops working after a PF configuration change, verify that no set skip on lo has been introduced.

Loopback -- Snort IPS Divert

The single loopback rule intercepts traffic destined for the proxy daemon ports and sends it to Snort's ipfw DAQ on divert port 700.

pf 
pass in quick on lo proto tcp to port { 8080 8110 9199 } divert-packet port 700

The three ports are loopback destinations used after SSLproxy decryption: 8080 for HTTP/HTTPS (to e2guardian), 8110 for POP3/POP3S/IMAPS (to p3scan), and 9199 for SMTP/SMTPS/SUBMISSION (to smtp-gated). Snort inspects the plaintext content and either drops the packet or reinjects it to continue to the downstream daemon. No scrub rule is applied to loopback: adding one causes PF to implicitly set flags S/SA, which breaks matching on established flows (only SYN packets would match the divert rule). Loopback traffic is local and not fragmented, so reassembly is not needed.

FTP Proxy

Active-mode FTP requires the server to initiate a data connection back to the client, which breaks through NAT without special handling. The ftp-proxy/* anchor must appear before the FTP divert rules so ftp-proxy's dynamically created rules for data connections are evaluated at the correct point in the chain. Both IPv4 and IPv6 are covered.

pf 
anchor "ftp-proxy/*"

pass in quick on $int_if inet  proto tcp from $int_net4 to !$int_if port 21 \
    divert-to lo port 8021
pass in quick on $int_if inet6 proto tcp from $int_net6 to !$int_if port 21 \
    divert-to lo port 8021

Traffic Normalisation

Scrub rules are applied per-interface and per-family. IPv4 and IPv6 require different treatment: no-df and random-id are IPv4-only options (IPv6 has no DF bit and no IP identification field), so only max-mss applies to IPv6.

pf 
# IPv6 -- max-mss only
match in on $ext_if inet6 scrub (max-mss $PF_MAX_MSS_EXT_IF)
match in on $int_if inet6 scrub (max-mss $PF_MAX_MSS_INT_IF)

# IPv4 -- full normalisation
match in on $ext_if inet  scrub (no-df random-id max-mss $PF_MAX_MSS4_EXT_IF)
match in on $int_if inet  scrub (no-df random-id max-mss $PF_MAX_MSS4_INT_IF)

MSS values are determined at boot by tn-mtu-probe.sh, which discovers path MTU and writes confirmed values to /etc/tn-interfaces for use by this ruleset, rad.conf, and the hostname files. MSS derivation: IPv6 MSS = wire MTU - 60 (IPv6 + TCP headers); IPv4 MSS = wire MTU - 40 (IPv4 + TCP headers). The no-df option clears the Don't Fragment bit to prevent PMTUD black holes. random-id replaces predictable IP ID fields to prevent OS fingerprinting.

NDP and MLD Exemptions

These rules must appear before the bogon blocks at section [I]. NDP uses fe80::/10 (link-local) as source addresses, and MLD uses ff00::/8 (multicast) -- both would be caught by the bogon block below. The quick pass rules here preempt the bogon blocks for these specific message types.

pf 
# NDP on WAN
pass in  quick on $ext_if inet6 proto icmp6 from fe80::/10 \
    icmp6-type { neighbrsol neighbradv routeradv } keep state
pass out quick on $ext_if inet6 proto icmp6 to fe80::/10 \
    icmp6-type { neighbrsol neighbradv routersol } keep state

# MLD on WAN
pass in  quick on $ext_if inet6 proto icmp6 icmp6-type { 130 131 132 143 } keep state
pass out quick on $ext_if inet6 proto icmp6 icmp6-type { 130 131 132 143 } keep state

Anti-Spoofing

Four categories of spoofing are blocked. Loopback-sourced packets arriving on real interfaces are silently dropped. Packets arriving on $int_if with a source address outside the legitimate LAN subnets are dropped; this prevents compromised LAN hosts from forging source addresses to bypass source-based firewall rules. NDP link-local traffic already passed at [G] is unaffected by the LAN antispoofing rule.

pf 
# Loopback source spoofing
block drop in quick on !lo inet  from 127.0.0.0/8 to any
block drop in quick on !lo inet6 from ::1 to any
block drop in quick        inet6 from ::1 to any

# LAN antispoofing -- block sources outside legitimate LAN subnets
block drop in quick on $int_if inet  from !$int_net4 to any
block drop in quick on $int_if inet6 from !$int_net6 to any

Bogon Blocks

Bogon blocks apply to WAN ingress only. The development environment uses RFC1918 addresses on the WAN side and a ULA upstream, so those ranges are excluded from the blocks. Restore them before deploying on hardware with a real ISP connection.

pf 
# IPv4 bogon block on WAN ingress
# PRODUCTION: restore 10/8, 172.16/12, 192.168/16 before deployment
block drop in quick on $ext_if inet from {
    0.0.0.0/8, 100.64.0.0/10,
    127.0.0.0/8, 169.254.0.0/16,
    192.0.0.0/24, 192.0.2.0/24,
    198.18.0.0/15, 198.51.100.0/24, 203.0.113.0/24,
    240.0.0.0/4, 255.255.255.255/32 } to any

# IPv6 bogon block on WAN ingress
# PRODUCTION: restore fc00::/7 before deployment with real ISP GUA on WAN
block drop in quick on $ext_if inet6 from {
    ::1/128, ::/128, ::ffff:0:0/96, 100::/64,
    2001::/32, 2001:2::/48, 2001:db8::/32, 2001:10::/28,
    fec0::/10, ff00::/8 } to any

Additional Spoofing Blocks

Link-local addresses arriving on $int_if with a non-link-local destination are spoofed and dropped silently. Teredo (UDP port 3544) and 6to4 (protocol 41) create unmonitored IPv6 tunnels that bypass the inspection chain; native dual-stack via NAT66 and NAT64 makes them unnecessary, so both are blocked on $ext_if.

pf 
block drop in quick on $int_if inet6 from fe80::/10 to !fe80::/10
block drop in quick on $ext_if proto udp to port 3544
block drop in quick on $ext_if proto 41

Dynamic Blocklists

Three tables apply early block rules after the spoofing sections. All three use block drop (silent discard) rather than block return: sending a response to a known malicious source reveals the firewall's presence and wastes bandwidth. The quick keyword stops evaluation immediately; no downstream pass rule can override these for matching traffic.

pf 
block drop quick from <snort_block>
block drop quick from <blocklist>
block drop quick from <bogons>

All three are logged to pflog1 via the earlier match log (to pflog1) rule. Tables accept both IPv4 and IPv6 entries natively. <snort_block> is populated by snortsentry in response to Snort alerts; <blocklist> and <bogons> are populated daily by pfblock.sh.

The Addons Anchor

The addons anchor is the injection point for all user-managed firewall rules. It sits after the global blocks and before the ICMP and service rules, giving user rules access to the same enforcement position as the base ruleset.

pf 
anchor "addons"

Rules are loaded into the anchor from /etc/pf/pf-addons.conf by the WebUI's PF management pipeline. The anchor contents can be replaced, flushed, or inspected without touching or reloading pf.conf.

sh 
# Load rules into the anchor
pfctl -a addons -f /etc/pf/pf-addons.conf

# View currently active addon rules
pfctl -a addons -s rules

# Flush all addon rules (leaves base ruleset intact)
pfctl -a addons -F rules

# View the anchor tables
pfctl -a addons -s Tables

User and Daemon Isolation

PF enforces network isolation for inspection daemon users to prevent a compromised daemon from initiating its own outbound connections or bypassing the inspection chain. Daemons receive traffic only via PF divert; direct connections from their user accounts are blocked on all interfaces.

pf 
# Block remote X11 on all non-loopback interfaces
block return in on !lo proto tcp to port 6000:6010

# Ports build user -- no outbound internet
block return out proto { tcp udp } user _pbuild

# Proxy daemons -- no direct network connections on any interface
block drop on any proto { tcp udp } user { _e2guardian _p3scan _smtp-gated }

# Dante -- no inbound connections except via SOCKS listener rules at [W]
block drop in on any proto { tcp udp } user _sockd

# Snort -- no outbound traffic generation
block drop out on any proto { tcp udp } user _snort

ICMPv4

ICMP unreachables and time-exceeded are permitted inbound for path MTU discovery and network diagnostics. Outbound is unrestricted. Inbound echo requests on the WAN interface are handled separately at section [T] and are not included here.

pf 
pass in  inet proto icmp icmp-type { unreach timex } keep state
pass out inet proto icmp                              keep state

ICMPv6 -- RFC 4890 Compliance

ICMPv6 rules are split into three groups following RFC 4890. Group 1 covers Packet Too Big (type 2), which is stateless and must be passed quickly on all interfaces. Group 2 covers error messages (types 1, 3, 4), which are stateful. Group 3 covers informational messages: NDP, MLD, and echo.

NDP and MLD on $ext_if are already handled at section [G] before the bogon blocks. The rules here cover $int_if NDP and MLD, and WAN outbound echo. Inbound WAN echo is consolidated at section [T].

pf 
# Group 1: Packet Too Big -- stateless, quick, all interfaces
pass in  quick on { $ext_if $int_if } inet6 proto icmp6 icmp6-type toobig
pass out quick on { $ext_if $int_if } inet6 proto icmp6 icmp6-type toobig

# Group 2: Error messages -- stateful
pass in  on $ext_if inet6 proto icmp6 icmp6-type { unreach timex paramprob } keep state
pass out on $ext_if inet6 proto icmp6 icmp6-type { unreach timex paramprob } keep state
pass in  on $int_if inet6 proto icmp6 icmp6-type { unreach timex paramprob } keep state
pass out on $int_if inet6 proto icmp6 icmp6-type { unreach timex paramprob } keep state

# Group 3: Echo on WAN outbound -- unrestricted. Inbound handled at [T].
pass out on $ext_if inet6 proto icmp6 \
    icmp6-type { echoreq echorep } keep state

# NDP on LAN -- mandatory for IPv6 address resolution
pass in  on $int_if inet6 proto icmp6 \
    icmp6-type { neighbrsol neighbradv routersol } keep state
pass out on $int_if inet6 proto icmp6 \
    icmp6-type { neighbradv routeradv } keep state

# MLD on LAN -- mandatory for IPv6 multicast group management
pass in  on $int_if inet6 proto icmp6 icmp6-type { 130 131 132 143 } keep state
pass out on $int_if inet6 proto icmp6 icmp6-type { 130 131 132 143 } keep state

# Echo on LAN -- full allow
pass in  on $int_if inet6 proto icmp6 from <lan_nets> \
    icmp6-type { echoreq echorep } keep state
pass out on $int_if inet6 proto icmp6 to <lan_nets> \
    icmp6-type { echoreq echorep } keep state

DHCP

DHCPv4 requires both unicast and broadcast forms because clients without an assigned address cannot use unicast. DHCPv6 covers the router as a client on $ext_if (requesting prefix delegation) and as a server on $int_if (responding to LAN client solicitations via rad). DHCPv6 solicitations use multicast as their destination and are passed without state because multicast has no predictable return path for PF to track.

pf 
# DHCPv4
pass in on $int_if inet proto udp from any port 68 to ($int_if)      port 67 keep state
pass in on $int_if inet proto udp from any port 68 to 255.255.255.255 port 67 keep state

# DHCPv6 -- WAN (router as client)
pass out on $ext_if inet6 proto udp from fe80::/10 to ff02::1:2 port 547 no state
pass in  on $ext_if inet6 proto udp from fe80::/10 port 547 \
    to fe80::/10 port 546 keep state

# DHCPv6 -- LAN (router as server via rad)
pass in  on $int_if inet6 proto udp from fe80::/10 to ff02::1:2 port 547 keep state
pass out on $int_if inet6 proto udp to fe80::/10 port 546 keep state

SSH

SSH is accepted on both interfaces ($int_if and $ext_if). The flags S/SA match restricts to SYN packets only, preventing mid-stream packet injection from establishing state. Brute-force protection limits concurrent connections and connection rate per source; offending addresses are added to <blocklist> and their existing states are flushed globally.

pf 
pass in quick on { $int_if $ext_if } proto tcp \
    to { ($int_if) ($ext_if) } port $ssh_port \
    flags S/SA keep state \
    (max-src-conn 10, max-src-conn-rate 5/30, overload <blocklist> flush global)

DNS

LAN clients are restricted to querying Unbound on the router; they cannot send queries directly to external resolvers. Unbound binds both IPv4 and IPv6 on the LAN interface, so both families are covered. The outbound rules on $ext_if permit Unbound's own upstream queries. Gateway processes querying Unbound on loopback use no state rules (local only, no return path tracking needed).

pf 
pass in on $int_if inet  proto { udp tcp } from $int_net4 to ($int_if) port 53 keep state
pass in on $int_if inet6 proto { udp tcp } from $int_net6 to ($int_if) port 53 keep state

pass out on $ext_if inet  proto { udp tcp } from ($ext_if) to any port 53 keep state
pass out on $ext_if inet6 proto { udp tcp } to any port 53 keep state

# Local processes querying Unbound on loopback
pass on lo inet  proto { udp tcp } to 127.0.0.1 port 53 no state
pass on lo inet6 proto { udp tcp } to ::1       port 53 no state

Application Proxy Diverts

The divert rules are the most operationally critical section of the ruleset. They must appear before the broad LAN pass rule at section [S]; all use quick so evaluation stops at the first matching divert rule and the broad pass at [S] never fires for diverted traffic.

Every protocol uses two rules (one inet, one inet6) both with divert-to lo. The to !$int_if match excludes all $int_if addresses (both IPv4 and IPv6 including link-local), preventing the router's own traffic from being diverted.

SSLproxy -- HTTP and HTTPS

pf 
# HTTP -- SSLproxy listens on 127.0.0.1:8081 and ::1:8081
pass in quick on $int_if inet  proto tcp from $int_net4 to !$int_if port www \
    divert-to lo port 8081
pass in quick on $int_if inet6 proto tcp from $int_net6 to !$int_if port www \
    divert-to lo port 8081

# HTTPS -- SSLproxy listens on 127.0.0.1:8443 and ::1:8443
pass in quick on $int_if inet  proto tcp from $int_net4 to !$int_if port https \
    divert-to lo port 8443
pass in quick on $int_if inet6 proto tcp from $int_net6 to !$int_if port https \
    divert-to lo port 8443

All HTTP and HTTPS traffic from LAN clients is intercepted by SSLproxy. SSLproxy decrypts TLS connections using a locally installed CA certificate, passes decrypted content to e2guardian for content filtering, and re-encrypts outbound. IPv6 clients are handled identically to IPv4 clients; SSLproxy re-originates the decrypted traffic as IPv4 on loopback.

SSLproxy to p3scan -- POP3, POP3S, and IMAPS

pf 
# POP3 -- SSLproxy listens on 127.0.0.1:8994 and ::1:8994
pass in quick on $int_if inet  proto tcp from $int_net4 to !$int_if port pop3 \
    divert-to lo port 8994
pass in quick on $int_if inet6 proto tcp from $int_net6 to !$int_if port pop3 \
    divert-to lo port 8994

# POP3S -- SSLproxy listens on 127.0.0.1:8995 and ::1:8995
pass in quick on $int_if inet  proto tcp from $int_net4 to !$int_if port pop3s \
    divert-to lo port 8995
pass in quick on $int_if inet6 proto tcp from $int_net6 to !$int_if port pop3s \
    divert-to lo port 8995

# IMAPS -- SSLproxy listens on 127.0.0.1:8993 and ::1:8993
pass in quick on $int_if inet  proto tcp from $int_net4 to !$int_if port imaps \
    divert-to lo port 8993
pass in quick on $int_if inet6 proto tcp from $int_net6 to !$int_if port imaps \
    divert-to lo port 8993

POP3, POP3S, and IMAPS are all diverted through SSLproxy to p3scan, which scans message attachments for malware using ClamAV before delivering them to the mail client. POP3S and IMAPS are decrypted by SSLproxy before reaching p3scan. The decrypted stream passes through the Snort IPS divert-packet rule at [D] on loopback port 8110 before reaching the daemon. Both IPv4 and IPv6 are fully covered for all three protocols.

SSLproxy to smtp-gated -- SMTP, SMTPS, Submission

pf 
# SMTP -- SSLproxy listens on 127.0.0.1:8464 and ::1:8464
pass in quick on $int_if inet  proto tcp from $int_net4 to !$int_if port smtp \
    divert-to lo port 8464
pass in quick on $int_if inet6 proto tcp from $int_net6 to !$int_if port smtp \
    divert-to lo port 8464

# SMTPS -- SSLproxy listens on 127.0.0.1:8465 and ::1:8465
pass in quick on $int_if inet  proto tcp from $int_net4 to !$int_if port 465 \
    divert-to lo port 8465
pass in quick on $int_if inet6 proto tcp from $int_net6 to !$int_if port 465 \
    divert-to lo port 8465

# SUBMISSION -- SSLproxy listens on 127.0.0.1:8466 and ::1:8466
pass in quick on $int_if inet  proto tcp from $int_net4 to !$int_if port submission \
    divert-to lo port 8466
pass in quick on $int_if inet6 proto tcp from $int_net6 to !$int_if port submission \
    divert-to lo port 8466

Outbound mail on all three standard ports (SMTP 25, SMTPS 465, submission 587) is intercepted by smtp-gated for content scanning. SMTPS and submission TLS is decrypted by SSLproxy before reaching smtp-gated.

imspector -- Instant Messaging

pf 
# IPv4 IM -- diverted to imspector
pass in quick on $int_if inet  proto tcp from $int_net4 to !$int_if \
    port $im_ports divert-to lo port 16667

# IPv6 IM -- explicit quick pass; imspector has no IPv6 socket
pass in quick on $int_if inet6 proto tcp from $int_net6 to !$int_if \
    port $im_ports

Instant messaging protocols (MSN/Live 1863, AIM/ICQ 5190, Yahoo 5050, IRC 6667) are diverted to imspector for logging and optional filtering on IPv4. imspector binds an IPv4 wildcard socket only and has no IPv6 socket support. The explicit quick pass on inet6 makes the gap visible in the ruleset and prevents IPv6 IM traffic from falling through silently to the broad LAN pass at [S]. If IM policy requires IPv6 inspection, block $im_ports on inet6 here instead.

NAT64 Local SOCKS

pf 
pass in quick on $int_if inet6 proto tcp \
    from $int_net6 to $nat64_int_ip4 port 1080 \
    divert-to %%INT_IP6%% port 1080
pass in quick on $int_if inet6 proto udp \
    from $int_net6 to $nat64_int_ip4 port 1080 \
    divert-to %%INT_IP6%% port 1080

nat64_int_ip4 is the NAT64 representation of the $int_if IPv4 address (%%INT_IP4%% maps to %%NAT64_INT_IP4%%). When an IPv6 LAN client tries to reach the SOCKS proxy via its NAT64-mapped address, this rule intercepts before the NAT64 af-to rule fires and diverts directly to Dante's IPv6 listener. The connection stays IPv6 end-to-end with no protocol translation needed.

Broad LAN Pass

This rule is placed after all divert rules. The quick divert rules above fire first for any matching traffic; everything else falls through to this broad pass.

pf 
pass in on $int_if from <lan_nets> keep state

This covers all remaining LAN traffic: router-addressed packets and forwarded traffic destined for $ext_if. Both IPv4 ($int_net4) and IPv6 ($int_net6) sources are covered via the <lan_nets> table. The WebUI is also reachable here; its traffic is not caught by the SSLproxy diverts which use to !$int_if, and is explicitly permitted at [W] as well.

WAN Ping

Inbound ICMP echo requests to ($ext_if) are permitted from any source on both IPv4 and IPv6. ($ext_if) resolves the current WAN address dynamically, correctly tracking both DHCP-assigned IPv4 and SLAAC/DHCPv6-PD-assigned IPv6 without requiring a rule reload. This section consolidates all WAN inbound echo -- it is not handled in the ICMPv4 or ICMPv6 sections above.

pf 
pass in on $ext_if inet  proto icmp  to ($ext_if) icmp-type  echoreq keep state
pass in on $ext_if inet6 proto icmp6 to ($ext_if) icmp6-type echoreq keep state

NAT Strategy

Three NAT mechanisms operate simultaneously. NAT44 masquerades IPv4 LAN clients behind the $ext_if IPv4 address. NAT66 masquerades IPv6 ULA LAN clients behind the $ext_if GUA. NAT64 allows IPv6 LAN clients to reach IPv4-only internet destinations via DNS64 (Unbound) and PF af-to translation.

($ext_if) with parentheses rather than a literal address resolves the current WAN address at translation time. This is correct for DHCP-assigned or SLAAC-assigned WAN addresses: no rule reload is needed if the WAN address changes.

NAT44

pf 
match out on $ext_if inet from $int_net4 to any nat-to ($ext_if)

NAT66

pf 
match out on $ext_if inet6 from $int_net6 to any nat-to ($ext_if)
pass out quick on $ext_if inet6 from ($ext_if) to any keep state

LAN clients use ULA addresses (%%INT_NET6%%) which are not globally routable. NAT66 masquerades them behind the WAN GUA assigned by SLAAC or DHCPv6-PD. The keep state on the outbound pass rule handles the return path automatically; no explicit inbound pass is needed.

NAT66 PMTUD limitation. The OpenBSD kernel does not generate ICMPv6 Packet Too Big messages for oversized forwarded packets through NAT66. This is a known, documented kernel limitation. Mitigation is source-based: rad advertises the confirmed wire MTU via the RA MTU option so SLAAC clients self-limit, and the PF scrub max-mss rule clamps TCP MSS on ingress for all clients regardless of their local MTU configuration.

Production: prefer native IPv6 routing over NAT66. If your ISP delegates a DHCPv6-PD prefix, assign it to $int_if, advertise it via rad, and remove the NAT66 match rule. Native routing is always preferred over NAT66 where globally routable prefixes are available.

NAT64

pf 
pass in on $int_if inet6 from $int_net6 to $nat64_pfx \
    af-to inet from ($ext_if) keep state

pass out on $ext_if inet from ($ext_if) to any keep state

The NAT64 prefix is the well-known prefix %%NAT64_PFX%% (RFC 6052), stored in the variable nat64_pfx. Unbound's DNS64 module synthesises AAAA records under this prefix for IPv4-only destinations. When an IPv6 LAN client sends a packet to an address in that prefix, PF's af-to translates IPv6 to IPv4, extracts the destination from the lower 32 bits of the NAT64 address, and forwards as IPv4. NAT44 masquerades the outbound packet. The following constraints must be observed or NAT64 breaks: do not add a protocol list, flags, quick, or a table reference to the af-to rule.

A prerequisite static route is required on $int_if:

text 
# /etc/hostname.%%INT_IF%%
route add -inet6 %%NAT64_PFX%% %%INT_IP6%%

Outbound General

These two rules are a safety net. All specific and quick rules above have already fired; these cover anything remaining.

pf 
pass out on $int_if from $int_if:network to any keep state
pass out on $ext_if from any             to any keep state

The WebUI Addons Management Pipeline

The WebUI does not write to pf.conf or call pfctl directly. All firewall changes flow through a privilege-separated pipeline that bridges the web server's unprivileged www user and the root-side daemons that operate PF.

Privilege Separation

The www user (slowcgi, all CGI scripts) runs inside the /var/www chroot. It cannot execute pfctl, read /etc/pf/, or write to /etc/pf/pf-addons.conf directly. Root-side daemons (pf_monitor.sh, pf_anchor_sync.sh, pf_delete_block.sh) run outside the chroot. They watch a queue directory for trigger files, execute pfctl, edit /etc/pf/pf-addons.conf, and write result JSON back into the queue for the CGI layer to serve to the browser.

The Queue Pattern

All firewall operations follow the same pattern. The browser posts to a CGI script running as www. The CGI validates input and drops a request or trigger file in the queue directory under /var/www/htdocs/tn/data/services/queue/pf-rules/. A root daemon picks up the trigger, executes the corresponding action (validate, apply, delete, or sync), and writes a JSON outcome file. The browser polls a CGI action that serves the file; direct fetches of the queue path are blocked by httpd.

Queue Directory Structure

Base path: /var/www/htdocs/tn/data/services/queue/pf-rules/

user-input/
Queue files written by CGI: ip-block.txt, ip-pass.txt, asn-block.txt, feed-urls.txt, custom-rules.txt, geoip-policy.json. Written by www, read by pf_validator.pl (root).
triggers/
Sentinel files that signal root daemons. Written by both www (CGI) and root daemons to sequence operations. Permissions: root:www 775.
staging/
Assembled conf files pending apply. Written by both pf_validator.pl (root) and pf_write_rules.pl (www CGI). Permissions: root:www 775. If set to 750, the CGI cannot write the deletion preview conf and will return a 500 error when the operator clicks Preview Removal.
validation-output/
verdict.json and full-context.txt written by pf_monitor.sh after running pfctl -nf on the staged conf.
delete-requests/
Per-deletion request files written by pf_active_rules.pl. Each file is a JSON object with the deletion type, target table, and entries to remove.
delete-outcome/
Result files written by pf_delete_block.sh after each deletion.
mirror/
Read-only copies of pf.conf and pf-addons.conf seeded at boot by pf_mirror_sync.sh.
parsed-rules.json
DAG of all custom rules in the addons anchor. Written by pf_rule_parser.pl on every anchor sync.
active-addons.json
Currently loaded anchor blocks with table entry counts. Written by pf_anchor_sync.sh every 30 seconds and after every change. Read by the active addon rules panel in the WebUI.

Root-Side Daemons

pf_monitor.sh
Core root daemon. Polls triggers/ every 2 seconds. Handles validate, apply, reset, delete-requests, test-deletion, and apply-deletion triggers. Run via pf_monitor_runner.sh, which restarts it if it exits.
pf_anchor_sync.sh
Reads live anchor state via pfctl -a addons -sr. If the anchor is empty and pf-addons.conf is non-empty, tests and loads it. Parses the conf into JSON blocks and writes active-addons.json. Calls pf_rule_parser.pl on every exit via a trap EXIT. Run every 30 seconds by pf_anchor_sync_runner.sh.
pf_validator.pl
Assembles the full staging/pf-addons.conf from all user-input queue files. Handles IP blocks, IP pass lists, ASN blocks, GeoIP blocks (reads RIPE country CIDR files), threat feed downloads, and custom rules. Outputs stats and warnings parsed by pf_monitor.sh.
pf_delete_block.sh
Processes delete-requests/*.json. Handles whole-block deletion (flush table, remove section from conf, reload anchor) and per-entry deletion (uses awk to remove specific IPs or CIDRs from a table definition, validates with pfctl -nf, removes entries from the live kernel table via pfctl -T delete, then reloads the anchor). Writes outcome JSON on completion.
pf_rule_parser.pl
Reads /etc/pf/pf-addons.conf, joins backslash continuation lines, classifies each line by type, extracts dependency tokens, and builds a directed acyclic graph (DAG). Writes parsed-rules.json atomically.

DAG-Based Rule Deletion

Custom rules in pf-addons.conf form a dependency graph. A filter rule (block drop quick from <user_block_ips>) depends on the table definition that precedes it. Deleting the table without deleting the filter rules that reference it leaves dangling references that cause pfctl -nf to fail.

pf_rule_parser.pl builds a DAG where each node is a rule and each edge represents a dependency. The graph is stored in parsed-rules.json as a reverse index: for each object name (table name, macro name), the graph lists every rule ID that depends on it. When the operator selects rules to delete, a breadth-first search (BFS) traversal of the reverse graph automatically identifies all rules that would become dangling. These are highlighted amber in the UI and included in the deletion set automatically.

The full deletion flow: parser builds the DAG on every anchor sync; WebUI loads parsed rules; the operator checks a rule and cascade detection highlights dependent rules; a preview renders the proposed conf as a diff; pfctl -nf validates the staged conf; on confirmation, pf_monitor.sh loads the conf into the live anchor, copies it to /etc/pf/pf-addons.conf, and refreshes parsed-rules.json.

sh 
# Verify parsed-rules.json exists and is current
ls -la /var/www/htdocs/tn/data/services/queue/pf-rules/parsed-rules.json

# Inspect the DAG structure
cat /var/www/htdocs/tn/data/services/queue/pf-rules/parsed-rules.json \
    | python3 -m json.tool | head -60

# Check the staged deletion conf
cat /var/www/htdocs/tn/data/services/queue/pf-rules/staging/pf-addons-deletion.conf

# Test it manually
pfctl -a addons -nf \
    /var/www/htdocs/tn/data/services/queue/pf-rules/staging/pf-addons-deletion.conf

Per-Entry IP Deletion

Table definitions in pf-addons.conf can contain hundreds of IP addresses. When the operator removes individual entries, pf_active_rules.pl validates each entry as a valid IP or CIDR and writes a delete-requests/<timestamp>.json with "type":"entry_delete". pf_delete_block.sh processes this by using awk to rewrite the table definition in-place, removing only the specified entries and rebuilding the correct backslash continuation syntax. It validates the result with pfctl -nf, then removes the entries from the live kernel table immediately via pfctl -t <tablename> -T delete before reloading the full anchor.

sh 
# Remove a specific IP from a table without reloading the anchor
pfctl -t user_block_ips -T delete 223.5.5.5

# Verify it is gone from the kernel table
pfctl -a addons -t user_block_ips -T show | grep 223.5.5.5

# Check the delete outcome log
ls -lt /var/www/htdocs/tn/data/services/queue/pf-rules/delete-outcome/

Blocklist Automation -- pfblock.sh

pfblock.sh runs daily at 11:00 via cron. It downloads threat intelligence feeds, extracts and deduplicates IP addresses, and atomically replaces the <blocklist> and <bogons> tables in the running PF instance.

Feed sources are declared in /var/www/htdocs/tn/data/db/pf/intel.txt with three categories: plain IP feed lists (IP: prefix), a bogon range feed (BOGONS:), and a CSV feed (CSV:). The current configured sources:

text 
IP: http://rules.emergingthreats.net/blockrules/compromised-ips.txt
IP: https://malware-filter.gitlab.io/malware-filter/urlhaus-filter-domains.txt
IP: https://lists.blocklist.de/lists/all.txt
IP: https://pgl.yoyo.org/adservers/iplist.php?ipformat=plain&showintro=0&mimetype=plaintext
BOGONS: https://www.cidr-report.org/bogons/freespace-prefix.txt
CSV: https://blocklist.net.ua/blocklist.csv

intel.txt is served through the CGI layer. The file lives at /var/www/htdocs/tn/data/db/pf/intel.txt, which is blocked by httpd. The WebUI fetches it via pf_active_rules.pl action:get_intel rather than a direct fetch.

The atomic replace operation (pfctl -T replace -f file) swaps the entire table contents as a single kernel operation with no window during which the table is empty.

/usr/local/sbin/pfblock.sh

# Check last run log

# Check current table sizes
pfctl -t blocklist -T show | wc -l
pfctl -t bogons    -T show | wc -l

# Add a single IP to blocklist immediately
pfctl -t blocklist -T add 198.51.100.1

# Remove an IP from blocklist
pfctl -t blocklist -T delete 198.51.100.1

Summary

  • Two pflog interfaces serve distinct roles. pflog0 receives block events via pflogd in binary pcap format. pflog1 receives every packet on $ext_if and $int_if via match log (to pflog1) on { $ext_if $int_if } and feeds the real-time WebUI display and 7-day archive. Loopback is deliberately excluded from logging.
  • Loopback is not skipped and has no default deny. PF must process loopback so the Snort IPS divert-packet rule fires on proxy re-originated flows. This is a deliberate departure from the standard OpenBSD set skip on lo pattern.
  • Every application proxy divert uses two rules (inet and inet6) with divert-to lo and an explicit inet/inet6 qualifier. Without the qualifier, PF resolves lo to ::1 only and IPv4 clients go unintercepted.
  • Three NAT mechanisms run simultaneously: NAT44 for IPv4 LAN clients, NAT66 for IPv6 ULA LAN clients, and NAT64 for IPv6 clients reaching IPv4-only destinations. NAT64 uses the well-known prefix %%NAT64_PFX%% via the nat64_pfx variable and the af-to inet rule. All three use ($ext_if) to resolve the current WAN address at translation time.
  • SSH is accepted on both interfaces ($int_if and $ext_if) with brute-force rate limiting. Offending sources are added to <blocklist> and flushed globally.
  • Inbound WAN echo (ICMPv4 and ICMPv6) is permitted from any source and is consolidated at section [T], not in the ICMPv4/ICMPv6 sections.
  • IMAPS (993) is fully inspected on both IPv4 and IPv6 via the SSLproxy to p3scan divert chain. Only plain IMAP (143) is absent from the ruleset -- it is not covered.
  • Three runtime-populated tables handle dynamic threat blocking: <blocklist> and <bogons> updated daily by pfblock.sh via atomic replace, and <snort_block> populated by snortsentry in response to Snort alerts.
  • The addons anchor is the injection point for all WebUI-managed rules. Its contents can be replaced or flushed without reloading pf.conf. The management pipeline uses a privilege-separated queue pattern: CGI scripts running as www drop request files, root daemons process them and write results back.
  • The DAG rule parser builds a dependency graph of all custom rules. Deleting a rule automatically selects all dependent rules that would become dangling. The entire deletion set is validated with pfctl -nf before any live change is made.
  • The staging directory must be root:www 775. If set to 750, the pf_write_rules.pl CGI cannot write the deletion preview conf and returns a 500 error.
  • All addresses in this documentation are for the VM-based development environment and are for illustration purposes only. Update $ext_if and $int_if in pf.conf for real hardware; all rules reference these variables and require no further changes.
