Foreword

This document covers TNWatch from the operator perspective: what it monitors, how the alert and digest system works, how to query the event database, and how to manage alert rules. For the internal module API and platform-agnostic developer reference see /usr/local/share/doc/TN/dev/tnwatch.md. Read the TNAudit document first. TNWatch reads from TNAudit.db directly and the two systems are closely integrated.

TNWatch: Log Analysis and Alerting

TNWatch parses log files and service status data from across the system, stores normalised events in a SQLite database, evaluates alert rules against those events, and delivers email notifications. It produces two kinds of email: immediate alerts when a threshold is crossed, and a daily digest summarising the past 24 hours of activity across all monitored sources.

TNWatch runs on a one-minute cycle via cron. Each cycle it parses all log sources incrementally, reading only new lines since the last run, then evaluates all alert rules. The daily digest runs separately at 09:00.

Architecture

TNWatch consists of one command-line script and three modules:

/usr/local/sbin/TNWatch.pl
The main script. Dispatches to one of eleven commands: init-db, parse, parse-all, check-alerts, send-digest, query, stats, add-rule, test-email, purge, status.
/etc/TNWatch/TNWatchParser.pm
Log parsing engine. One parser per source. Each parser reads its log file incrementally using a byte-position state stored in the database, emits normalised event hashrefs, and returns a new state. The PF parser calls pfctl directly rather than reading a log file.
/etc/TNWatch/TNWatchDatabase.pm
All SQLite operations. Manages the events, alert_rules, digest_schedule, and parse_state tables. Provides event insertion, querying, alert rule management, parse state tracking, and maintenance functions.
/etc/TNWatch/TNWatchMail.pm
Email delivery. Renders HTML and plain-text versions of both the digest and immediate alert emails. Delivers via /usr/sbin/sendmail (femail on this system). All output is 7-bit ASCII; UTF-8 is stripped before sending.

The modules live at /etc/TNWatch/, outside the httpd chroot. TNWatch runs as root. It never runs inside the chroot and has no CGI counterpart.

The event database is at:

sh 
/var/www/htdocs/tn/data/db/TNWatch.db

Log Sources

TNWatch parses 13 log sources. Each source has its own parser in TNWatchParser.pm and its own position state in the parse_state table.

tnaudit
Reads TNAudit.db directly via SQLite. Emits a file_change event for every file with status other than ok, verified, or baseline. Always emits one integrity_check summary event per parse cycle.
pf
Calls pfctl -si and pfctl -s info directly; no log file is read. Records PF status, state table counts, inserts and removals per second, and state churn rate. Emits a pf_summary event every cycle and a block_spike event if more than 500 new states were inserted since the last cycle.
authlog
Reads /var/log/authlog. Captures SSH authentication failures, invalid user attempts, root login refusals, doas executions, and successful SSH logins. Disconnection lines are suppressed as too noisy.
httpd
Reads /var/www/htdocs/tn/data/logs/httpd/httpd_error.log. Captures 4xx and 5xx responses. 4xx responses to suspicious URIs (admin panels, PHP paths, environment files, shell tools) are elevated to warning severity.
tnwaf_access
Reads /var/www/htdocs/tn/data/logs/waf/access.log. Captures requests with suspicious URI patterns (XSS strings, SQL injection fragments, path traversal).
tnwaf_security
Reads /var/www/htdocs/tn/data/logs/waf/security.log. Captures WAF security events: rate limits, pattern blocks, XSS attempts, SQL injection attempts.
tnwaf_error
Reads /var/www/htdocs/tn/data/logs/waf/error.log. Captures HTTP 429, 400, 403, and 5xx responses from the WAF error stream. 404s are only captured if the URI is suspicious. 500-level errors are always critical severity.
snort
Reads /var/www/htdocs/tn/data/logs/snort/alert.log. Parses Snort fast alert format. Priority 1 alerts are critical, priority 2 are high, priority 3 and above are low.
e2guardian
Reads /var/www/htdocs/tn/data/logs/e2guardian/access.log. Captures DENIED and BLOCKED actions. Domains categorised as malware, phishing, or virus are critical severity.
unbound
Reads /var/www/htdocs/tn/data/logs/unbound/unbound.log. Captures NXDOMAIN and REFUSED responses.
syslog
Reads /var/www/htdocs/tn/data/logs/system/messages. Captures lines containing error, crit, emerg, alert, or panic keywords. Suppresses known noisy patterns (motd, prelogin). Skips integrity runner housekeeping lines.
daemon
Reads /var/www/htdocs/tn/data/logs/system/daemon. Captures signal-level events from smtpd, ntpd, dhcpd, unbound, pflogd, and collectd. Suppresses routine operational noise (peer adjustments, delivery confirmations, lease activity).
services
Reads /var/www/htdocs/tn/data/logs/bootlog/services.json, the live service status file updated every 15 seconds by the process monitor. Emits service_down, service_degraded, service_high_mem (above 10% memory), and services_summary events. Uses file mtime to skip re-parsing an unchanged file.

Incremental Parsing

Every parser stores the byte position of the last-read line in the parse_state table. On the next run the parser seeks to that position and reads only new lines. Running --parse-all or --parse <source> multiple times is safe; already-seen events are never duplicated.

If a log file is rotated and the stored position is larger than the current file size, the seek is skipped and parsing starts from the beginning of the new file. This prevents reading stale position data after a log rotation.

The PF and services parsers do not use byte positions. The PF parser tracks kernel counter values between runs to compute delta rates. The services parser compares the file mtime and skips parsing if it has not changed since the last run.

Cron Schedule

cron 
# Parse all sources and check alert rules every minute
* * * * *   root  /usr/local/sbin/TNWatch_runner.sh >/dev/null 2>&1

# Daily digest at 09:00
0 9 * * *   root  /usr/local/sbin/TNWatch.pl --send-digest

# Weekly purge: remove events older than 7 days (Sunday 08:00)
0 8 * * 0   root  /usr/local/sbin/TNWatch.pl --purge --days 7

The runner uses a lock file at /var/run/TNWatch.lock to prevent overlapping runs. If a previous run is still active when the next minute fires, the new invocation exits immediately. The lock is cleaned up on exit via a trap.

The runner log is at /var/log/TNWatch_runner.log. The main TNWatch script logs to /var/log/TNWatch.log. Both are separate files.

Alert Rules

Alert rules are stored in the alert_rules table in TNWatch.db. Each rule watches a specific source and event type. When the count of matching events in the rule's time window reaches the threshold, an immediate email is sent. An alert is suppressed if the same rule fired within the last window period; this prevents alert storms from a single sustained incident.

Eight default rules are seeded when the database is initialised:

tnaudit_change
Source: tnaudit, event: file_change. Threshold: 1 event in 300 seconds. Severity: critical. Fires immediately on the first integrity change detected.
pf_block_spike
Source: pf, event: block_spike. Threshold: 1 in 300 seconds. Severity: warning. Fires when more than 500 new PF states are inserted in a single parse interval.
tnwaf_rate_limit
Source: tnwaf, event: rate_limit. Threshold: 1 in 300 seconds. Severity: warning.
snort_critical
Source: snort, event: alert_critical. Threshold: 1 in 300 seconds. Severity: critical. Fires on any Snort priority-1 alert.
snort_high
Source: snort, event: alert_high. Threshold: 1 in 300 seconds. Severity: warning. Fires on any Snort priority-2 alert.
auth_failures
Source: authlog, event: auth_failure. Threshold: 5 in 600 seconds. Severity: warning. Fires on a burst of SSH authentication failures.
httpd_5xx_burst
Source: httpd, event: 5xx_error. Threshold: 10 in 300 seconds. Severity: warning.
service_down
Source: services, event: service_down. Threshold: 1 in 300 seconds. Severity: critical. Fires immediately when any monitored service stops.

Command Reference

Initialise the database

sh 
TNWatch.pl --init-db

Creates the database schema and seeds the default alert rules. Safe to run on an existing database; uses CREATE TABLE IF NOT EXISTS and INSERT OR IGNORE throughout. Run after first install or after upgrading TNWatch.

Parse log sources

sh 
# Parse all sources (what cron runs)
TNWatch.pl --parse-all

# Parse one source
TNWatch.pl --parse authlog
TNWatch.pl --parse tnaudit
TNWatch.pl --parse pf

Check alert rules

sh 
TNWatch.pl --check-alerts

Evaluates all enabled rules against recent events. Sends immediate email for any rule whose threshold is reached. Called by the runner after every --parse-all.

Query events

sh 
# All critical events in the last 24 hours
TNWatch.pl --query --severity critical

# SSH failures in the last 6 hours
TNWatch.pl --query --source authlog --event-type auth_failure --since 6h

# All events from a specific IP
TNWatch.pl --query --src-ip 203.0.113.42

# PF block spikes as JSON
TNWatch.pl --query --source pf --event-type block_spike --json

# Integrity changes in the last 7 days, limit 20
TNWatch.pl --query --source tnaudit --since 7d --limit 20

The --since option accepts: Xh (hours), Xd (days), Xm (minutes), Xw (weeks), or a raw Unix timestamp. The default window without --since is the last 24 hours.

Statistics

sh 
# Event counts grouped by source and severity (last 24h)
TNWatch.pl --stats

# Last 7 days
TNWatch.pl --stats --since 7d

# One source, as JSON
TNWatch.pl --stats --source tnwaf --since 48h --json

Add an alert rule

sh 
# Alert on SSH brute force: 20 failures in 5 minutes
TNWatch.pl --add-rule ssh_brute \
  --source authlog \
  --event-type auth_failure \
  --threshold 20 \
  --window 300 \
  --severity critical

# Alert on WAF flood: 100 rate limits in 60 seconds
TNWatch.pl --add-rule waf_flood \
  --source tnwaf \
  --event-type rate_limit \
  --threshold 100 \
  --window 60

Send the daily digest manually

sh 
TNWatch.pl --parse-all && TNWatch.pl --send-digest

Test email delivery

sh 
TNWatch.pl --test-email
# Then check the mail UI at /tn/view/mail

Sends a digest with synthetic test data to root. Does not use live database data. Use this after install or after mail configuration changes to confirm delivery is working.

Operational status

sh 
TNWatch.pl --status

Shows database path, size, event count, oldest and newest event timestamps, and all active alert rules with their thresholds and windows.

Purge old events

sh 
# Default: purge events older than 90 days
TNWatch.pl --purge

# Aggressive retention for space-constrained hardware
TNWatch.pl --purge --days 7

The cron schedule runs a 7-day purge every Sunday at 08:00, aligned with the RPi 5 deployment target where database growth must be managed. On larger hardware increase this to 30 or 90 days.

The Daily Digest

The digest covers the previous 24 hours and includes sections for: file integrity (TNAudit status, changed file list), PF (firewall status, state table counts, block spikes), HTTP errors, TNWAF events (rate limits, pattern blocks, XSS and SQLi attempts), services (running/down/degraded counts), Snort alerts by severity, e2guardian blocks, unbound NXDOMAIN and refused counts, and authentication (SSH failures, successes, doas executions, root login attempts, top source IPs).

The subject line is appended with [!] CHANGES DETECTED if any of the following conditions are true: TNAudit reports changed files, any service is down, any Snort critical alert fired, or any root login was attempted. This makes the digest immediately identifiable in a mailbox without opening it.

Email is delivered to root via /usr/sbin/sendmail (femail on this system). The mail is readable in the Tangent Networks mail UI at /tn/view/mail. All email content is filtered to 7-bit ASCII before delivery; no UTF-8 characters appear in the output regardless of what the log files contain.

Typical Diagnostic Workflows

Investigate recent SSH failures:

sh 
TNWatch.pl --query --source authlog --severity critical --since 6h

Check what fired overnight:

sh 
TNWatch.pl --stats --since 24h

Find all events from a suspected source IP:

sh 
TNWatch.pl --query --src-ip 203.0.113.42 --since 7d --json

Check integrity status directly from the TNWatch event database:

sh 
TNWatch.pl --query --source tnaudit --since 7d --json

Troubleshooting

Check that the database, modules, and executables are present:

sh 
ls -l /var/www/htdocs/tn/data/db/TNWatch.db
ls -l /etc/TNWatch/
ls -l /usr/local/sbin/TNWatch.pl /usr/local/sbin/TNWatch_runner.sh

Confirm the database is healthy and contains events:

sh 
TNWatch.pl --status

Check the runner log for recent activity and errors:

sh 
tail -40 /var/log/TNWatch_runner.log
tail -40 /var/log/TNWatch.log

Check that the lock file is not stale. A stale lock prevents the runner from executing.

sh 
cat /var/run/TNWatch.lock
# If the PID is not running, the lock is stale. Remove it:
rm /var/run/TNWatch.lock

If the database is stale or the events table is missing after a schema change:

sh 
# TNWatch.db is rebuilt from live log sources on every parse-all.
# It is safe to delete and re-initialise.
rm /var/www/htdocs/tn/data/db/TNWatch.db
TNWatch.pl --init-db
TNWatch.pl --parse-all

Test email delivery end-to-end:

sh 
TNWatch.pl --test-email

TNWatch.db is a derived database. All its content can be reconstructed from the live log files by running TNWatch.pl --parse-all. It does not need to be backed up with the same priority as TNAudit.db, auth.db, or the configuration files. The audit history in TNAudit.db is the primary integrity record.

Summary

  • TNWatch parses 13 log sources every minute, stores normalised events in TNWatch.db, and evaluates 8 default alert rules after every parse cycle. Immediate email alerts fire when a threshold is crossed.
  • The daily digest runs at 09:00. It covers file integrity, PF state, HTTP errors, WAF events, services, Snort, e2guardian, unbound, and SSH authentication. The subject line is flagged when any critical condition is present.
  • All parsing is incremental. Byte position per source is stored in the parse_state table. Re-running any parse command is safe and never duplicates events.
  • TNWatch.db is a derived cache. It can be deleted and rebuilt from live logs at any time with --init-db followed by --parse-all.
  • Events are purged after 7 days by a cron job every Sunday at 08:00. Increase the retention period on hardware with more storage.
  • The runner lock file is at /var/run/TNWatch.lock. A stale lock prevents the runner from executing. Remove it manually if the previous run exited without cleaning up.
