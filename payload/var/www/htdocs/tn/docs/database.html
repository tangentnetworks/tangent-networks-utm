Foreword

This document covers the auth.db schema: every table, every column, the index strategy, the views, and the operational procedures for initialisation, onboarding, and emergency recovery. Read this before the module documents. TNAuth, TNSecurity, TNConfig, and TNSecurityCheck all operate against this schema and their behaviour only makes sense with it in front of you.

Database and Schema

The authentication database is a single SQLite file:

text 
/var/www/htdocs/tn/data/db/auth.db

Inside the chroot the same path resolves as /htdocs/tn/data/db/auth.db. The file is owned www:www, mode 0600. No other database file is used by the authentication and session layer.

The schema lives at /var/www/htdocs/tn/data/db/schema.sql and is applied dynamically by the installer tools. The current version is 2.1.0.

Runtime Configuration

Every connection opens with two pragmas applied immediately:

sql 
PRAGMA foreign_keys = ON;
PRAGMA journal_mode = WAL;

foreign_keys = ON is mandatory. Without it SQLite silently ignores all ON DELETE CASCADE rules, so child rows (sessions, security questions, recovery codes) would not be cleaned up when a user is deleted. The application depends entirely on the database enforcing these constraints; it contains no application-layer cascade logic.

journal_mode = WAL permits one writer and multiple concurrent readers simultaneously. This is required for the slowcgi multi-process model where several worker processes may be reading session data at the same moment a login is being written.

Connections also set RaiseError => 1 and sqlite_use_immediate_transaction => 1. RaiseError converts all database failures into fatal exceptions rather than silently returning undef, preventing silent data corruption. The immediate transaction mode acquires a write lock at the start of every transaction rather than on the first write, which eliminates "database is locked" errors under concurrent write load from slowcgi workers.

Tables

users

One row per registered account. The primary key is a 64-character hex string generated by TNSecurity::generate_token(16), which is sha256_hex(random_string . time()). Text primary keys are used throughout rather than integer rowids to avoid predictable ID enumeration.

pf 
CREATE TABLE IF NOT EXISTS users (
    id              TEXT    PRIMARY KEY,
    username        TEXT    UNIQUE NOT NULL,
    email           TEXT    UNIQUE,
    password_hash   TEXT    NOT NULL,
    salt            TEXT    NOT NULL,
    role            TEXT    NOT NULL DEFAULT 'user',
    created_at      INTEGER NOT NULL,
    last_login      INTEGER,
    login_count     INTEGER DEFAULT 0,
    locked_until    INTEGER DEFAULT 0,
    locked          INTEGER DEFAULT 0,
    failed_attempts INTEGER DEFAULT 0,
    CONSTRAINT chk_role CHECK (role IN ('user', 'admin'))
);
password_hash
Carries an algorithm prefix. Current hashes begin with pbkdf2: followed by hex-encoded derived key bytes (PBKDF2-HMAC-SHA256, 100,000 iterations, 32-byte output). Legacy hashes are plain 64-character hex from the old iterated SHA-256 scheme and are transparently upgraded on the next successful login.
salt
A 32-character alphanumeric string generated from /dev/urandom.
email
Nullable; not required at registration. SQLite's standard behaviour that NULL != NULL means multiple accounts without email addresses do not violate the uniqueness constraint.
locked / locked_until
Both integers. When locked = 1 and locked_until = 0 the lock is permanent until cleared manually. When locked_until is a future epoch the lock expires automatically on the next login attempt; no cron job is required to clear it.
Timestamps
All timestamp columns (created_at, last_login, locked_until) are INTEGER Unix epoch seconds, never ISO strings.

Indexes: idx_users_username (login lookup), idx_users_email (registration duplicate check), idx_users_role (admin token listing).

sessions

One row per active session. Session IDs are 64-character hex strings. The value stored here is the raw session ID, not the signed cookie value. The cookie holds session_id.hmac_signature. TNSecurity verifies the signature and strips it before any database lookup occurs.

pf 
CREATE TABLE IF NOT EXISTS sessions (
    session_id    TEXT    PRIMARY KEY,
    user_id       TEXT    NOT NULL,
    created_at    INTEGER NOT NULL,
    last_activity INTEGER NOT NULL,
    expires_at    INTEGER NOT NULL,
    ip_address    TEXT,
    user_agent    TEXT,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

Session lifetime is read from security.conf [session] SESSION_LIFETIME, floored at 900 seconds and capped at 86400 seconds. The default is 7200 seconds (2 hours). last_activity is updated on every validated request. expires_at is fixed at creation time and is not extended by activity; it is an absolute expiry, not a sliding idle timeout.

Sessions created by create_session.pl from the command line are identifiable in the table by ip_address = '127.0.0.1' and user_agent = 'CLI'.

Indexes: idx_sessions_user_id, idx_sessions_expires_at (cleanup), idx_sessions_last_activity.

security_questions

Stores hashed security question answers for the password reset flow. Each user may have one or more rows. Answers are hashed with the same PBKDF2 algorithm used for passwords, with an independent salt per answer in answer_salt.

pf 
CREATE TABLE IF NOT EXISTS security_questions (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id     TEXT    NOT NULL,
    question    TEXT    NOT NULL,
    answer_hash TEXT    NOT NULL,
    created_at  INTEGER NOT NULL,
    answer_salt TEXT,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE
);

The answer_salt column was added in schema v2.0.1. Rows created before that migration may carry a NULL salt; TNSecurity::verify_password() handles the legacy path transparently. Index: idx_security_questions_user_id.

recovery_codes

Single-use recovery codes for account access when security questions are unavailable. Stored as sha256_hex(code) with no salt. No salt is needed because recovery codes are long random strings from /dev/urandom and are not susceptible to dictionary attack. The hashing contract is defined by TNSecurity::hash_recovery_code() and TNSecurity::verify_recovery_hash().

pf 
CREATE TABLE IF NOT EXISTS recovery_codes (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id    TEXT    NOT NULL,
    code_hash  TEXT    NOT NULL,
    used       INTEGER NOT NULL DEFAULT 0,
    used_at    INTEGER,
    created_at INTEGER NOT NULL,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE CASCADE,
    CONSTRAINT chk_used CHECK (used IN (0, 1))
);

Once used, used = 1 and used_at is stamped. Codes are never deleted on use; the audit trail is preserved. Indexes: idx_recovery_codes_user_id, idx_recovery_codes_used.

registration_tokens

Invitation tokens for controlled user registration. After the first admin account is created, all subsequent registrations require a token. Tokens are generated by an admin via the API and distributed out of band.

pf 
CREATE TABLE IF NOT EXISTS registration_tokens (
    token       TEXT    PRIMARY KEY,
    created_by  TEXT,
    created_at  INTEGER NOT NULL,
    used        INTEGER NOT NULL DEFAULT 0,
    used_by     TEXT,
    used_at     INTEGER,
    FOREIGN KEY (created_by) REFERENCES users(id) ON DELETE SET NULL,
    FOREIGN KEY (used_by)    REFERENCES users(id) ON DELETE SET NULL,
    CONSTRAINT chk_used CHECK (used IN (0, 1))
);

Both foreign keys use ON DELETE SET NULL so the token audit trail survives deletion of the issuing or using account. Indexes: idx_registration_tokens_used, idx_registration_tokens_created_by.

rate_limits

Tracks per-identifier rate limiting for the password reset flow. The identifier is a username string. The limit_type value is 'password_reset'. The unique constraint on (identifier, limit_type) enables atomic upsert via INSERT ... ON CONFLICT DO UPDATE.

pf 
CREATE TABLE IF NOT EXISTS rate_limits (
    id            INTEGER PRIMARY KEY AUTOINCREMENT,
    identifier    TEXT    NOT NULL,
    limit_type    TEXT    NOT NULL,
    count         INTEGER NOT NULL DEFAULT 0,
    window_start  INTEGER NOT NULL,
    violations    INTEGER NOT NULL DEFAULT 0,
    last_request  INTEGER NOT NULL,
    blocked_until INTEGER DEFAULT 0,
    CONSTRAINT uq_identifier_type UNIQUE (identifier, limit_type)
);

When blocked_until is a future epoch the identifier is blocked. The block clears automatically when the epoch passes. Thresholds and lockout duration are read from security.conf [rate_limit]. Indexes: idx_rate_limits_identifier, idx_rate_limits_type, idx_rate_limits_blocked_until.

security_log

Append-only audit log written by TNSecurity::log_security_event(). Never truncated by the application. The user_id foreign key uses ON DELETE SET NULL so log entries survive user deletion.

pf 
CREATE TABLE IF NOT EXISTS security_log (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp  INTEGER NOT NULL,
    level      TEXT    NOT NULL,
    event      TEXT    NOT NULL,
    details    TEXT,
    ip_address TEXT,
    user_id    TEXT,
    username   TEXT,
    FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE SET NULL,
    CONSTRAINT chk_level CHECK (level IN
        ('debug','info','warning','error','critical'))
);

Indexes: idx_security_log_timestamp, idx_security_log_level, idx_security_log_user_id, idx_security_log_event.

Views

Three read-only views are defined for common queries. They are never written to by the application.

v_active_sessions
Joins sessions and users. Returns all sessions where expires_at > now(). Columns include username, role, ip_address, user_agent, and time_remaining in seconds.
v_user_stats
Per-user aggregate: active session count, unused recovery code count, login count, last login, and failed attempt count. Used by the admin user management panel.
v_security_summary
Security log events grouped by calendar date and level, ordered newest first. Used by the dashboard security widget.

Initialisation and User Onboarding

When a user installs the UTM appliance, the core package management tool TN_PKG_INSTALL.sh handles the automated initialization of the database (Stage 13). At this point, the schema is applied, paths are provisioned, but no users exist in the system database.

To complete the setup initialization, the system administrator must perform the following onboarding sequence:

  1. Open a local browser and navigate to the appliance setup interface endpoint: https://%%INT_IP4%%/register.html
  2. Complete the submission form to create the primary root administrative profile.
  3. Upon successful submission, the interface displays the critical account Recovery Codes alongside multi-user registration Invite Codes. The invite codes are utilized subsequently to authorize and onboard secondary system administrators.

Emergency Recovery Utilities

When the web user interface becomes completely inaccessible, administrative maintenance is conducted via three standalone, Taint-safe Perl scripts located in /usr/local/sbin/. These rescue tools run outside the chroot boundaries and must be executed by the root user.

1. In-Place Account Rescue (user_rescue.pl)

If an administrative user has forgotten their password or becomes locked out due to exceeding the maximum allowed failed authentication attempts, running user_rescue.pl provides an interactive console workflow to triage and repair the account record.

Operational Boundary Constraint: This script allows the root console operator to reset user passwords and clear locked status variables. However, it will NOT permit the regeneration or creation of registration invite codes or account recovery codes.

sh 
# perl -T /usr/local/sbin/user_rescue.pl

2. Cold Credential Hard Reset (create_user.pl)

If account recovery sequences are exhausted or the primary administrator account is unrecoverable, a complete identity hard reset is required. The operator manually removes the active SQLite database file to wipe the lockout state, and executes create_user.pl with the schema SQL file to completely re-initialize the data store and force-inject a brand-new administrator account directly from the shell.

sh 
# Clear the active auth database to break out of lockout
# rm /var/www/htdocs/tn/data/db/auth.db

# Hard-provision the baseline schema and first administrator account
# perl -T /usr/local/sbin/create_user.pl --schema /var/www/htdocs/tn/data/db/schema.sql

3. CLI Session Cookie Generation (create_session.pl)

Utilized for debugging, verification tasks, or gaining immediate interface access during web login frontend asset failures. This tool securely verifies user credentials on the local terminal, registers a valid tracking row inside the database sessions table, and outputs a cryptographically signed cookie payload.

Sessions created via this utility are explicitly flagged in the tracking logs with an source IP address of 127.0.0.1 and a User-Agent signature of CLI.

sh 
# perl -T /usr/local/sbin/create_session.pl

Performing a hard reset via create_user.pl requires dropping the database file, which permanently clears out any existing user definitions, active security questions, and invite profiles. Ensure you have verified your system paths before destroying state files.

Manual Queries

sh 
# List all users with role and lock state
sqlite3 /var/www/htdocs/tn/data/db/auth.db \
  "SELECT username, role, locked, failed_attempts,
          datetime(last_login,'unixepoch') FROM users"

# List active sessions
sqlite3 /var/www/htdocs/tn/data/db/auth.db \
  "SELECT username, role, ip_address,
          datetime(expires_at,'unixepoch') AS expires
   FROM v_active_sessions"

# Force logout all users
sqlite3 /var/www/htdocs/tn/data/db/auth.db "DELETE FROM sessions"

# Unlock a locked account
sqlite3 /var/www/htdocs/tn/data/db/auth.db \
  "UPDATE users SET locked=0, failed_attempts=0 WHERE username='alice'"

# Recent security events
sqlite3 /var/www/htdocs/tn/data/db/auth.db \
  "SELECT datetime(timestamp,'unixepoch'), level, event, details
   FROM security_log ORDER BY timestamp DESC LIMIT 20"

# Count unused registration tokens
sqlite3 /var/www/htdocs/tn/data/db/auth.db \
  "SELECT COUNT(*) FROM registration_tokens WHERE used=0"

Direct sqlite3 writes bypass all application-layer validation and the PBKDF2 hashing contract. Utilize user_rescue.pl for in-place account maintenance. Reserve raw terminal database writes exclusively for emergency session truncation and fast user account unlocking tasks.

Summary

  • Single SQLite file at data/db/auth.db, owned www:www, mode 0600. Every connection enables foreign keys and WAL mode. RaiseError and immediate transactions are always on.
  • Seven tables: users, sessions, security_questions, recovery_codes, registration_tokens, rate_limits, security_log. All timestamps are INTEGER Unix epoch seconds.
  • PRAGMA foreign_keys = ON is mandatory at every connection. Without it SQLite silently ignores all ON DELETE CASCADE rules.
  • CSRF tokens have no database table. They are stateless HMAC values derived from a time window and validated entirely in memory by TNSecurity.
  • Initial automated setup is deployed via TN_PKG_INSTALL.sh. The administrative user onboards via /register.html to extract initialization codes and recovery seeds.
  • Emergency account recovery utilizes user_rescue.pl for interactive triage. Cold-purged initializations run via create_user.pl, and create_session.pl bypasses broken logins via signed cookie streams.
  • The security_log table is append-only and never truncated. All emergency tool operations write EMERGENCY_-prefixed events to it.
