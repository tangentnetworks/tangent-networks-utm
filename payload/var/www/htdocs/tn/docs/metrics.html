Foreword

This document covers the three subsystems that discover and report the live state of the system to the dashboard: firewall health collection via pfhealth.pl, system metrics collection via collectd_exporter.pl, and service status monitoring via monitor.pl. All three follow the same architectural pattern -- a Perl script collects data, writes JSON to a known path under data/stats/ or data/logs/bootlog/, and a shell runner keeps the script executing at a fixed interval. The dashboard reads the JSON files. Nothing communicates over TCP. If a panel is stale or dark, the first question is always whether the relevant runner is alive.

Metrics & Discovery

Overview

Three independent scripts feed the dashboard with live data. Each is taint-safe Perl (perl -T), writes output atomically to a fixed path, and runs inside a persistent shell loop rather than cron because cron's minimum granularity of one minute is too coarse for a live dashboard. All three runners follow the same pattern: write PID on start, trap exit to clean up the PID file, loop indefinitely with a fixed sleep interval.

The three scripts are deliberately independent. pfhealth.pl knows nothing about collectd_exporter.pl. A failure in one does not affect the others. The dashboard reads each output file directly without any aggregation layer between the scripts and the UI.

pfhealth.pl -- Firewall Health Collection

pfhealth.pl queries the OpenBSD packet filter from three sources on each run and writes a single JSON file used by the firewall statistics panel. It runs every six seconds via pf_stats_runner.sh.

The three data sources are systat -n pf for PF status and uptime, pfctl -si for state table metrics, and pfctl -vvsr for per-rule counters. Each source contributes a distinct section to the output JSON.

State Table Metrics

From pfctl -si the script extracts four counters: current entries (active state table entries at this moment), searches (lifetime state lookups since boot), inserts (states created since boot), and removals (states destroyed since boot). The dashboard displays current entries as the primary active states figure. Inserts minus removals gives net state growth -- a persistently growing number suggests a state leak or a SYN flood in progress.

Rule Efficiency Analysis

From pfctl -vvsr the script reads every rule's evaluation count, packet count, byte count, and active state count. For each rule it calculates an efficiency ratio: evaluations divided by packets matched. A low ratio means the rule matches most of what it evaluates -- it is in the right place in the ruleset. A high ratio means the rule is being evaluated against a lot of traffic that it never matches -- either the rule should be moved higher in the ruleset, given a quick keyword, or its match criteria should be refined.

The classification thresholds are: a ratio below 5 is Efficient; between 5 and 20 is Review; above 20 is Optimize. A rule with evaluations greater than zero but zero packets matched is classified as Unused -- it has never matched anything and is wasting evaluation cycles on every packet. Bytes are converted to IEC units (KiB, MiB, GiB using base-1024) for display.

Output

data/stats/pf_stats.json
Written on every run. Permissions www:www 0644. Contains a timestamp, PF status and since fields, the four state table counters, and the full per-rule array with efficiency classification. The dashboard firewall panel reads this file every 15 seconds.

Runner

pf_stats_runner.sh runs pfhealth.pl every six seconds. Six seconds was chosen because PF counters change rapidly under load, the script completes in under 100ms, and the browser refreshes at 15 seconds -- data is never more than six seconds stale when the browser fetches it. The runner does not use set -e, so a Perl error on one iteration does not kill the loop.

data/run/webui/pf_stats_runner.pid
PID of the running pf_stats_runner.sh process.

collectd_exporter.pl -- System Metrics

collectd_exporter.pl bridges collectd's Unix socket interface and the dashboard's JSON layer. collectd runs continuously and maintains live metric values accessible via its socket. The exporter queries the socket on demand and writes structured JSON to per-plugin output files. If the socket is unavailable -- for example because collectd is restarting -- the companion module collectd_rrd_exporter.pm reads the last known values from collectd's RRD files and returns them in the same JSON format. The dashboard never sees a failure; the source field in each metric record indicates whether the value came from the live socket (unixsock) or from the archive (rrd).

collectd Configuration

collectd must have the unixsock plugin loaded with the socket file pointing to data/sockets/collectd/collectd.sock, and the rrdtool plugin loaded with its base directory pointing to data/rrd/collectd. The Hostname directive in collectd.conf must be set to tangent -- this value is used to construct metric IDs and RRD file paths, and a mismatch will cause all socket queries and RRD lookups to fail silently.

The CPU aggregation plugin is configured to average across all CPU cores and expose the result under the cpu_avg-cpu-average plugin instance. The interface plugin monitors all interfaces except lo0, enc0, and pflog0 -- so %%EXT_IF%% (WAN) and %%INT_IF%% (LAN) are the monitored interfaces. The processes plugin tracks 19 named daemons by exact process name.

Execution Modes

The exporter supports four modes selected by how it is invoked. With no arguments it exports all registered plugins into a single aggregate JSON file. With one or more plugin names as arguments it exports only those plugins, each to its own output file. In CGI mode (when QUERY_STRING is set) it accepts an action=plugin parameter, writes the output file, and also prints the JSON to stdout. Unknown plugin names are rejected in all modes -- only names present in the internal metrics registry are accepted.

RRD Fallback

collectd_rrd_exporter.pm is invoked transparently whenever the Unix socket is unreachable, returns an error, or returns no parseable data. It looks up the RRD file path for each metric, fetches the last five minutes of AVERAGE data, and scans backwards to find the last non-NaN value. Multi-DS RRDs (such as disk I/O with separate read and write data sources) return one value per data source. The source field in the output JSON will read rrd when fallback is active. OpenBSD's collectd PF plugin creates RRD filenames that may contain literal spaces -- the fallback module handles these with an explicit path mapping.

Output

data/stats/collectd_exporter.json
Full aggregate JSON containing all plugins. Written when the exporter is invoked with no arguments.
data/stats/collectd/<plugin>/metrics
Per-plugin JSON output. Written when a specific plugin is requested. The dashboard_stats_runner.sh writes cpu, memory, df, and interfaces here every six seconds.

Runner

dashboard_stats_runner.sh runs a batch export of cpu memory df interfaces every six seconds. These are the four panels that update in near-real-time on the dashboard. Other plugins -- pf, ping, processes, system -- can be exported on demand via CGI or added to a separate lower-frequency runner.

monitor.pl -- Service Status

monitor.pl polls every configured service every six seconds and writes three outputs simultaneously: a plain-text log for manual inspection, a consolidated JSON file read by the Services panel, and individual per-service JSON status files for services that the dashboard fetches independently. Service definitions live in a separate configuration module at /etc/tangent_services.pm -- adding or removing a monitored service requires only a one-line change to that file, with no changes to the main script.

Service Types

Each service in the configuration is either a pid_file service or an rcctl service. For pid_file services the monitor reads the named PID file to get the process ID and then queries ps for metrics. The PID file must exist, be readable, and contain only a numeric PID. For rcctl services the monitor uses pgrep -n with the configured process name. The process name must match what pgrep sees -- this may differ from the rcctl service name (for example ftp-proxy is configured as name => "ftp-proxy").

For each running process the monitor collects PID, user, CPU%, memory%, RSS, VSZ, command, arguments, and TCP port listeners via netstat -an -p tcp. All subprocess calls use the list form of open("-|", ...) rather than backticks, which avoids spawning a shell and prevents injection.

Aggregation Groups

Related processes can be grouped into a single logical entry in the JSON output using the %aggregation hash in tangent_services.pm. Services listed in an aggregation group are excluded from standalone JSON output and appear only inside their group. The group status is running if all member services are running, degraded if some but not all are running, and stopped if none are running. The pmacct three-instance architecture is expressed as one aggregation group with three member services.

Output

data/logs/bootlog/services.json
Consolidated JSON snapshot. Contains a timestamp and a map of all services. Aggregated groups appear as single entries with a subprocesses array and rolled-up aggregate metrics. This is the primary feed for the Services panel in the dashboard.
data/logs/bootlog/services.log
Plain-text status log. One block per service in alphabetical order by key. Useful for quick manual inspection. Written on every run.
data/services/status/<key>/status
Individual JSON status file per service key or aggregation group key. Written on every run for consumption by dashboard components that fetch service status independently.

Runner

service_monitor_runner.sh invokes monitor.pl every six seconds. Because monitor.pl runs to completion on each invocation (it is not a daemon), the effective poll rate is the sleep interval plus script execution time. On a typical system monitor.pl completes in well under one second, so the effective rate closely tracks six seconds. A non-zero exit from monitor.pl does not abort the loop -- the runner continues regardless.

The Runner Pattern

All three runners share the same design. It is worth understanding because the same pattern is used across all 17 WebUI runners in the system.

On start the runner writes its own PID to a known file and registers a trap that removes the PID file on any exit. It then loops indefinitely: run the script, sleep for the configured interval, repeat. There is no set -e -- a script failure on one iteration does not kill the loop. Stdout and stderr are redirected to /dev/null when launched from start_service in rc.local -- errors appear in the relevant service log, not on the console.

The interval of six seconds was chosen because script execution times are well under 100ms, the browser refresh rate for live panels is 15 seconds, and six seconds guarantees data is never more than six seconds stale when the browser fetches it. It is not a magic number -- the sleep value in any runner can be adjusted freely without touching the monitored script.

The PID file serves two purposes. Externally, other scripts and the dashboard use it to determine whether a runner is alive. Internally, rc.local's start_service function reads it to guard against double-starting a runner that is already running. If the PID file exists and the process is alive, start_service returns without launching a second instance. If the PID file exists but the process is gone -- a stale file from a crash -- start_service removes it and proceeds with a fresh start.

Security

All three scripts run under Perl taint mode (perl -T). This means all data from external sources -- environment variables, command-line arguments, file contents, subprocess output -- is treated as tainted and must pass through an explicit regex validation before use in any system call or file operation. The PATH is set to a minimal safe value at startup and IFS, CDPATH, ENV, and BASH_ENV are deleted. CGI input in collectd_exporter.pl is validated against the metrics registry before any processing -- an unknown plugin name causes an immediate error rather than a silent skip.

Recovery

sh 
# Check if a runner is alive
kill -0 $(cat /var/www/htdocs/tn/data/run/webui/pf_stats_runner.pid) \
    && echo "running" || echo "dead"

# Restart pf_stats_runner manually
kill $(cat /var/www/htdocs/tn/data/run/webui/pf_stats_runner.pid) 2>/dev/null
/usr/local/sbin/pf_stats_runner.sh &

# Restart dashboard_stats_runner manually
kill $(cat /var/www/htdocs/tn/data/run/webui/dashboard_stats_runner.pid) 2>/dev/null
/usr/local/sbin/dashboard_stats_runner.sh &

# Restart service_monitor_runner manually
kill $(cat /var/www/htdocs/tn/data/run/webui/service_monitor_runner.pid) 2>/dev/null
/usr/local/sbin/service_monitor_runner.sh &

# Run pfhealth.pl manually and check output
/usr/local/sbin/pfhealth.pl
cat /var/www/htdocs/tn/data/stats/pf_stats.json | grep TS

# Export a single collectd plugin manually
/usr/local/sbin/collectd_exporter.pl cpu
cat /var/www/htdocs/tn/data/stats/collectd/cpu/metrics

# Check collectd socket exists
ls -la /var/www/htdocs/tn/data/sockets/collectd/collectd.sock

# Check collectd exporter log
tail -20 /var/www/htdocs/tn/data/logs/collectd/exporter.log

# Run monitor.pl manually and check output
/usr/local/sbin/monitor.pl
cat /var/www/htdocs/tn/data/logs/bootlog/services.json | grep timestamp

Summary

  • pfhealth.pl queries PF state tables and per-rule counters every six seconds and classifies rule efficiency automatically -- no manual pfctl parsing required to identify ruleset bottlenecks.
  • collectd_exporter.pl provides live system metrics via collectd's Unix socket with automatic transparent fallback to RRD files if collectd is temporarily unavailable.
  • monitor.pl tracks 19 system daemons, aggregates related processes into logical groups, and writes structured JSON that feeds the Services panel every six seconds.
  • All three use the same runner pattern: write PID, trap exit, loop indefinitely. Understanding one runner means understanding all 17.
  • If a dashboard panel is stale or dark, check the corresponding runner PID file first. If the runner is alive and the panel is still stale, check the output file timestamp and then the relevant log.
