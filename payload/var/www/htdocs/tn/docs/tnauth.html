Foreword

TNAuth is the user and session management layer. It handles registration, authentication, session creation and validation, security questions, recovery codes, registration tokens, and password reset rate limiting. All database access for these operations goes through TNAuth. This document covers each subsystem, the account lifecycle, and the rate limiting and lockout mechanics.

TNAuth

Location: /var/www/htdocs/tn/data/lib/TNAuth.pm
Version: 2.2.0
Database: data/db/auth.db

Database Connection

get_db_handle() opens a new connection to auth.db on every call. In the slowcgi CGI model each request is a short-lived process, so connection pooling is not used. Every connection sets:

  • PRAGMA foreign_keys = ON -- required for cascade deletes.
  • PRAGMA journal_mode = WAL -- required for concurrent reads during writes.
  • RaiseError => 1 -- all failures are fatal exceptions.
  • sqlite_use_immediate_transaction => 1 -- acquires write lock at transaction start, preventing "database is locked" errors under concurrent slowcgi worker load.

User Registration

register_user($username, $password, $email, $security_questions, $token)

The first user registered becomes the admin. All subsequent registrations require a valid unused registration token. check_registration_status() determines which path applies by counting rows in the users table.

Registration steps:

  1. Check registration status (first user or token required).
  2. If a token is required, validate it via validate_registration_token().
  3. Check for duplicate username and email (email check is skipped when email is blank).
  4. Hash the password with TNSecurity::hash_password().
  5. Generate a 64-character hex user ID with TNSecurity::generate_token(16).
  6. Assign role: 'admin' for first user, 'user' for all others.
  7. Insert the users row with created_at = time().
  8. Insert security question rows, each with its own PBKDF2-hashed answer and salt.
  9. Mark the registration token as used (sets used = 1, used_at, used_by).

Email uniqueness is only checked when an email address is supplied. Passing an empty string is treated the same as NULL -- the unique constraint is not checked against other null-email rows.

Authentication

authenticate_user($username, $password)

Authentication steps:

  1. DEVEL mode check: if DEVEL is active and the password is the literal string 'DEVEL_BYPASS', return a synthetic admin session without any database lookup. The bypass is logged as a warning.
  2. Fetch user record from the database by username.
  3. Check lockout state. If locked = 1 and locked_until is a future epoch, return failure. If locked_until has passed, clear the lock automatically (sets locked = 0, locked_until = 0, failed_attempts = 0).
  4. Verify password with TNSecurity::verify_password(). On failure, increment failed_attempts. If the new count reaches the configured threshold, set locked = 1 and locked_until = time() + LOCKOUT_DURATION.
  5. On success: check needs_rehash(). If the stored hash uses the legacy algorithm, re-hash with PBKDF2 immediately and update the database.
  6. Reset failed_attempts = 0, update last_login, increment login_count.
  7. Return success with user_id, username, and role.

The lockout threshold is read from security.conf [rate_limit] MAX_LOGIN_ATTEMPTS. TNAuth enforces a floor of 3 and a ceiling of 10 regardless of the configured value. The lockout duration has a floor of 300 seconds.

Session Management

create_session

create_session($user_id, $ip, $user_agent)

Generates a 64-character hex session ID, calculates expiry from the SESSION_LIFETIME config value (floored at 900s, capped at 86400s), and inserts a row into the sessions table. Returns the raw session ID. The caller (control.pl) then passes it to TNSecurity::sign_session_id() to produce the cookie value.

validate_session

validate_session($session_id)

Queries sessions joined with users where session_id = ? and expires_at > time(). Updates last_activity on every valid hit. Returns a hashref with session_id, user_id, username, role, and session_age (seconds since creation). Returns undef if the session is absent or expired.

Note that the session ID passed here is the raw ID after signature verification by TNSecurity. TNAuth never sees the signed cookie value.

destroy_session

destroy_session($session_id)

Deletes the session row. Called by the logout handler. Does not affect other sessions for the same user.

cleanup_expired_sessions

cleanup_expired_sessions()

Deletes all sessions where expires_at < time(). Called by the session cleanup cron job (every Monday at 15:01 via TNAuth::cleanup_expired_sessions() invoked from a scheduled script). Returns a count of deleted rows.

Security Questions

get_security_questions($username) returns an arrayref of question strings for the given user. Used by the password reset flow to present questions to the user without revealing the answers.

verify_security_answers($username, $answers) accepts a hashref mapping question strings to answer strings. Each answer is verified against its stored PBKDF2 hash with TNSecurity::verify_password(). All stored questions must have a correct answer supplied -- a single incorrect or missing answer returns failure.

The available question list returned by get_available_questions() is hardcoded in the module:

  • What was your first pet's name?
  • What city were you born in?
  • What is your mother's maiden name?
  • What was the name of your first school?
  • What is your favourite book?

Recovery Codes

verify_recovery_code($username, $code) fetches all unused codes for the user and iterates through them, calling TNSecurity::verify_recovery_hash() for each. On a match, the code is marked used (used = 1, used_at = time()). The search always iterates all codes rather than stopping early, to avoid timing attacks that could distinguish between "no codes exist" and "codes exist but none match".

Registration Tokens

generate_registration_token() creates a new 64-character hex token and inserts it into the registration_tokens table. Returns the token value to the caller (admin API handler) for distribution.

get_unused_tokens() returns all tokens where used = 0, ordered newest first. Used by the admin token management UI.

validate_registration_token($token) checks that the token exists and has used = 0. Token consumption (marking used) happens inside register_user() after successful account creation.

User Deletion

delete_user($user_id) runs in an explicit transaction. It manually deletes child rows in order (security_questions, recovery_codes, sessions) before deleting the users row. The explicit deletes are belt-and- suspenders alongside the ON DELETE CASCADE foreign keys -- the comment in the source notes that PRAGMA foreign_keys = ON must be set on every connection for the cascades to fire. On any error the transaction is rolled back and a failure result is returned.

Password Reset Rate Limiting

check_reset_attempts($username) uses the rate_limits table to track password reset attempts per username. The logic:

  1. Read the current row for (username, 'password_reset').
  2. If blocked_until is a future epoch, return { allowed => 0 }.
  3. If the count is at or above the max but the block has expired, reset the counter and allow the request.
  4. Otherwise increment the counter via upsert. If the new count exceeds the max, set blocked_until = time() + LOCKOUT_DURATION.

clear_reset_attempts($username) deletes the rate_limits row for the username. Called by control.pl after a successful password reset to reset the counter so the user is not rate-limited on their next legitimate reset request.

Password Update

update_password($user_id, $new_password) calls TNSecurity::hash_password() and updates the users row. Used by both the password reset flow in control.pl and the emergency repair tool.

Pledge and Unveil Requirements

TNAuth performs SQLite reads and writes, reads key files, and writes to the security log. Any CGI script that calls TNAuth functions must have the following pledge and unveil in place before the first TNAuth call:

perl 
# Minimum pledge for any script using TNAuth
OpenBSD::Pledge::pledge("stdio rpath wpath cpath flock")
    or die "pledge: $!";

# Minimum unveil paths
OpenBSD::Unveil::unveil("$app_root/data/db",   "rwc") or die $!;
OpenBSD::Unveil::unveil("$app_root/data/keys", "r")   or die $!;
OpenBSD::Unveil::unveil("$app_root/data/logs", "rwc") or die $!;
OpenBSD::Unveil::unveil("/dev/urandom",         "r")   or die $!;
# Plus Perl library paths as required

control.pl implements a full pledge and unveil block. All other CGI scripts that call TNSecurityCheck (which calls TNAuth internally) must implement their own equivalent pledge block. The pledge contract is always the caller's responsibility -- TNAuth and TNSecurityCheck do not issue pledge calls themselves. See each backend CGI document for the specific pledge block used in that script.

Summary

  • Every get_db_handle() call opens a fresh connection with foreign keys on, WAL mode on, RaiseError on, and immediate transaction mode on.
  • The first registered user becomes admin. All subsequent registrations require a valid single-use token.
  • Login lockout uses a configurable threshold (floor 3, ceiling 10) and duration (floor 300s). Timed lockouts clear automatically on the next login attempt -- no cron cleanup needed.
  • Transparent PBKDF2 rehash: legacy SHA-256 hashes are upgraded on every successful login without any user interaction.
  • Session IDs stored in the database are raw hex. The signed cookie value (session_id.hmac) is produced by TNSecurity, never by TNAuth.
  • delete_user() explicitly deletes child rows in a transaction as belt-and-suspenders alongside the ON DELETE CASCADE foreign keys.
  • Pledge and unveil are always the calling CGI's responsibility. TNAuth does not issue pledge calls.
