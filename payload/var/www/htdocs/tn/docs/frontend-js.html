Foreword

This document covers the JavaScript layer for all dashboard views, the manage view tab system, and the supporting utility and infrastructure scripts. The authentication, session, CSRF, power management, theme, and SPA shell scripts are documented separately in the View System and Authentication document. The CGI endpoints called by these scripts are documented in the CGI Scripts document.

Dashboard JavaScript: View Scripts and Utilities

Each dashboard view loaded by view.js into #main-content includes one or more inline or embedded scripts. These scripts initialise when the view HTML is injected, run for the lifetime of that view, and are torn down by the nuclear cleanup in view.js when the user navigates away. No view script persists across navigations.

All view scripts follow the principle that JavaScript handles UI state and API calls while Perl generates all application HTML. No view script constructs application-level HTML from scratch except for card and list components built from data returned by the CGI backend. All user-facing text rendered from API data is passed through an escapeHtml() or equivalent function before insertion into the DOM.

Dashboard View: view/dashboard

The dashboard loads three independent scripts: metrics.js, pfstats.js, and servicemonitor.js. Each polls a separate data source and updates its own section of the page. None depends on the others.

metrics.js

Renders collectd-derived system metrics into four cards: CPU, memory, disk, and network. Each card maps to a JSON file under /data/stats/collectd/ served directly from the webroot. The metric keys follow the collectd naming convention using the hostname tangent as the RRD prefix, for example tangent/cpu_avg-cpu-average/cpu.

Polling runs every 10 seconds via a global interval stored as window.dashMetricsInterval. Initialisation uses a polling guard that checks for #card-cpu every 500ms for up to 10 seconds before giving up. This is the standard SPA-safe initialisation pattern used by all three dashboard scripts.

If any metric file is absent, empty, or contains invalid JSON (which happens when collectd is mid-write), the fetch returns null and that card is silently skipped for that cycle. The existing card value is left unchanged. No console noise is produced for these transient failures. DOM update errors, which indicate genuine bugs, are always logged.

The network card handles two interfaces. The WAN interface (%%EXT_IF%%) is the primary card value showing RX throughput. The LAN interface (%%INT_IF%%) is shown in a footer element. The disk card computes a percentage-used value from used, free, and reserved raw byte values before rendering.

Exposes window.refreshSystemMetrics for manual trigger.

pfstats.js

Renders PF firewall statistics from /data/stats/pf_stats.json. Updates five sections every 15 seconds: top vitals (current state count, search count, PF health status, uptime, and state churn rate), a rule audit table showing the top 15 rules by packet count with an efficiency badge (Efficient, Review, or Unused), a throughput bar chart showing the top 10 rules by raw bytes, and a last-update timestamp.

The rule audit table and its mobile list mirror are both rendered from the same data, allowing the view to present appropriately on desktop and mobile without layout switching logic. Table columns are pinned with a dynamically injected <colgroup> element on first render to prevent column width jitter on updates.

All DOM elements are built programmatically using document.createElement and DocumentFragment to avoid repeated reflows. Uses the same 500ms polling guard as metrics.js, checking for #stat-states before starting.

servicemonitor.js

Fetches status files from /data/services/status/ for eight security and network services: Snort IDS, Snort IPS, Snort Sentry, ClamAV, Unbound DNS, DHCPv4, IPv6 Router Advertisements, and E2Guardian. Each service has its own status file path. All eight are fetched in parallel via Promise.all.

Renders a desktop table (#security-service-table) and a mobile card list (#mobile-service-view). Each entry shows display name, running or down status badge, PID, and CPU and memory percentage. Updates every 30 seconds via a global interval stored as window.dashServiceInterval.

If a status file fetch fails, the service is rendered as down with zeroed metrics rather than omitted or erroring out. Uses the same DOM-polling initialisation pattern, checking for either the table or the mobile container before starting.

Exposes window.refreshDashboard for manual trigger.

Logs View: view/logs

logs.js

Implements the LogVisualizer class. Handles log source selection, date selection, server-side filter selection, client-side search, auto-refresh, and clipboard copy. Backend is logs.pl which supports 25 log sources.

On initialisation it detects mobile by user agent and screen width. Auto-refresh is disabled on mobile devices to conserve battery and cannot be re-enabled on mobile. The refresh interval is 8 seconds on desktop.

The date selector is populated by calling logs.pl?action=available_dates on source change. Available archive dates are shown with file size in KB. Gap entries (dates where no archive exists) are appended as disabled options with a classified reason label: System Offline, Empty Log, Archive Missing, or Future. A rotation status element is injected adjacent to the date selector showing last rotation date and a gap summary.

Log rendering uses DocumentFragment for a single-pass DOM update. Each line is classified as info, warning, or error based on the level field from the backend. Timestamps are extracted and rendered as a separate styled span. Service labels from bracketed log formats are separated and styled independently. Client-side search applies a data-log-visible attribute to each line element rather than removing elements, which allows filter toggling without re-fetching. The search field supports both plain string matching and regex, falling back to plain string on regex parse error.

Smart error handling: when logs.pl returns a structured error response rather than log data, the error type field is used to select an appropriate panel style and human-readable explanation. Error types map to distinct visual treatments so the operator can understand immediately whether the system was offline, the service was silent, the archive is missing, or the date is in the future.

Empty file handling: a zero-byte log file is treated separately from an error. A contextual warning panel is shown that explains quiet-mode services may produce empty logs without indicating a problem.

Firewall View: view/firewall

firewall.js

Renders PF firewall log entries fetched from firewall.pl. The backend parses pflog formatted log files and returns server-rendered HTML for the log list and pagination controls. The JavaScript layer handles filtering, date selection, auto-refresh, modal management, and log detail display.

State tracks the current filter set (date, interface, IP family, protocol, port, action, and limit), all fetched logs, and modal pagination state. Auto-refresh runs every 15 seconds. The date selector is populated with the last 7 days, defaulting to today.

Three modals are in use. The filter modal allows the operator to configure all filter parameters before applying them as a batch. The paginated modal shows the full log set when the View All button is clicked, with 10 entries per page and previous and next controls. The detail modal shows all parsed fields for a selected log entry (time, action, rule, interface, source, destination, protocol, port, direction, reason, and payload) plus a toggleable raw JSON view with a copy button.

Initialisation uses the same polling guard pattern as the dashboard scripts, checking for the main container every 100ms for up to 5 seconds. The log cards are built using document.createElement. Click events on cards are wired directly per card, not via delegation, since the card set is replaced on each fetch.

Error states render a retry button using a data-action attribute picked up by a delegated listener on the main container, consistent with the CSP no-inline-handler policy used throughout the application.

Exposes window.firewallForceFetch, window.firewallOpenPaginatedModal, and window.firewallState.

Services View: view/services

services.js

Renders service status cards for all monitored system services, with start, stop, and restart controls. Data comes from services.pl (status) and manage_services.pl (control). Polls status every 15 seconds.

Rendering uses an incremental update strategy. On first load or if the service count changes, a full render replaces the grid. On subsequent updates, only the changed fields within each existing card are updated in place, avoiding flicker. Each card shows a status dot, status badge, CPU percentage, memory percentage, PID, and restart and start or stop buttons.

A grace period system handles slow-starting services. When a start or restart action succeeds, services in the SLOW_START_SERVICES set are shown as Starting for up to 15 seconds rather than immediately reflecting a stopped status if the process has not yet registered. pmacct has an extended 30-second grace period and additional intermediate refresh checks at 3, 8, 15, and 22 seconds because it runs as three aggregated processes with background jobs.

Service control uses CSRF token fetched from control.pl/api/csrf at startup and refreshed on 401 responses. Control actions open a modal immediately showing a spinner, then update to success or error state when the Perl backend returns. Success modals auto-close after 3 seconds. Error modals remain until dismissed. The modal includes a service state panel showing current PID, CPU, memory, RSS, VSZ, user, command, and type from the outcome data.

Six skeleton loading placeholders are shown while the first fetch is in progress. The grid uses delegated click handling for all control buttons to avoid re-binding on each render. A manual refresh button with working and failed visual states is also supported.

Exposes window.fetchStatus for manual trigger.

System Mail View: view/mail

mail.js

Lists, views, and deletes root mailbox messages via mail.pl. Auto-refreshes every 30 seconds. The mail list is paginated with 20 items per page on desktop and 10 on mobile, adjusting automatically on window resize via a debounced handler.

State tracks all mail objects, the current page, selected IDs as a Set, a CSRF token fetched from control.pl/api/csrf, and a delete callback for the confirmation modal. All DOM element references are cached by ID on initialisation.

Each mail item renders a checkbox, a clickable content area showing subject and sender, and an individual delete button. Clicking the content area opens a view modal that fetches the raw message from /data/inbox/mail/ and displays it as plain text. The modal shows subject, sender, date, and size. A copy button copies the message body to clipboard.

Selection state is managed through a Set of file IDs. The select-all checkbox operates on the current page only. Batch operations include batch delete and batch mark as read. Both batch delete and individual delete route through a confirmation modal to prevent accidental deletion. The delete callback pattern stores the pending operation as a function reference in state and executes it only when the user confirms.

Stats in the header show total message count, unread count, high-priority count (messages with error, fail, critical, urgent, or alert in the subject), and total size. Priority detection and read state are client-side only; the backend does not track read state.

External Traffic View: view/external

wan.js

Renders WAN interface traffic flows from fetch_wan.pl (live) and search_wan.pl (archive). Merged from three previously separate files. Provides live polling, archive search, paginated flow browsing, and per-flow JSON detail.

Live mode polls fetch_wan.pl every 10 seconds showing the 20 most recent flows as a preview. Each fetch uses an AbortController to cancel any in-flight request before starting a new one, preventing race conditions on slow connections. After three consecutive errors the polling stops and an error state is shown.

Archive search mode submits filter parameters (date, hour, minute, protocol, source IP, destination IP, port) to search_wan.pl. When results are returned they are displayed in the paginated modal automatically. When no results are found, a message is shown and live polling resumes after 3 seconds. When an archive search error occurs, the error is shown and live polling resumes after 3 seconds.

Date selects in the search form are populated with the last 7 days with human-readable labels (Today, Yesterday, day names).

Three modals are in use. The search modal collects archive search parameters. The paginated modal shows any flow set (live or archive) with 20 entries per page. Closing the paginated modal restarts live polling. The flow detail modal shows the raw JSON for a selected flow with a copy button using the Clipboard API with execCommand fallback.

Flow cards are built using document.createElement and createElementNS for SVG elements. HTML is never inserted via innerHTML for flow data. Error states in the main container use data-action attributes picked up by a delegated listener.

Exposes window.initializeExternalView, window.forceRefresh, window.openPaginatedModal, and window.resetFilters.

Internal Traffic View: view/internal

lan.js

Renders LAN interface traffic flows from fetch_lan.pl. Mirrors the architecture of wan.js but covers only the live feed; there is no archive search for LAN traffic. Polls every 10 seconds.

The LAN view additionally tracks multicast traffic as a distinct counter alongside IPv4 and IPv6. Counters are taken directly from the backend response (total, ipv4, ipv6, multicast fields) rather than computed client-side.

Filtering supports protocol, source IP, source MAC, source port, destination port, and IP family. Active filters are shown as badge labels below the filter button. Filters are applied as query parameters to fetch_lan.pl. Auto-refresh pauses while filters are active to prevent the filter state from being overwritten by background polls.

Three modals: a slide-in filter modal, a paginated full-flow modal (20 entries per page, fetching from the backend with offset), and a flow detail modal showing raw JSON with clipboard copy. The paginated modal fetches each page from the backend rather than paginating a client-side dataset, so it always shows fresh data.

Auto-refresh can be toggled via a button in the view header. Toggle state is reflected on the button class and a status label. A last-updated timestamp shows how long ago the last successful fetch completed.

Exposes window.lanEnhanced (with refresh, toggleAutoRefresh, openModal, and state accessors) and window.lanRetry.

Manage View: view/manage

The manage view is the most complex in the system. It loads five scripts: ui.js, managenav.js, pf.js, e2g.js, and unbound.js. The scripts must load in this order: ui.js first (required by pf.js), then managenav.js, then the three tab scripts.

ui.js

Shared UI primitive library exposing window.UI with three functions. Replaces all native confirm(), alert(), and prompt() calls in the manage view.

UI.confirm(options) shows a themed confirmation dialog with title, body text, optional bullet-list consequences, optional warning callout, configurable button labels, and a variant (danger, warning, info) that controls icon and button colour. Returns via onConfirm and onCancel callbacks. Supports Escape key and backdrop click to cancel. Focuses the confirm button on open for keyboard users.

UI.toast(message, type, duration) shows a self-dismissing notification in the bottom-right corner. Types are success, error, info, and warning. Default duration is 3500ms. Toasts animate in and out. Multiple toasts stack vertically.

UI.error(message, anchorEl) inserts an inline error banner directly after a given anchor element. Removes any existing banner at that anchor before inserting. Auto-removes after 6 seconds. Passing null as message removes an existing banner without adding one.

All three functions inject their styles into a single <style> element on first use. The element is checked by ID before injection so styles are never duplicated. Dark mode is handled via .dark class selectors on the HTML element, consistent with the rest of the application. No Tailwind dark: utilities are used; all colours are defined as explicit CSS rules.

managenav.js

The tab navigation controller for the manage view. Mirrors the architecture of view.js with three internal modules: TabState, TabRegistry, and TabSwitcher, plus a public window.ManageNav API.

Tab IDs are pf, e2guardian, and unbound. On switch, TabSwitcher fires a manageTabLeaving event and calls deactivate hooks for the outgoing tab, updates the DOM (removes and adds active classes on content panels and buttons), updates state, fires activate hooks for the new tab, and dispatches a manageTabChanged event for legacy compatibility.

TabRegistry allows tab scripts to register onActivate and onDeactivate lifecycle hooks via window.ManageNav.register(tabId, hooks). If a tab is already active when a script registers, its onActivate hook is called immediately. All hooks are wrapped in try-catch so one broken hook does not prevent others from running.

Dispatches manageNavReady on completion. Tab buttons are re-cloned before binding to prevent duplicate listener accumulation across view reloads.

Public API: ManageNav.register(tabId, hooks), ManageNav.switchTo(tabId), ManageNav.getCurrentTab(), ManageNav.debug.getState(). Also maintains window.switchManageTab as a legacy alias.

pf.js

The PF firewall management tab. Consolidates what were previously seven separate scripts. Requires ui.js. Boots via the manageTabChanged event when the pf tab becomes active.

Six input sections are managed independently: IP and CIDR, ASN, GeoIP, IP list feeds, a rule builder, and a validation workflow.

IP and ASN sections accept user input, validate format client-side (IP, CIDR, IPv6, ASN normalisation from bare numbers to AS-prefixed format), and write to the queue via pf_write_input.pl. Enter key submits the input field. Each submitted entry is rendered as a card with a delete button that calls pf_delete_input.pl. On success, the card animates out using CSS class transitions rather than immediate removal. Queue entries are loaded from the backend on tab activation via pf_read_input.pl.

The GeoIP section fetches the country list from /data/db/GeoIP/isocountrycodes.html, parses the label elements using DOMParser, and renders them as a searchable checkbox grid. Search filters visible labels in real time. Select all and clear buttons toggle the entire visible set. The GeoIP policy (selected country codes and block or pass action) is written to the queue as a JSON file via pf_write_input.pl with type geoip.

The feeds section loads the protected system intel feeds from /data/db/pf/intel.txt as read-only display items. A separate user feed input allows adding custom HTTP or HTTPS URLs with a block or pass action. User feeds are written to the queue and shown in a separate user feeds list with delete buttons.

The rule builder assembles PF rule syntax from a form with action, direction, quick, log, interface, address family, protocol, source address and port, destination address and port, flags, and state fields. The assembled tokens are shown as a colour-coded live preview. Rules are added to a staging list before submission. The staging list shows numbered rules with per-token syntax highlighting and individual remove buttons. Staged rules can be validated via pf_validate_rule.pl for basic client-side syntax checking, or submitted as a batch to the queue via pf_write_input.pl with type custom_rule, gated behind a UI.confirm dialog listing all staged rules as consequences.

The validation workflow sends a validate trigger to pf_trigger.pl, then polls /data/services/queue/pf-rules/validation-output/verdict.json at one-second intervals for up to 120 seconds. When the verdict file appears, a modal shows pass or fail with statistics (IPs added, ASNs, countries, feeds, rejected, duplicates) or error details. On pass, Apply Changes and View Full Output buttons are available. Apply sends an apply trigger and closes the modal. View Full Output fetches the full validation text from /data/services/queue/pf-rules/validation-output/full-context.txt. Reset sends a reset trigger via pf_trigger.pl, gated behind a UI.confirm dialog listing consequences.

The default rules viewer is a collapsible accordion that fetches /data/services/queue/pf-rules/current and renders it with syntax highlighting: action keywords (pass, block, match), protocol names, port numbers, and directive keywords are each styled as distinct token classes. The accordion auto-refreshes every 30 seconds while open. Refresh and copy buttons are provided.

All state-mutating operations require a CSRF token. The token is fetched at init and stored in module scope. ensureCSRF() fetches a new token if none is present. All fetch calls use credentials: 'same-origin' and X-Requested-With: XMLHttpRequest.

Public API on window.PF: applyRules, closeVerdictModal, showFullOutput, copyOutput. These are used by dynamically rendered modal button handlers.

e2g.js

The e2guardian content filter management tab. Boots via the manageTabChanged event when the e2guardian tab becomes active. Manages filter mode switching, feed management, user feed list editing, and filter log viewing.

Mode switching (general or childsafe) sends a request to e2g_mode_switch.pl which queues the operation and polls for up to 25 minutes. Progress is shown in the UI during the wait. An anti-spam check prevents queuing a second job while one is in progress.

The feed list fetches available feeds from e2g_feeds.pl with a mode parameter (general, childsafe, or custom). User feeds are managed through e2g_write_input.pl (add and remove actions). Log viewing calls e2g_get_log.pl and e2g_get_log_info.pl to fetch log content and metadata.

unbound.js

The Unbound DNS management tab. Boots exclusively via the manageNavReady event, then registers onActivate and onDeactivate lifecycle hooks with ManageNav. No timers start at load time. Nothing runs until the tab is first opened by the user.

On activation it loads DNS statistics from /data/db/unbound/stats.json and starts a 30-second polling interval. The stats file maps to eight display elements: total queries, cached queries, cache hit rate, cache size, uptime, rate-limited queries, recursion count, and average response time. On deactivation the polling stops, the initialised flag resets, and any open modals are removed.

Six operations are available, all wired via a single delegated click handler that checks whether the unbound content panel is active before processing: refresh stats, purge cache (full flush, requires confirmation), flush domain (single domain cache removal, validates domain format, requires confirmation), dump cache (export), reload configuration (reloads unbound.conf without restart), and DNS lookup (test resolution for a domain). Enter key in both domain input fields triggers their respective operations.

Each operation uses three modal types. Confirmation modals (with optional detail block showing the domain name) resolve as Promises before the operation proceeds. Operation feedback modals show a spinner while the request is in flight, then resolve to a success or error state with optional output text. Success modals auto-close after 4 seconds. Validation error modals replace the browser alert() for input errors. All modals support backdrop click to close. No native confirm(), alert(), or prompt() is used.

Domain validation uses a hostname regex before any network request is made. All text rendered into modal bodies passes through escapeHtml(). No globals are exported.

Integrity View: view/integrity

integrity.js

Manages the file integrity verification UI for all 18 TNAudit check groups. Communicates with integrity_request.pl, integrity_status.pl, and integrity_files.pl. Calls control.pl/api/csrf for the CSRF token.

Initialisation uses a MutationObserver-based element wait rather than a polling interval, which is more efficient in the SPA context since the observer fires immediately when the element appears rather than waiting for the next interval tick. The observer watches document.body with childList: true, subtree: true.

On load it fetches a summary from integrity_status.pl with action summary, which returns the current state of all check groups from TNAudit.db. Cards are rendered from an HTML template element (#check-card-tpl) using cloneNode(true), keeping card construction declarative. Each card shows the check name, directory path, description, a status dot and badge, file count, change count, and status string, plus a Verify button and conditionally an Accept Changes button.

The checksData object is the single source of truth for all card states. DOM updates always read from checksData, never from the DOM itself. updateGlobalStats() derives the banner totals and overall status from checksData entries.

Verify requests send to integrity_request.pl with a timestamp-based request time field (format YYYY-MM-DD-HH-MM-SS) that links the request file to its outcome file. The Perl backend blocks for up to 120 seconds polling for the runner's response. Concurrent checks are tracked in an activeChecks Map to prevent duplicate submissions. Each check in progress shows a modal with a spinner and status text, which updates to a results panel or error on completion. Success modals auto-close after 3 seconds.

Accept Changes requests send action update to the same endpoint, triggering TNAudit.pl --update-baseline for the check. Missing files are reported back but not silently accepted. The modal for an accept operation shows accepted count, skipped missing count, error count, and duration, with file lists for any items that could not be accepted.

Verify All iterates the current checksData keys sequentially with a 500ms delay between each request to avoid overwhelming the queue runner. The button is disabled and shows VERIFYING while running.

The violations accordion uses a lazy-load pattern. The accordion content is not fetched until first expansion. Once loaded for a given check, the result is cached in FileAccordion.loadedChecks and not re-fetched on subsequent opens. When violations exist, the accordion auto-expands once (tracked by FileAccordion.autoExpanded to prevent re-expansion on subsequent status updates). The violation count badge updates on every updateGlobalStats() call.

Violations are fetched by calling integrity_files.pl, waiting 500ms for the backend to write the output file, then fetching /data/db/TNAuditFilesList.json. The response can be grouped by check name or flat. On a healthy system both paths render a clean-state panel with a checkmark icon. On a system with violations, each file entry shows filepath, size, permission mode, status badge with icon, and a truncated SHA-256.

Documentation SPA: documentation.html

doc.js

A self-contained documentation SPA. Loads documentation fragments into #doc-main-content from the docs/ directory. Not a view loaded by view.js; it runs on its own standalone documentation.html page.

Twenty-two documents are configured across five sections: Getting Started (preface, quick reference, metrics, orchestration), System and Network (network, firewall, content filter, flow accounting, runners and daemons), Integrity and Alerting (TNAudit, TNWatch), UI Infrastructure and Auth (database, TNEnv, TNConfig, TNAuth, TNSecurity, TNSecurityCheck, TNWAF), Backend (CGI scripts), and Frontend (SPA architecture, CSS, JavaScript). An Installation document closes the list.

Theme management is stored in localStorage under the key tn-docs-theme and falls back to the OS prefers-color-scheme media query on first visit. The theme is applied as a data-theme attribute on the document root rather than a class, keeping it independent of the main dashboard theme mechanism. A toggle button in the header switches between light and dark.

Navigation links use data-doc attributes to identify their target document. All click handlers are wired via addEventListener with no inline handlers, maintaining strict CSP script-src 'self' compliance. Sidebar sections are collapsible with open and closed state tracked via aria-expanded attributes. On mobile, clicking a nav link closes the sidebar. Outside-click handling closes the sidebar when the viewport is 768px or narrower.

Hash-based routing resolves the initial document from window.location.hash on load. The URL hash is updated via history.replaceState on each navigation to allow bookmarking.

Code copy buttons are wired after every fragment load via wireCodeCopyButtons(), which checks a data-wired attribute to prevent duplicate binding. Copy uses the Clipboard API with a textarea execCommand fallback for environments without clipboard access. The button label changes to Copied for 2 seconds on success.

Fragment fetches use Accept: text/html and X-Requested-With: XMLHttpRequest with cache: 'no-cache'. Load and error states are shown in the content area rather than blocking the page. Errors render a callout-danger panel with the HTTP status message.

Maintenance Page: maintenance.html

maintenance.js

Runs on maintenance.html, the page shown after a system restart or shutdown is triggered by powermgmt.js.

Reads the action query parameter from the URL. If the action is shutdown, a static shutdown message panel is displayed and the script exits immediately. No polling occurs because the system is not coming back.

If the action is restart, a spinner panel is shown and the script begins polling /cgi-bin/control.pl/api/health every 5 seconds. A countdown timer shows seconds until the next poll attempt. Any HTTP response from the health endpoint, regardless of status code, means the server has returned and is accepting connections. The script redirects to index.html 800ms after detecting a response. Network errors (where fetch() throws rather than returning a response) mean the system is still down; polling continues. This distinction ensures the redirect fires on server availability rather than on a specific response body.

The self-starting initialisation pattern used by metrics.js, pfstats.js, and servicemonitor.js (polling for a specific element before starting) is necessary because these scripts are loaded into the SPA shell before their target HTML is present in the DOM. The DOMContentLoaded event fires for the shell page, not for dynamically injected view content. The polling approach handles arbitrary injection timing without tight coupling to the view loading sequence.

The manage view scripts (pf.js, e2g.js, unbound.js) must register with ManageNav before their tab is first activated. If a script fails to load, its tab will be visible but unresponsive. The manageNavReady event used by unbound.js provides a clean registration point after managenav.js completes initialisation. Scripts that listen for manageTabChanged directly (such as pf.js) must guard against firing before the DOM is ready.

Summary

  • The dashboard loads three independent scripts: metrics.js (collectd system stats, 10s poll), pfstats.js (PF rule and state stats, 15s poll), and servicemonitor.js (eight security service statuses, 30s poll). All three use a DOM-polling guard for SPA-safe initialisation.
  • logs.js implements a full log viewer with 25 sources, archive date selection with gap classification, server-side line filtering, client-side regex search, mobile battery conservation, and smart error panels that explain why a log is unavailable.
  • firewall.js and wan.js share a similar architecture: live polling, filter modal, paginated full-set modal, and per-entry detail modal. wan.js adds archive search via search_wan.pl. lan.js is live-only but adds multicast tracking and a side-panel filter modal.
  • services.js uses incremental card updates to avoid flicker, a grace period system for slow-starting services, and a service-specific timeout map for the control queue.
  • The manage view is controlled by managenav.js, which provides a lifecycle hook system (onActivate, onDeactivate) for the three tab scripts. ui.js provides shared confirm, toast, and error primitives used by pf.js and unbound.js.
  • pf.js consolidates seven previously separate PF management scripts into one module covering IP, ASN, GeoIP, feeds, rule builder, and a validation workflow with verdict polling and apply or reset actions.
  • unbound.js is lifecycle-driven: nothing starts until the tab is opened, everything stops when it is left. Operations use Promise-based custom modals instead of native browser dialogs.
  • integrity.js uses a MutationObserver for initialisation, a template element for card rendering, a single-source-of-truth data object for all card states, and a lazy-load accordion for violation file lists.
  • doc.js is a standalone documentation SPA with theme persistence, hash routing, collapsible sidebar sections, and clipboard copy buttons wired after each fragment load.
  • maintenance.js distinguishes shutdown (static message, no polling) from restart (health endpoint polling, redirect on any response).
