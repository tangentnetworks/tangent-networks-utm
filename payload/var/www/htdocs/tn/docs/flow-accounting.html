Foreword

This document covers flow accounting -- the capture, aggregation, and storage of network traffic data using pmacct. Three pmacctd instances run simultaneously, each with a distinct role: real-time WAN display, real-time LAN display, and persistent per-quarter-hour JSON logs for historical analysis. It also covers the MFS truncation utility that keeps the real-time pipe files within bounds.

pmacct data feeds the network traffic panels in the WebUI dashboard. The MFS pipe files are the live data source; the timestamped JSON log files under network/pmacct/ext/ are the historical record.

Flow Accounting

Architecture Overview

pmacct captures traffic by opening BPF devices on the network interfaces. Three pmacctd instances run from rc.local, each reading a separate configuration file. Two instances write to MFS-backed pipe files for real-time WebUI consumption. One instance writes to timestamped JSON files on persistent storage for historical analysis.

ext_if_json_mfs -- WAN real-time pipe
Captures on %%EXT_IF%% (WAN). Aggregates by source host, destination host, source port, destination port, protocol, TCP flags, and ToS. Writes JSON to the MFS pipe file every 10 seconds. WebUI reads this for live WAN traffic display.
int_if_json_mfs -- LAN real-time pipe
Captures on %%INT_IF%% (LAN). Aggregates by source host, source MAC, source port, destination host, and protocol. MAC address capture identifies individual LAN devices beyond IP alone. Writes JSON to the MFS pipe file every 10 seconds.
ext_if_json_log -- WAN persistent log
Captures on %%EXT_IF%% (WAN). Same aggregation fields as the WAN MFS instance. Writes to a new timestamped JSON file every 60 seconds under network/pmacct/ext/ using the filename pattern ext_if-YYYYMMDD-HHMM.json. Quarter-hour files are merged daily by the rc.local boot routine into a single dated JSON file and kept for 7 days.

MFS Pipe Files

The two real-time pipe files live on a 64 MB memory filesystem mounted at boot from fstab:

fstab 
swap /var/www/htdocs/tn/data/pipes/pmacct mfs rw,nosuid,nodev,noatime,noexec,-s=64m 0 0

Writing to MFS rather than disk eliminates I/O overhead for files that are updated every 10 seconds continuously. The noatime flag suppresses access time updates. The 64 MB allocation is generous -- the two pipe files are capped at 16 MB each by newsyslog before SIGHUP rotation, and pmacct_mfs_manage.sh keeps them at 200 lines in practice.

MFS contents are lost on reboot. rc.local pre-creates the pipe files with correct permissions before starting pmacctd. If the MFS mount fails or the files are absent, pmacctd will fail to open its output path and start without writing any data. After any unexpected reboot, verify the MFS is mounted and the files exist before checking pmacct status.

Instance Configurations

ext_if_json_mfs -- WAN real-time

conf 
pcap_interface: %%EXT_IF%%
promisc: true
plugins: print[jsonpipe]
aggregate[jsonpipe]: src_host, dst_host, src_port, dst_port, proto, tcpflags, tos
print_output_file[jsonpipe]: /var/www/htdocs/tn/data/pipes/pmacct/ext_if_json.log
print_output[jsonpipe]: json
plugin_pipe_size: 4194304
print_refresh_time[jsonpipe]: 10
print_output_file_append: true
print_history[jsonpipe]: 15m
print_cache_entries[jsonpipe]: 8192
pidfile: /var/www/htdocs/tn/data/run/pmacct/pmacct-ext_if_json_mfs.pid
imt_path: /var/www/htdocs/tn/data/tmp/pmacct/pmacct-ext_if_json_mfs.imt

print_refresh_time: 10 -- flushes the aggregation cache and appends a JSON record to the pipe file every 10 seconds. print_history: 15m -- each record covers a 15-minute accounting window. plugin_pipe_size: 4194304 -- 4 MB internal pipe buffer between the capture thread and the print plugin. This instance does not set snaplen explicitly, using the pmacct default (96 bytes) -- sufficient for header capture used in flow aggregation.

int_if_json_mfs -- LAN real-time

conf 
pcap_interface: %%INT_IF%%
snaplen: 1500
promisc: true
plugins: print[jsonpipe]
aggregate[jsonpipe]: src_host, src_mac, src_port, dst_host, proto
print_output_file[jsonpipe]: /var/www/htdocs/tn/data/pipes/pmacct/int_if_json.log
print_output[jsonpipe]: json
plugin_pipe_size: 4194304
print_refresh_time[jsonpipe]: 10
print_output_file_append: true
print_history[jsonpipe]: 15m
print_cache_entries[jsonpipe]: 8192
pidfile: /var/www/htdocs/tn/data/run/pmacct/pmacct-int_if_json_mfs.pid
imt_path: /var/www/htdocs/tn/data/tmp/pmacct/pmacct-int_if_json_mfs.imt

The LAN instance adds src_mac to the aggregation fields. MAC addresses identify LAN devices independently of their IP address, which matters when DHCP reassigns addresses or a device uses multiple IPs. snaplen: 1500 captures full Ethernet frames on the LAN interface. The aggregation set is narrower than the WAN instance -- TCP flags and ToS are omitted as they are less relevant for LAN device identification.

ext_if_json_log -- WAN persistent log

conf 
pcap_interface: %%EXT_IF%%
snaplen: 1500
promisc: true
plugin_pipe_size: 1048576
plugins: print[jsonlog]
aggregate[jsonlog]: src_host, dst_host, src_port, dst_port, proto, tcpflags, tos
print_output_file[jsonlog]: /var/www/htdocs/tn/data/network/pmacct/ext/ext_if-%Y%m%d-%H%M.json
print_output[jsonlog]: json
print_refresh_time[jsonlog]: 60
print_output_file_append: true
print_history[jsonlog]: 15m
print_cache_entries[jsonlog]: 8192
pidfile: /var/www/htdocs/tn/data/run/pmacct/pmacct-ext_if_json_log.pid
imt_path: /var/www/htdocs/tn/data/tmp/pmacct/pmacct-ext_if.imt

print_refresh_time: 60 -- writes a new record every 60 seconds. The filename pattern ext_if-%Y%m%d-%H%M.json produces one file per minute, giving 15 files per quarter-hour window. plugin_pipe_size: 1048576 -- 1 MB internal pipe buffer, smaller than the MFS instances because the flush interval is longer and the write destination is persistent storage rather than MFS.

Startup Sequence

rc.local starts the three pmacctd instances in two phases. The MFS instances start immediately at boot. The log instance waits until the next quarter-hour boundary before starting, so that its timestamped output files align cleanly to 15-minute boundaries (00, 15, 30, 45 minutes past the hour).

The quarter-hour alignment calculation: rc.local computes how many seconds remain until the next 15-minute boundary and sleeps that long in a background subshell before starting ext_if_json_log. If the system boots exactly on a boundary, the sleep is skipped. The alignment subshell runs in the background -- rc.local does not block waiting for it.

A background permission fixer loop also starts at boot. It runs find every 8 seconds on the MFS pipe directory and the persistent log directory, setting 644 permissions on any new files pmacct creates. This is necessary because pmacctd creates new output files with restrictive permissions inherited from its umask, which prevents the www user from reading them.

Daily Log Consolidation

The rc.local pmacct section also handles log consolidation from the previous day at each boot. It finds all JSON and text files from yesterday's date under network/pmacct/ext/, concatenates them into a single dated file (YYYY-MM-DD.json), and removes individual quarter-hour files. Files older than 7 days are deleted. This runs synchronously at boot before the pmacctd instances start.

Consolidation runs at boot only. If the system runs continuously for multiple days without a reboot, per-minute JSON files accumulate in network/pmacct/ext/ without being consolidated. A dedicated daily consolidation cron job is a deferred improvement. For now, the consolidation happens on the next boot.

MFS Pipe Management -- pmacct_mfs_manage.sh

pmacct_mfs_manage.sh is a ksh utility that keeps the two MFS pipe files within a fixed line count. It runs every 60 seconds via pmacct_mfs_manage_runner.sh, which loops indefinitely from rc.local.

The script truncates each pipe file to the last 200 lines when the file exceeds that count. It preserves the inode by writing back via redirection (cat tmpfile >| logfile) rather than replacing the file -- this is critical because pmacctd holds the file open by inode. Replacing the file would leave pmacctd writing to the old unlinked inode while the WebUI reads an empty new file.

Per-file locking uses a lock directory (logfile.lock/) rather than a lock file, because directory creation is atomic on all POSIX filesystems. The lock timeout is 5 seconds. If a lock cannot be acquired and the file is open by another process, the script skips silently -- this is the normal case when pmacctd is actively writing.

newsyslog Integration

The two MFS pipe files have newsyslog entries that provide a size-based safety net at 16 MB, sending SIGHUP to the respective pmacctd instance after rotation so it re-opens the file:

conf 
/var/www/htdocs/tn/data/pipes/pmacct/ext_if_json.log  www:wheel  644  0  16000  *  Z  pmacct-ext_if_json_mfs.pid  SIGHUP
/var/www/htdocs/tn/data/pipes/pmacct/int_if_json.log  www:wheel  644  0  16000  *  Z  pmacct-int_if_json_mfs.pid  SIGHUP

The count field is 0 -- no rotated copies are kept. The size field is 16000 KB. In normal operation pmacct_mfs_manage.sh keeps files well under this limit. The newsyslog entry fires only if the management script fails or falls behind. The PID filenames in newsyslog match the pidfile directives in the respective configs exactly.

Operational Reference

sh 
# Check all three pmacctd instances are running
pgrep -a pmacctd

# Check PID files
ls /var/www/htdocs/tn/data/run/pmacct/

# Verify MFS is mounted
mount | grep pipes/pmacct

# Check MFS pipe file sizes and line counts
wc -l /var/www/htdocs/tn/data/pipes/pmacct/ext_if_json.log

# Tail live WAN traffic feed
tail -f /var/www/htdocs/tn/data/pipes/pmacct/ext_if_json.log

# Check persistent log files
ls -lh /var/www/htdocs/tn/data/network/pmacct/ext/ | tail -20

# Check pmacct_mfs_manage_runner is running
pgrep -f pmacct_mfs_manage_runner

# Check pmacct startup log
tail -50 /var/www/htdocs/tn/data/logs/pmacct/pmacct.log

Summary

  • Three pmacctd instances serve two purposes: real-time WebUI feed (MFS pipe files, 10-second flush) and persistent historical logging (timestamped JSON files, 60-second flush, 7-day retention).
  • The WAN real-time and log instances capture on %%EXT_IF%% with full flow fields including TCP flags and ToS. The LAN real-time instance captures on %%INT_IF%% and adds MAC address to identify devices independently of IP assignment.
  • The log instance starts quarter-hour aligned to produce clean 15-minute boundary files. MFS instances start immediately at boot.
  • pmacct_mfs_manage.sh keeps MFS pipe files at 200 lines with inode-preserving truncation and per-file locking. newsyslog provides a 16 MB size-based safety net with SIGHUP re-open signalling.
  • Daily log consolidation runs at boot, merging previous-day per-minute files into a single dated JSON. A dedicated consolidation cron job is a deferred improvement for systems that run continuously.
