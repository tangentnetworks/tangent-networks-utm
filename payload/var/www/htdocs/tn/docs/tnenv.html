Foreword

TNEnv is loaded first in every script and every TN module. It sets up the library path, cleans the environment for taint mode, configures I/O encoding, and provides the path resolution functions that every other module depends on. This document is short because TNEnv is intentionally minimal -- it does one thing well and stays out of the way.

TNEnv

Location: /var/www/htdocs/tn/data/lib/TNEnv.pm
Version: 1.0.0
Load order: First. Always before any other TN module.

Purpose

TNEnv solves a specific problem: CGI scripts, command-line tools, and Perl modules all run from different working directories and are invoked in different ways. Without a common bootstrap, each script would need to independently calculate its own @INC path, set $ENV{PATH}, and configure file handle encoding. TNEnv centralises all of that so each consumer can simply say use TNEnv; and proceed.

The module is self-locating. It uses dirname(__FILE__) to find its own position on disk and derives all other paths from that anchor. It does not use FindBin, Cwd::getcwd, or any path that depends on where the calling script lives. This makes it chroot-safe: the same module works identically whether the process sees /var/www/htdocs/tn/data/lib (host) or /htdocs/tn/data/lib (inside the chroot).

What Happens at Load Time

Two BEGIN blocks run at compile time in order. The first cleans the environment:

perl 
$ENV{PATH} = '/bin:/usr/bin:/usr/local/bin:/sbin:/usr/sbin';
delete @ENV{qw(IFS CDPATH ENV BASH_ENV)};
$ENV{LD_LIBRARY_PATH} = '/usr/local/lib:/usr/lib';

PATH is set to a known safe value before taint mode will allow any external command. The shell injection vectors (IFS, CDPATH, ENV, BASH_ENV) are removed. LD_LIBRARY_PATH is set explicitly for shared library resolution inside the chroot, where DBD::SQLite and other XS modules need to find libsqlite3 and libpthread.

I/O encoding is then set on all three standard handles:

perl 
binmode(STDIN,  ':encoding(UTF-8)');
binmode(STDOUT, ':raw');
binmode(STDERR, ':encoding(UTF-8)');

STDOUT is set to raw (:raw) deliberately. The JSON serialiser (JSON->new->utf8->encode()) produces raw UTF-8 octets. Adding a :encoding(UTF-8) layer on top would cause Perl to treat those octets as Latin-1 and double-encode them, corrupting non-ASCII output. STDIN and STDERR use UTF-8 encoding layers because they handle text, not binary CGI output.

The second BEGIN block locates the library directory, untaints the path, and adds it to @INC:

perl 
my $lib_path = dirname(__FILE__);
$lib_path = abs_path($lib_path) if -d $lib_path;
if ($lib_path =~ m{^([-/\w.]+)$}) {
    unshift @INC, $1 unless grep { $_ eq $1 } @INC;
}

The resolved path is stored in $TNEnv::LIB_PATH for other modules to reference. All subsequent path functions derive from this anchor.

Path Resolution

The directory layout the path functions assume:

text 
/var/www/htdocs/tn/          <-- get_app_root()
    cgi-bin/
        control.pl
        router.pl
    data/
        lib/                 <-- $TNEnv::LIB_PATH  (TNEnv.pm lives here)
        db/                  <-- get_db_path()
        keys/                <-- get_keys_path()
        config/              <-- get_config_path()
        logs/                <-- get_logs_path()
        scripts/

get_app_root() walks two levels up from LIB_PATH (data/lib/../.. = app root). All other functions are aliases to get_data_path(subdir). The internal _resolve_path() helper tries Cwd::abs_path() first -- authoritative when the path exists -- and falls back to a pure-string segment-by-segment collapse when the path does not yet exist (install time) or is unavailable inside the chroot.

The exported path functions:

get_app_root()
Returns the application root: /htdocs/tn inside the chroot, /var/www/htdocs/tn outside.
get_data_dir()
Returns app_root/data.
get_db_path()
Returns app_root/data/db. Auth.db lives here.
get_keys_path()
Returns app_root/data/keys. HMAC and session key files live here.
get_config_path()
Returns app_root/data/config. security.conf lives here.
get_logs_path()
Returns app_root/data/logs.

Taint Utilities

Three untainting helpers are exported for use across all modules and scripts:

untaint_path($path)
Rejects any path containing .. before the pattern match, then allows only alphanumerics, forward slashes, underscores, hyphens, and dots. Returns undef on failure.
untaint_filename($filename)
Rejects path separators and .. sequences explicitly, then allows only word characters, hyphens, and dots. Returns undef on failure. Use for bare filenames, not full paths.
untaint_identifier($id)
Allows only [a-zA-Z0-9_]. Suitable for database IDs and internal identifiers. Note: does not allow hyphens, so it is too restrictive for usernames (use the /^([a-zA-Z0-9_-]{3,32})$/ pattern from TNSecurity for those).

Environment Verification

verify_environment() checks that all required directories and the config file exist. Returns an arrayref of error strings, or undef if everything is in order. Used at startup by diagnostic tools; not called during normal CGI request handling.

perl 
use TNEnv;
my $errors = TNEnv::verify_environment();
if ($errors) {
    for my $err (@$errors) { warn "ENV ERROR: $err\n" }
    exit 1;
}

Usage Pattern

Every CGI script and every TN module loads TNEnv first. CGI scripts do it via the bootstrap block in their BEGIN:

perl 
#!/usr/bin/perl -T
use strict;
use warnings;
use FindBin qw($RealBin);
use File::Spec;

BEGIN {
    my $lib_path = File::Spec->catdir(
        $RealBin, File::Spec->updir, 'data', 'lib'
    );
    # ... untaint and add to @INC ...
}

use TNEnv;    # must be first TN module
use TNConfig;
use TNSecurity;
use TNAuth;

TN modules (TNSecurity, TNConfig, TNAuth, etc.) each have their own BEGIN block that resolves the lib path from dirname(__FILE__) for standalone use. When loaded through a CGI script that has already set up @INC, the redundant path add is a no-op because of the unless grep { $_ eq $lib_path } @INC guard.

TNEnv exports get_data_path, get_db_path, get_keys_path, get_config_path, and untaint_path into the caller's namespace automatically on use TNEnv;. All other functions must be called with the package prefix: TNEnv::get_app_root().

Summary

  • TNEnv must be loaded first in every script and module. It sets PATH, removes shell injection vectors, sets LD_LIBRARY_PATH for chroot shared libs, and configures I/O encoding.
  • STDOUT is set to :raw deliberately. Adding a UTF-8 encoding layer on top of the JSON serialiser would double-encode non-ASCII output.
  • All path functions derive from $TNEnv::LIB_PATH, which is resolved from dirname(__FILE__). The module is self-locating and chroot-safe.
  • get_app_root() returns /htdocs/tn inside the chroot. All other path helpers call get_data_path(subdir) internally.
  • Three taint utilities are available: untaint_path, untaint_filename, untaint_identifier. Note that untaint_identifier does not allow hyphens and is unsuitable for usernames.
