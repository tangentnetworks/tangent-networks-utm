Foreword

This document covers TNAudit from the operator perspective: what it monitors, how to run it, how to read its output, and what to do when it reports a change. For the internal module API and developer integration guide see the separate developer documentation in /usr/local/share/doc/TN/dev/tnaudit.md.

Read the Quick Reference first if you have not already. The integrity check system is listed there under the data directory map and the runner table.

TNAudit: File Integrity Monitoring

TNAudit monitors the SHA-256 hash of every file in a configured set of directories and individual files. It records a baseline at a known-good time, then verifies the current state against that baseline on demand or on a schedule. Any file that has been modified, deleted, or created since the baseline was taken is reported as a change.

On this system TNAudit monitors 18 check groups spanning the Tangent Networks application code, system configuration, system binaries, libraries, headers, the package database, and the DNS resolver tree. A full baseline covers several thousand files. Verification of the full set runs in seconds.

Architecture

TNAudit consists of a single command-line script and four support modules:

/usr/local/sbin/TNAudit.pl
The main script. Parses command-line options and dispatches to create-baseline, verify, status, list-files, history, or init-db. Runs under Perl taint mode (-T). Always run as root.
/etc/TNAudit/TNAuditConfig.pm
Reads and parses integrity_checks.conf. Validates each check definition. Loads the global excludes file. Merges per-check and global exclude patterns.
/etc/TNAudit/TNAuditDatabase.pm
All SQLite operations. Creates and manages the three-table schema in TNAudit.db. Provides baseline, verify, status query, and change history functions.
/etc/TNAudit/TNAuditExcludes.pm
Pattern matching for file exclusion. Supports shell glob (default), regex (/pattern/), and exact (exact:name) pattern types.
/etc/TNAudit/TNAuditScanner.pm
File system traversal and SHA-256 hashing. Uses File::Find for recursive directory walks. Skips symlinks. Records size, mode, uid, gid, mtime, and SHA-256 for each file.

The modules live at /etc/TNAudit/, outside the httpd chroot. TNAudit runs as root from cron and from the integrity runner shell script. It never runs inside the chroot and is never called by a CGI script directly.

The database lives at:

sh 
/var/www/htdocs/tn/data/db/TNAudit.db

This path is inside the webroot so the www user can read it for the dashboard UI. Direct HTTP access to TNAudit.db is blocked by the TNWAF serve_data() denylist, which denies all requests for .db files.

Configuration Files

Two configuration files control what TNAudit monitors and what it ignores.

integrity_checks.conf

Located at /var/www/htdocs/tn/data/config/integrity_checks.conf. Each active line defines one check group in pipe-delimited format:

text 
type|check_name|display_name|path|description|excludes

Where type is dir for recursive directory scan or file for a single file. The check_name is the short identifier used in API calls, log entries, and the queue system. The excludes field is optional and takes comma-separated patterns applied only to that check, in addition to the global excludes.

The 18 active checks on this system are:

assets
Web Assets. Path: /var/www/htdocs/tn/assets. JavaScript and CSS files served by the dashboard.
cgi
CGI Scripts. Path: /var/www/htdocs/tn/cgi-bin. All 24 Perl CGI endpoints.
config
Configuration. Path: /var/www/htdocs/tn/data/config. Security configuration files including security.conf, integrity_checks.conf, and integrity_excludes.conf.
keys
Encryption Keys. Path: /var/www/htdocs/tn/data/keys. The HMAC and session key files. A change to any file in this directory invalidates all session tokens and breaks all active sessions.
lib
Perl Modules. Path: /var/www/htdocs/tn/data/lib. The core TN*.pm modules: TNAuth, TNConfig, TNEnv, TNSecurity, TNSecurityCheck, and TNWAF.
etc
System Configuration. Path: /etc. Full recursive coverage of all system configuration. Per-check excludes: master.passwd, passwd, random.seed, pf/feeds. The password and shadow files are excluded because they change on user operations. The PF feeds directory is excluded because it is updated hourly from external blocklist sources.
usr_local_etc
Package Configuration. Path: /usr/local/etc. Third-party package configuration including Snort, e2guardian, ClamAV, and all other installed service configs.
usr_bin
User Binaries. Path: /usr/bin. Standard user commands and utilities. Per-check excludes: *.old, *.backup, *~.
usr_sbin
System Binaries. Path: /usr/sbin. Essential system administration binaries. Per-check excludes: *.old, *.backup, *~.
usr_libexec
System Utilities. Path: /usr/libexec. Internal binaries run by other programs. Per-check excludes: *.old, *.backup, *~.
local_bin
Local Binaries. Path: /usr/local/bin. User-installed executables and scripts. Per-check excludes: *.old, *.backup, *~.
sbin
System Scripts. Path: /usr/local/sbin. Service management and monitoring scripts including all Tangent runners. Per-check excludes: *.old, *.backup, *~.
usr_lib
Base System Libraries. Path: /usr/lib. Base system shared libraries. Useful after a sysupgrade to verify the library set matches the expected post-upgrade state.
usr_local_lib
Package Libraries. Path: /usr/local/lib. Third-party package shared libraries.
usr_include
System Headers. Path: /usr/include. System-wide C header files.
usr_local_include
Package Headers. Path: /usr/local/include. Third-party C header files.
pkg
Package Database. Path: /var/db/pkg. Metadata for all installed packages. The global excludes strip the +CONTENTS, +DESC, +COMMENT, and other per-package metadata files that update on every pkg_add or pkg_delete operation.
var_unbound
DNS Resolver. Path: /var/unbound. The caching DNS resolver tree. Unbound runs chrooted to this directory.

To add a new check, append a line to integrity_checks.conf then run TNAudit.pl --create-baseline --check <name>. The new card appears in the UI automatically on the next page load.

If you keep dated backup files using the compact YYYYMMDD format (for example script.sh-20260225) in any monitored directory, add the pattern *-[0-9][0-9][0-9][0-9][0-9][0-9][0-9][0-9] to integrity_excludes.conf. The existing global exclude pattern covers only the hyphenated YYYY-MM-DD format. Without this addition, compact-date backups in sbin, etc, or any other active check will be reported as unexpected new files on every verification run.

integrity_excludes.conf

Located at /var/www/htdocs/tn/data/config/integrity_excludes.conf. Contains global exclude patterns applied to every check. Patterns are matched against the bare filename, not the full path, unless a regex pattern is used. The three pattern types are:

Shell glob (default)
Plain pattern, matched case-sensitively against the filename. Wildcards: * matches any characters, ? matches one character, [abc] and [0-9] match character classes. Examples: *.backup, *~, *-[0-9][0-9][0-9][0-9]-[0-9][0-9]-[0-9][0-9].
Regex
Pattern wrapped in forward slashes. Full Perl regex syntax. Matched against the full file path. Example: /\/lists\// excludes any file under a directory named lists.
Exact
Prefixed with exact:. Matches the bare filename exactly. Example: exact:master.passwd.

The global excludes cover editor temporary files (*.swp, *~), backup extensions (*.bak, *.old, *.orig, date-stamped variants), version control directories (.git, CVS), log and temp files, runtime state files (*.pid, *.lock, *.sock), SSH and SSL private key material, OpenBSD shadow password files, Snort dynamic shared objects, ClamAV signature databases, and RRD databases.

Database Schema

TNAudit.db contains three tables. The files table holds one row per monitored file with its baseline SHA-256, size, mode, ownership, and modification time. The changes table records every detected modification with old and new hashes and sizes. The checks table stores check metadata from the configuration. All three tables are created automatically by init_db() on first use. The database runs in WAL mode for concurrent reads by the dashboard UI.

File status values in the files table:

baseline
Initial state after a baseline run, or after an operator has accepted a change via --update-baseline. Treated as clean.
verified
File was checked and its current SHA-256 matches the baseline.
modified
File exists but its SHA-256 no longer matches the baseline.
missing
File was in the baseline but no longer exists on disk.
error
File exists but could not be read during verification.

New files found during verification (files present on disk but not in the baseline) are reported in the JSON output with status new. They are not written to the files table until a baseline or update-baseline run.

Command Reference

All commands require root. Run from the host shell, not from inside the chroot.

Create or refresh the baseline

sh 
# Full baseline across all checks. Clears and rebuilds the files and checks tables.
# The changes table (audit history) is preserved.
TNAudit.pl --create-baseline

# Single check baseline. Replaces rows for that check only.
TNAudit.pl --create-baseline --check cgi

# With JSON output
TNAudit.pl --create-baseline --json

A full baseline deletes all rows from the files and checks tables before scanning. This ensures that check names removed from the configuration file do not leave stale rows behind that would appear as ghost cards in the UI. The changes table is never cleared; audit history is permanent unless explicitly purged.

A full baseline on the complete 18-check set takes several minutes on first run because it hashes every file under /usr/lib, /usr/local/lib, /usr/include, and /var/db/pkg. Subsequent verification runs are fast because they only compare hashes. Run the full baseline only after a confirmed clean state: after a fresh install, after a reviewed sysupgrade, or after deliberately accepting all pending changes.

Verify integrity

sh 
# Verify all checks
TNAudit.pl --verify

# Verify one check
TNAudit.pl --verify --check cgi

# Verify and get JSON output (used by the runner)
TNAudit.pl --verify --check cgi --json

Verification compares the current SHA-256 of every file in the baseline against the stored hash. It also scans the monitored paths for new files not in the baseline. The result JSON includes counts of verified, modified, missing, and new files, plus a list of changed file paths with their old and new hashes.

Query status

sh 
# Status of all checks from the database
TNAudit.pl --status --json

# Status of one check
TNAudit.pl --status --check cgi --json

Status queries read the current state of the files table without re-hashing anything. The result reflects the outcome of the last verification run. Use --verify to get a fresh result.

List monitored files

sh 
TNAudit.pl --list-files --check cgi --json

Returns all files in the baseline for the named check, with their stored hashes, sizes, modes, and last-checked timestamps. Check name is required for this action.

Show change history

sh 
# Last 50 changes for a check (default)
TNAudit.pl --history --check cgi

# Last 20 changes
TNAudit.pl --history --check cgi --limit 20

History reads the changes table, which is never cleared by baseline operations. It shows every modification, deletion, and accepted change recorded for the named check, newest first.

Accept a changed file

When a file has been legitimately modified after a package update, a configuration edit, or a sysupgrade, run a targeted baseline to accept the new state:

sh 
# Accept all changes in one check
TNAudit.pl --create-baseline --check cgi

# Accept everything (use after a reviewed sysupgrade)
TNAudit.pl --create-baseline

Initialise the database

sh 
TNAudit.pl --init-db

Creates the database and schema if they do not exist. Safe to run on an existing database; the CREATE TABLE IF NOT EXISTS guards prevent data loss. Run this after first install before any baseline.

Environment overrides

All three paths can be overridden for testing without touching the live database:

sh 
TAUDIT_DB=/tmp/test.db \
TAUDIT_CONFIG=/tmp/test_checks.conf \
TAUDIT_EXCLUDE=/tmp/test_excludes.conf \
TNAudit.pl --create-baseline

The Runner Pipeline

The web UI integrity panel triggers on-demand verifications through a queue system rather than calling TNAudit.pl directly. This keeps the CGI layer unprivileged and avoids blocking the HTTP response on a potentially slow hash operation.

The pipeline has three components:

/usr/local/sbin/integrity_check_runner.sh
The outer loop. Writes its PID to /var/www/htdocs/tn/data/run/webui/integrity_check.pid and calls integrity_check.sh in an infinite loop with a one-second sleep between iterations. Started by rc.local at boot.
/usr/local/sbin/integrity_check.sh
The queue processor. Reads request files from /var/www/htdocs/tn/data/queue/integrity/request/. For each request it calls TNAudit.pl with the appropriate action and check name, writes the JSON result to /var/www/htdocs/tn/data/queue/integrity/outcome/, and updates the cached status in /var/www/htdocs/tn/data/services/status/integrity/. Cleans outcome files older than five minutes. Exits silently if the queue is empty.
/var/www/htdocs/tn/cgi-bin/integrity_request.pl
The CGI endpoint. Creates a request file in the request directory. The file name encodes a Unix timestamp; the file contents contain the action and check name. Returns immediately to the browser. The actual verification happens asynchronously.

Request file format:

text 
# filename: request-1741768800
# contents: one line, two fields
verify cgi

Valid actions in a request file are verify, baseline, and update. The runner maps these to TNAudit.pl --verify, --create-baseline, and --update-baseline respectively.

The outcome file contains the raw JSON output from TNAudit.pl, written as-is:

json 
{
  "success": true,
  "action": "verify",
  "check": "cgi",
  "status": "verified",
  "files": 24,
  "verified": 24,
  "modified": 0,
  "missing": 0,
  "new": 0,
  "changes": 0,
  "duration": "0.42s"
}

The runner log is at /var/www/tmp/integrity_runner.log. It records every request processed, the TNAudit.pl exit code, duration, and any parse errors.

Cron Schedule

TNAudit runs one scheduled job: a full baseline every Sunday at 14:00.

cron 
0 14 * * 0   root   TNAudit.pl --create-baseline >> /var/www/tmp/integrity_baseline.log 2>&1

The baseline runs on Sunday to align with the TNWatch weekly purge cycle and to avoid interfering with the daily digest at 09:00. The baseline log at /var/www/tmp/integrity_baseline.log accumulates all weekly runs. Review it after any system change to confirm the baseline completed cleanly.

The weekly cron baseline unconditionally replaces the previous baseline. If the system has been compromised between Sunday runs, running the baseline will overwrite evidence of the change, recording the attacker's modified files as the new known-good state. For this reason the changes table is never cleared by a baseline operation; it accumulates the full history of every detected modification. Review the changes table and the TNWatch digest before accepting a new baseline after any anomaly.

Integration with TNWatch

TNWatch reads TNAudit.db directly. The TNWatchParser parse_tnaudit function opens TNAudit.db via SQLite, queries the files table for rows with status other than ok, verified, or baseline, and emits a file_change event to the TNWatch event database for each modified or missing file. It also emits one integrity_check summary event per parse cycle with total and changed file counts.

The default TNWatch alert rule tnaudit_change fires an immediate email the first time a file_change event appears. The daily digest includes the TNAudit section with the last known status, total file count, change count, and a list of any changed file paths detected in the past 24 hours.

Troubleshooting

Check that the database and config exist and are readable:

sh 
ls -l /var/www/htdocs/tn/data/db/TNAudit.db
ls -l /var/www/htdocs/tn/data/config/integrity_checks.conf
ls -l /etc/TNAudit/

Run a quick status check to confirm the database is readable and the last verification results are present:

sh 
TNAudit.pl --status --json | head -40

Check the runner is alive:

sh 
cat /var/www/htdocs/tn/data/run/webui/integrity_check.pid
kill -0 $(cat /var/www/htdocs/tn/data/run/webui/integrity_check.pid) && echo alive

Inspect the runner log for recent activity:

sh 
tail -40 /var/www/tmp/integrity_runner.log

Inspect the weekly baseline log:

sh 
tail -20 /var/www/tmp/integrity_baseline.log

Query the database directly to see the file status breakdown:

sh 
sqlite3 /var/www/htdocs/tn/data/db/TNAudit.db \
  "SELECT check_name, status, COUNT(*) FROM files GROUP BY check_name, status ORDER BY check_name"

If a check shows no files at all, the baseline for that check has never been run or was cleared. Run a targeted baseline:

sh 
TNAudit.pl --create-baseline --check cgi

If a check path does not exist, TNAuditConfig will skip it with a warning on stderr. The check will not appear in the UI until the path exists and a baseline is run.

TNAuditScanner skips symlinks deliberately. The file count shown in the UI card is the count of regular files only, which matches what find /path -type f would return. Do not compare it against ls -la counts, which include symlinks.

Summary

  • TNAudit monitors 18 check groups via SHA-256 hashing. A full baseline covers the Tangent application code, all system configuration, binaries, libraries, headers, the package database, and the DNS resolver tree.
  • The database is at /var/www/htdocs/tn/data/db/TNAudit.db. Modules are at /etc/TNAudit/. Neither lives inside the httpd chroot. TNAudit always runs as root.
  • A full baseline runs every Sunday at 14:00 via cron. Log at /var/www/tmp/integrity_baseline.log. The changes table accumulates history permanently and is never cleared by a baseline run.
  • The web UI triggers on-demand verifications through a queue in data/queue/integrity/. The runner processes requests within one second and writes results to data/queue/integrity/outcome/. Runner log at /var/www/tmp/integrity_runner.log.
  • TNWatch reads TNAudit.db directly on every parse cycle. Any file with modified or missing status triggers an immediate alert email via the tnaudit_change rule and appears in the daily digest.
  • Compact-date backups in the format file-YYYYMMDD are not covered by the existing global exclude pattern. Add *-[0-9][0-9][0-9][0-9][0-9][0-9][0-9][0-9] to integrity_excludes.conf if you keep backups in this format inside monitored directories.
