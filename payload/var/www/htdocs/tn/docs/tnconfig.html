Foreword

TNConfig reads and caches security.conf, controls the DEVEL mode toggle, and enforces a fail-closed policy when the config file is absent or unreadable. This document covers the config file format, every section and key, the fail-closed behaviour, and the DEVEL mode credential contract.

TNConfig

Location: /var/www/htdocs/tn/data/lib/TNConfig.pm
Version: 2.1.0
Config file: /var/www/htdocs/tn/data/config/security.conf

Config File Format

security.conf is an INI-style file. Sections are denoted by [section_name] headers. Keys and values are separated by =. Comments begin with #. Quoted values are accepted but not required -- the parser strips leading and trailing single and double quotes.

ini 
[mode]
DEVEL = 0
DEVEL_PASSWORD_HASH = pbkdf2:a3f8...
DEVEL_PASSWORD_SALT = rXk2mN...

[security]
ENABLE_CSRF = 1
ENABLE_SRI = 1
ENABLE_RATE_LIMIT = 1
ENABLE_SESSION_CHECK = 1
ENABLE_ORIGIN_CHECK = 1

[session]
SESSION_LIFETIME = 7200
SESSION_COOKIE_NAME = tn_session
SESSION_SECURE = 0
SESSION_HTTPONLY = 1
SESSION_SAMESITE = Strict

[csrf]
TOKEN_LIFETIME = 3600

[rate_limit]
MAX_LOGIN_ATTEMPTS = 5
MAX_RESET_ATTEMPTS = 3
MAX_REQUESTS_PER_MINUTE = 60
LOCKOUT_DURATION = 1800

[logging]
LOG_LEVEL = info
LOG_TO_FILE = 1
LOG_TO_SYSLOG = 0

Section Reference

[mode]

DEVEL
Set to 1 to enable DEVEL mode, 0 to disable. This is the only value that changes at runtime (via the /devel/enable and /devel/disable API endpoints). See the DEVEL Mode section below.
DEVEL_PASSWORD_HASH
PBKDF2 hash of the DEVEL mode password, set by init_db.pl at install time. Format: pbkdf2: followed by hex-encoded derived key bytes. If blank, the DEVEL enable endpoint always returns 401.
DEVEL_PASSWORD_SALT
Salt for the DEVEL password hash. Must be set alongside DEVEL_PASSWORD_HASH.

[security]

Boolean flags (1 = enabled, 0 = disabled). All default to 1 if the section is absent.

ENABLE_CSRF
Enables CSRF token validation on all state-changing requests.
ENABLE_SRI
Enables server-side SRI hash verification in TNWAF before serving JS assets.
ENABLE_RATE_LIMIT
Enables per-IP rate limiting in TNWAF and per-username rate limiting for password reset.
ENABLE_SESSION_CHECK
Enables session validation in TNSecurityCheck.
ENABLE_ORIGIN_CHECK
Enables Origin/Referer header validation for non-GET requests.

These flags exist to support testing and debugging in isolated environments. Setting any of them to 0 in production removes a security layer. All five should be 1 in any deployment that handles real traffic.

[session]

SESSION_LIFETIME
Session duration in seconds. Floored at 900 (15 minutes) and capped at 86400 (24 hours) by TNAuth regardless of the value here. Default: 7200.
SESSION_COOKIE_NAME
Cookie name. Default: tn_session.
SESSION_SECURE
Sets the Secure cookie attribute. Set to 1 when the UI is served over HTTPS. Default: 0 (HTTP only, appropriate for LAN appliance use).
SESSION_HTTPONLY
Sets the HttpOnly cookie attribute. Default: 1.
SESSION_SAMESITE
Cookie SameSite attribute. Accepted values: Strict, Lax, None. An invalid value is rejected and defaults to Strict with a warning logged. Default: Strict.

[csrf]

TOKEN_LIFETIME
CSRF token window in seconds. Floored at 60 seconds by TNSecurity. TNSecurity validates the current window and the previous window, so a token fetched just before a boundary remains valid just after it. Default: 3600.

[rate_limit]

MAX_LOGIN_ATTEMPTS
Failed login attempts before account lockout. TNAuth enforces a floor of 3 and a ceiling of 10 regardless of this value. Default: 5.
MAX_RESET_ATTEMPTS
Password reset attempts before rate limit lockout. Default: 3.
MAX_REQUESTS_PER_MINUTE
Per-IP request rate limit enforced by TNWAF. Floor of 10 req/min enforced regardless of this value. Default: 60.
LOCKOUT_DURATION
Duration in seconds for both account lockouts and rate-limit blocks. TNAuth enforces a floor of 300 seconds for login lockouts. TNWAF enforces a floor of 60 seconds for IP blocks. Default: 1800 (30 minutes).

[logging]

LOG_LEVEL
Minimum log level to write. Accepted values: debug, info, warning, error, critical. Default: info.
LOG_TO_FILE
Write security events to data/logs/security.log. Default: 1.
LOG_TO_SYSLOG
Write security events to syslog. Default: 0.

Fail-Closed Behaviour

When load_config() encounters any failure -- the file is absent, unreadable, fails to open, or parses successfully but is missing the mandatory [mode] section -- it calls _fail_closed() rather than falling back to defaults.

_fail_closed() does the following:

  • Sets DEVEL = 0.
  • Sets DEVEL_PASSWORD_HASH and DEVEL_PASSWORD_SALT to empty strings, making the DEVEL enable endpoint permanently return 401.
  • Enables all five security flags.
  • Warns to STDERR with a TNConfig CRITICAL: prefix so the error appears in the web server error log.
  • Sets $CONFIG_LOADED = 1 so subsequent calls do not retry.

This means a missing or corrupted config file results in a fully locked-down system rather than a system that fails open with a known default password. The failure is loud (STDERR warning visible in httpd logs) but safe.

set_defaults() exists in the module and is used only by create_default_config() during installation. It must never be called from load_config(). Historically it set DEVEL = 1 and contained a known password. Calling it as a fallback on a missing config file would expose that credential in production.

Config Caching

The config is loaded once per process and cached in the module-level %CONFIG hash. The $CONFIG_LOADED flag prevents re-reading on every call. In the CGI model, each slowcgi worker is a separate process, so each worker loads the file once on its first request. There is no shared memory between workers.

TNConfig::reload_config() clears the cache and forces a re-read on the next call. It is not called during normal request handling but is available for use in long-running scripts that need to pick up config changes.

Runtime Write: DEVEL Toggle

The only value that changes at runtime is DEVEL. write_config() performs a targeted in-place replacement of the DEVEL = N line rather than rewriting the entire file. This preserves all comments, the [paths] section, and any operator edits.

The write is protected by flock(LOCK_EX) on both the read and write file handles to prevent interleaved writes from concurrent slowcgi workers. The sequence is: shared-lock read, close (release lock), exclusive- lock write, close (release lock).

If the DEVEL = N line is not found in the file, write_config() logs a warning and returns 0 without writing.

DEVEL Mode

DEVEL mode is a runtime bypass intended for local testing and development. When active it disables CSRF validation, SRI verification, origin checks, and session authentication. It also grants a static 'DEVEL_BYPASS' password that authenticates any username as admin without a database lookup.

The DEVEL enable endpoint (POST /devel/enable) requires a password that is verified against the PBKDF2 hash stored in security.conf [mode]. If no hash is configured, the endpoint always returns 401. The hash is set at install time by init_db.pl:

perl 
# In init_db.pl (install time only):
my ($hash, $salt) = TNSecurity::hash_password($devel_password_input);
# Written to security.conf [mode]:
# DEVEL_PASSWORD_HASH = $hash
# DEVEL_PASSWORD_SALT = $salt

router.pl enforces an additional production guard: if DEVEL mode is active and the server is bound to a non-loopback address, it returns 503 and refuses to route any request. This prevents an accidental production deploy with security bypasses active.

DEVEL mode must be 0 in any deployment that handles real traffic. The security bypasses it enables are comprehensive -- authentication, CSRF, SRI, and origin validation are all disabled simultaneously. The production guard in router.pl catches the most common accident (forgetting to disable it before go-live), but it is not a substitute for verifying the config before deployment.

Accessing Config Values

perl 
use TNConfig;

# Boolean test
if (TNConfig::is_devel_mode()) { ... }

# Read a value with fallback default
my $lifetime = TNConfig::get_config('session', 'SESSION_LIFETIME') // 7200;

# All values (for diagnostics)
my $all = TNConfig::get_all_config();

get_config() returns undef if either the section or the key is absent. Callers should always provide a // default.

Summary

  • Config is read from data/config/security.conf, loaded once per process, and cached. reload_config() forces a re-read for long-running scripts.
  • A missing, unreadable, or malformed config triggers fail-closed behaviour: DEVEL=0, all security flags on, no password set. The failure is logged to STDERR but does not crash the process.
  • set_defaults() is for installation only. It must never be called as a fallback from load_config().
  • The only runtime write is the DEVEL toggle. write_config() does a targeted in-place replacement of the DEVEL = N line with flock protection.
  • DEVEL mode requires a PBKDF2 password configured at install time. With no hash set, the enable endpoint always returns 401. The production guard in router.pl adds a second line of defence against accidental activation.
  • All floor and ceiling enforcement on rate limits and session lifetime happens in TNAuth and TNWAF, not in TNConfig. TNConfig returns the raw value from the file.
