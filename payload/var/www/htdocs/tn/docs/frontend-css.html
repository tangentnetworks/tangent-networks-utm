Foreword

This document covers the CSS architecture of the Tangent Networks dashboard. All stylesheets are served from /assets/css/. All fonts are localised and served from the same origin. There are no inline styles anywhere in the application and no external font or stylesheet dependencies.

CSS Architecture: Stylesheets and Design System

The dashboard uses a layered CSS architecture. The foundation is a compiled Tailwind CSS v4 output that provides the utility class system. On top of that sit purpose-specific stylesheets for authentication, session state, button components, documentation, and the development tools panel. Each file has a single clear responsibility and none of them overlap.

Stylesheet Inventory

tangent.css

The primary stylesheet. This is the compiled output of Tailwind CSS v4.1.7, scoped and configured for the Tangent Networks design system. It provides the full utility class set used across both index.html and view.html, including layout, spacing, typography, colour, borders, shadows, transitions, and responsive variants.

The design system built on this foundation uses blue-700 as the primary interactive colour (buttons, active states, focus rings), gray-50 and gray-900 as the light and dark page backgrounds respectively, and a consistent border-radius and shadow scale across cards, modals, and form elements.

Dark mode is implemented via Tailwind's class strategy. The dark class is toggled on the root <html> element by ts.js, and all dark mode variants throughout the markup use the dark: prefix. No media query detection is used; the theme is entirely user-controlled.

Responsive breakpoints follow Tailwind's defaults. The sidebar is off-canvas on mobile and permanently visible from the sm breakpoint upward. The sm:ml-64 offset on the main content area aligns it with the sidebar width at the same breakpoint. The navbar is fixed at the top on all screen sizes.

session.css

Manages visual states tied to session validity. Controls the #session-loading spinner shown while session.js performs its initial authentication check. Once session.js adds the session-valid class to the document root, the loading overlay is hidden and the dashboard content becomes visible. This creates a clean gate: the UI is not accessible until the session check completes, with no flash of unauthenticated content.

Also handles the devel-active body class applied when DEVEL mode is detected, which adjusts layout to accommodate the DEVEL warning banner injected by session.js.

buttons.css

Defines the button component variants used across the dashboard. The power management modals in powermgmt.js reference variants by data-variant attribute (primary, danger, safe, neutral) rather than hardcoded class strings, keeping the visual definition in CSS and the logic in JavaScript. Also covers the back-to-top button's tn-invisible visibility state used by ts.js.

docs.css

Styles the documentation pages served at /documentation.html. Provides the doc-foreword, doc-code-block, doc-code-header, doc-code-lang, doc-code-copy, callout, callout-tip, callout-warning, callout-danger, callout-note, and doc-summary classes used throughout all operator documentation files. Definition list styling for dl, dt, and dd elements is also defined here, providing the consistent term-and-description layout used across the TNAudit, TNWatch, and CGI script documents.

devel.css

Styles the developer tools panel. Never loaded in production. Injected at runtime by devel.js as the first action when DEVEL mode is confirmed, by appending a <link> element to the document head. Provides styles for the floating panel, the collapsed pill button, the expanded panel state, the info block, action buttons with their colour variants (danger, primary, success, muted), the status message area, and the DEVEL warning banner. Because this file is only present in the DOM when DEVEL mode is active, its class names have no risk of colliding with production styles.

auth.css (implied)

The authentication pages (index.html, register.html, pwreset.html) use an auth-* prefixed class namespace visible throughout auth.js: auth-form-input, auth-form-label, auth-question-wrap, auth-req--met, auth-req--unmet, auth-section-title, auth-code-block, auth-continue-btn, and others. These are scoped to the auth flow and do not appear in the dashboard shell.

Typography

All fonts are localised. No requests are made to Google Fonts, Adobe Fonts, or any external font provider. Font files are served from the same origin as the application, eliminating any third-party font loading latency or privacy exposure from font service requests.

Design Principles

There are no inline styles anywhere in the application. All visual decisions are expressed through CSS classes. This keeps the HTML clean and ensures that visual changes are made in one place. The powermgmt.js modal system is a good example: buttons are styled by data-variant attribute selectors in buttons.css rather than by class strings injected from JavaScript.

The Tailwind utility approach means that component appearance is assembled from small, composable classes at the HTML level rather than through component-specific CSS rules. The exception is the non-Tailwind stylesheets (session.css, buttons.css, docs.css, devel.css, auth.css), which use conventional class selectors for components with enough shared state or complexity that utility classes alone would be unwieldy.

The active navigation link in the sidebar uses four Tailwind classes as a compound state: border-l-4 border-blue-400 bg-blue-50 dark:bg-blue-900/20. These are managed programmatically by view.js via the ActiveLinkManager, which adds and removes them as a set defined in VIEW_SYSTEM_CONFIG.ACTIVE_LINK_CLASSES. The configuration is the single source of truth for the active style, keeping the CSS class names out of scattered JS logic.

Because tangent.css is a compiled Tailwind v4 output, it only contains the utility classes actually used in the markup at compile time. Adding new Tailwind classes to HTML or JS without recompiling will have no effect. Recompile tangent.css after any markup changes that introduce new utility classes.

Summary

  • All stylesheets are served from /assets/css/. All fonts are localised. There are no external CSS or font dependencies and no inline styles anywhere in the application.
  • tangent.css is the compiled Tailwind v4.1.7 output. Dark mode is class-based, toggled on the root element by ts.js. The primary colour is blue-700. Responsive sidebar visibility switches at the sm breakpoint.
  • session.css controls the session loading overlay and the session-valid gate that prevents unauthenticated content from flashing before the session check completes.
  • buttons.css defines button variants referenced by data-variant attribute in the power management modals, keeping visual definitions in CSS and behaviour in JavaScript.
  • docs.css provides all styles for the operator documentation pages including callouts, code blocks, definition lists, and the summary section.
  • devel.css is never loaded in production. It is injected at runtime by devel.js only when DEVEL mode is active.
  • Active sidebar link state is managed as a class set defined in VIEW_SYSTEM_CONFIG.ACTIVE_LINK_CLASSES, making the visual definition easy to change without touching the link management logic.
