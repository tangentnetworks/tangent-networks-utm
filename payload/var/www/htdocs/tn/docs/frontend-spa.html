Foreword

This document covers the client-side JavaScript layer of the Tangent Networks dashboard: the authentication system on the sign-in page, the session and CSRF infrastructure shared across both pages, the SPA view management system, the power management module, and the UI utility layer. The Perl CGI endpoints that these scripts call are documented in the CGI Scripts document. Read that first if you have not already.

View System and Authentication: JavaScript Architecture

The dashboard is served across two HTML pages. index.html is the sign-in page. view.html is the authenticated dashboard shell. Both are static HTML files. All dynamic behaviour is delivered by JavaScript fetching content from the Perl CGI backend.

The application follows a strict separation: JavaScript handles UI state, navigation, and API calls. Perl generates all HTML content served into the dashboard. JavaScript never constructs application HTML directly.

Page Structure

index.html

The sign-in page. Loads two scripts: token.js then auth.js, both deferred and SRI-protected. token.js must load before auth.js because auth.js references window.TNToken at call time. Also loads ts.js for theme toggle and UI utilities. The page is a PWA with full Apple splash screen and icon declarations in the head for all current iPhone and iPad form factors.

The login form submits to # and is intercepted by auth.js. The page also links to register.html and pwreset.html, both opening in new tabs.

view.html

The authenticated dashboard shell. The page structure is fixed: a top navbar, a collapsible sidebar, an empty <main id="main-content"> element, a back-to-top button, and a footer. All dashboard content loads dynamically into #main-content by view.js.

The sidebar is organised into three navigation sections. System covers Overview, Logs, Firewall, Services, and System Mail. Network covers External Traffic and Internal Traffic. Manage System covers System Setup and Integrity Check. Each link uses a /view/ path prefix that view.js intercepts.

Scripts load in this order: session.js in <head> deferred (runs first, guards the page), then in the body: token.js, ts.js, view.js, powermgmt.js. All are deferred and SRI-protected with SHA-384 integrity hashes.

The sidebar footer contains three power state buttons: logout (green), restart (blue), and poweroff (red). These are wired by powermgmt.js, not view.js.

Script Reference

token.js

Manages the CSRF token lifecycle. Runs as a self-contained IIFE. Exposes window.TNToken as the public API.

On initialisation it fetches a token from /cgi-bin/control.pl/api/csrf and starts a 30-minute auto-refresh interval. The fetch uses up to three retry attempts with a one-second delay between each. The refresh interval is a separate setInterval call; if a refresh is already in progress when the timer fires, the duplicate is silently skipped via an isRefreshing guard flag.

The token is held in a module-scoped variable, never in localStorage or sessionStorage. It is cleared from memory on logout via window.TNToken.clear(), which also stops the auto-refresh timer.

The public API on window.TNToken:

TNToken.fetch()
Fetches a fresh token from the server. Returns a Promise resolving to the token string.
TNToken.get()
Returns the current token string, or null if not yet loaded. Logs a warning if called before the token is ready.
TNToken.has()
Returns true if a non-empty token is loaded.
TNToken.refresh()
Fetches a new token and updates the stored value. Returns a Promise.
TNToken.clear()
Clears the token from memory and stops the auto-refresh timer. Called on logout.
TNToken.initialize()
Fetches the initial token and starts auto-refresh. Called automatically on DOM ready.
TNToken.startAutoRefresh() / TNToken.stopAutoRefresh()
Manual control of the refresh interval.

Security considerations: the token is in-memory only and is cleared on logout. Auto-refresh prevents token expiry during long sessions without requiring user interaction. The retry mechanism handles transient network failures at startup without silently leaving the form unprotected. All state-changing requests in the application gate on TNToken.has() before proceeding and refuse to submit if the token is absent.

session.js

Guards the authenticated dashboard. Runs as an immediately-invoked async function, which means it executes as soon as the script tag is evaluated, before the rest of the page finishes loading. If the session check fails for any reason, the user is redirected to /index.html before any dashboard content is visible.

On a successful session check it does the following: populates window.currentUser with username, user ID, role, session age, DEVEL mode flag, and an authenticatedAt timestamp; adds the session-valid class to the document root element; starts a 5-minute periodic recheck interval; calls updateUserInfo() to expose session data to other scripts; and conditionally activates DEVEL mode if the server reports it enabled.

The periodic recheck calls /cgi-bin/control.pl/api/session every five minutes. The server-side session TTL is 30 minutes, so this polling interval catches expiry within one cycle. If the server returns a non-ok HTTP status the user is redirected immediately. If the session is found to be unauthenticated an alert is shown before redirect. Network errors and JSON parse failures during a periodic check are logged but do not redirect, as a transient network blip should not terminate an active session. Genuine expiry will be caught on the next polling cycle.

The window.logout function sends a POST to /cgi-bin/control.pl/auth/logout with the current CSRF token from window.TNToken, then redirects to /index.html in the finally block regardless of whether the server call succeeded. This ensures the user always lands on the login page even if the logout request fails, while the CSRF token ensures the server destroys the session row in the database when the request succeeds.

DEVEL mode handling: if data.devel_mode is true, showDevelWarning() injects a dismissible warning banner into the page body and adds devel-active to the body class. loadDevelScript() then dynamically appends a <script> tag for devel.js without an SRI integrity attribute. This is intentional: in DEVEL mode the server disables SRI enforcement, so the attribute would be redundant. In production data.devel_mode is always false, so neither function ever runs.

Security considerations: the session check runs before any view content loads, making it impossible to reach the dashboard UI with an expired or invalid session. The session-valid class on the document root can be used by CSS to hide content until session validity is confirmed. The periodic recheck provides ongoing enforcement for long-lived tabs. Logout always redirects regardless of server response, preventing the user from remaining on a page after session destruction.

auth.js

Handles the full authentication UI on index.html: login, registration, and a four-step password reset flow. Runs as a self-contained IIFE. Exposes window.TNAuth.

The minimum password length constant (MIN_PASSWORD_LENGTH = 12) is the single authoritative value for client-side validation, matching the server-side minimum in TNSecurity::validate_password().

All user-facing text input is passed through a sanitize() function that HTML-encodes the six characters &, <, >, ", ', and / before the value is used anywhere. Password fields are not sanitized because the raw value is needed for hashing on the server. The sanitized username is what gets sent to the API.

Every API call checks for the CSRF token via window.TNToken.get() before building the request body. If the token is absent the form shows an error and the request is not sent. All requests use credentials: 'same-origin' to ensure the session cookie is included.

Backend error messages are mapped to user-friendly strings via a local getUserFriendlyError() table before display. Raw server error strings are never shown to the user. Unmapped errors fall through to a generic message.

The login handler validates the username length client-side and checks that the password meets the minimum length before sending anything to the server. On success it redirects to /view.html.

Registration checks username format (3 to 32 characters, letters, numbers, dash, and underscore), optional email format, password strength against five criteria (minimum length, uppercase, lowercase, digit, special character), password confirmation match, and that all five security questions are answered with distinct questions selected. The password strength indicator updates live as the user types. On success the response includes recovery codes and optionally registration tokens (issued to the first admin), which are displayed in a structured block before the user is redirected to login. The form is hidden once the success block is shown to prevent double-submission.

The password reset flow has four steps managed by a module-scoped resetState object. Step 1 collects the username and fetches security questions. Step 2 displays those questions and verifies answers. Step 3 is the alternative recovery code path. Steps 2 and 3 both converge at step 4, which accepts the new password. The server issues a short-lived HMAC-signed reset token after step 2 or step 3 succeeds; this token is stored in resetState.resetToken and submitted with the new password in step 4. The reset token has a 10-minute TTL on the server. Incorrect security answers at step 2 are subject to rate limiting enforced by the server.

The event listener setup on DOMContentLoaded detects which page is loaded by checking for the presence of specific elements before attaching handlers. The login handler is only attached when the form is present and the step elements are absent, preventing it from firing on the reset page.

Security considerations: CSRF token is required before any form submission. Input is sanitized before use. Password is never sanitized or logged. Error messages from the server are never displayed raw. The reset token flow requires server-side verification at every step, meaning a client that bypasses step 2 cannot reach step 4 without a valid server-issued token. Recovery codes are 32-character hex strings validated on the client before submission.

devel.js

The developer tools panel. Only ever loaded dynamically by session.js when the server confirms DEVEL mode is active. Never present in production. Never referenced by a static script tag in any HTML file.

On load it immediately injects devel.css by appending a <link> element to the document head. This is the only way devel.css enters the page, ensuring it is never loaded in production.

The panel is a floating element appended to document.body. In its collapsed state it shows a small pill button labelled DEV TOOLS. Clicking the pill expands the panel to show current username, role, and session age from window.currentUser, and four action buttons.

Disable DEVEL Mode
Shows a confirmation dialog, then POSTs to /cgi-bin/control.pl/devel/disable. On success shows an alert instructing the operator to restart httpd and slowcgi.
Regenerate SRI Hashes
POSTs to /cgi-bin/control.pl/devel/regenerate_sri. Triggers the server to recompute SHA-384 hashes for all static assets and update the stored values.
Show Session Info
GETs /cgi-bin/control.pl/api/session and logs the full response to the browser console as a grouped output.
Clear Console
Calls console.clear().

Keyboard shortcuts: Ctrl+Shift+D toggles the panel, Ctrl+Shift+C clears the console, Ctrl+Shift+I shows session info.

Status messages within the panel auto-dismiss after five seconds.

view.js

The SPA view management system. Handles sidebar navigation, content loading, active link state, and script lifecycle management for all nine dashboard views. Exposes window.ViewSystem.

The nine configured views are: dashboard, logs, firewall, services, mail, external, internal, manage, and integrity. Each maps a view ID to a display title and a server template path under /view/. These paths are served by the Perl backend, which returns complete HTML fragments including any view-specific inline scripts.

The system is built from four cooperating modules.

ViewState is a closure that holds the current view ID and the active link element reference. It provides get, set, and reset functions. No state is stored in the DOM or in any global variable outside the closure.

ActiveLinkManager manages the sidebar link highlighting. When a view is activated, it removes the active CSS classes from the previously active link and applies them to the new one, updating aria-current attributes accordingly. The view ID is extracted from the link href by parsing the URL and finding the path segment after /view/. If the user clicks a link for the view already loaded, the click is a no-op.

ContentLoader handles all content fetching and lifecycle management. Its loadViewContent() function follows a strict seven-step sequence on every navigation: clean up the previous view, show a loading spinner, validate the view configuration, fetch the HTML from the server, insert the HTML into #main-content, execute any scripts embedded in the new content, and dispatch a viewLoaded custom event.

The cleanup step is a full DOM replacement rather than an innerHTML clear. The entire <main> element is replaced with a fresh element carrying the same attributes. This fires a viewCleanup custom event first, giving view scripts an opportunity to clean up before they are destroyed. The replacement ensures that no event listeners, timers, or object references from the previous view survive to interfere with the next one.

Scripts embedded in the fetched HTML are not executed automatically by innerHTML assignment. _executeViewScripts() iterates over all <script> elements found in the new content and re-creates each one as a new element, which triggers execution. Inline scripts are wrapped in an immediately-invoked function expression with a try-catch so that a runtime error in one view script does not prevent other scripts in the same view from running. External script tags have their data-view attribute set and error handlers attached.

EventHandler sets up click listeners on all sidebar links matching #nav-system a[href^="/view/"]. It calls removeEventListener before addEventListener on each link to prevent duplicate handlers if the setup runs more than once.

On initialisation the system activates the dashboard view by default, then loads its content via AJAX. The AJAX loading flag (ENABLE_AJAX_LOADING) can be set to false for UI-only testing without a backend.

Fetch requests to the view templates include X-Requested-With: XMLHttpRequest so the server can distinguish AJAX requests from full page loads, and cache: 'no-cache' to prevent stale view content.

The public API on window.ViewSystem:

ViewSystem.getCurrentView()
Returns the current view ID string.
ViewSystem.loadView(viewId)
Programmatically loads a view by ID. Can be called by view scripts or from the console.
ViewSystem.debug.getState()
Returns the current view ID and full configuration object. Useful during development.
ViewSystem.debug.reset()
Resets view state. Useful for testing.

powermgmt.js

Handles the three power state actions in the sidebar footer: logout, restart, and poweroff. Runs as a self-contained IIFE.

All three actions require user confirmation through custom modal dialogs. The module replaces all use of browser-native confirm(), alert(), and prompt() with styled modal implementations that match the system visual language.

Logout shows a single confirmation modal then POSTs to /cgi-bin/control.pl/auth/logout with the CSRF token. On success it redirects to /index.html.

Restart shows two sequential confirmation modals. The first warns that network connections will be interrupted. The second asks for a final confirmation. Only after both are accepted does the power action proceed.

Poweroff shows a confirmation modal then a typed confirmation modal requiring the operator to type POWEROFF in capital letters before the action is accepted. An incorrect entry clears the input and shows an inline error rather than dismissing the modal. This two-factor confirmation prevents accidental shutdown.

The power action sequence for restart and shutdown is: POST to /cgi-bin/power_mgmt.pl with the action and CSRF token; on success, silently POST logout to destroy the session (failure here is non-fatal since the system is going down); clear the CSRF token from memory via window.TNToken.clear(); disable all buttons on the page to prevent further requests; redirect to /maintenance.html?action=restart or /maintenance.html?action=shutdown.

The CSRF token is retrieved via a polling helper that waits up to 5 seconds for window.TNToken to be populated by token.js. This handles the case where powermgmt.js initialises before the token fetch completes.

The four modal types are: spinner (non-interactive, shown during async operations), confirm (yes/no with configurable button labels and variants), alert (single OK button, wraps the confirm modal with no cancel), and prompt (text input with exact-match validation). All modals use a dark overlay and centred card, support keyboard navigation via Escape and Enter, and close if the user clicks the overlay background.

Security considerations: all destructive actions require at least one explicit confirmation. Poweroff requires typed confirmation to prevent accidental execution. The CSRF token is included with every mutating request. The session is destroyed silently before the maintenance redirect so no authenticated requests can fire from the page after the system begins shutting down. All buttons are disabled after a power action is queued to prevent duplicate requests.

ts.js

UI utility script. Initialises four independent features on DOMContentLoaded: theme toggle, sidebar toggle, user dropdown, and back-to-top button. Also initialises a mobile menu handler, though the mobile menu button is not present in the current view.html markup; the handler degrades gracefully if the element is absent.

Theme toggle switches the dark class on the root <html> element and swaps the sun and moon SVG icons. There is no persistence to localStorage; the theme resets to the default on page reload.

Sidebar toggle adds and removes sidebar--closed on the aside element and updates aria-expanded and aria-hidden accordingly.

User dropdown toggles the hidden class on the dropdown menu. It closes on outside click via a document-level listener and on link click within the menu. Event propagation is stopped on the toggle button click to prevent the document listener from immediately closing the menu that was just opened.

Back-to-top button becomes visible after 300 pixels of scroll and triggers smooth scroll to top on click.

Script Load Order and Dependencies

On index.html the load order is: token.js, then auth.js, then ts.js. The comment in the HTML explicitly documents that token.js must precede auth.js because auth.js references window.TNToken. Both are deferred, so they execute after HTML parsing in declaration order.

On view.html the load order is: session.js (in head, deferred), then token.js, ts.js, view.js, powermgmt.js (all in body, deferred). session.js is in the head to signal intent that it runs first, though deferred execution means it runs after parsing in declaration order with the body scripts. powermgmt.js polls for window.TNToken rather than assuming it is already loaded, which decouples its initialisation from the exact token fetch timing.

SRI Integrity

Every script tag on both pages carries a crossorigin="anonymous" attribute and a SHA-384 integrity hash. The browser will refuse to execute any script whose hash does not match, providing protection against CDN compromise or file tampering. devel.js is the only script loaded without an integrity attribute, and only when the server has already confirmed DEVEL mode is active, at which point SRI enforcement is disabled server-side. In production this code path never runs.

SRI hashes are regenerated via the devel/regenerate_sri endpoint whenever static assets change during development. This is exposed as a button in the DEVEL tools panel.

After any change to a static asset file in production, the corresponding SRI hash in the HTML must be updated before the file is deployed. A hash mismatch will cause the browser to block the script entirely, breaking the page for all users. Use the DEVEL mode SRI regeneration tool during development to keep hashes current, then commit both the asset and the updated HTML together.

View scripts embedded in HTML returned by the Perl backend are re-executed by view.js via DOM element replacement on every navigation. These scripts do not carry SRI attributes and are not subject to SRI checking. They are server-generated, served over the authenticated session, and covered by the same transport security as all other CGI responses.

Summary

  • Two pages: index.html for sign-in, view.html as the authenticated dashboard shell. All dashboard content loads dynamically into a single #main-content element. JavaScript handles UI and API calls; Perl generates all application HTML.
  • token.js manages the CSRF token in memory only. It auto-fetches on load, retries up to three times, refreshes every 30 minutes, and clears the token on logout. All state-changing requests gate on token availability before submitting.
  • session.js validates the session before any dashboard content loads. On success it populates window.currentUser, marks the document as session-valid, and starts a 5-minute periodic recheck. Transient network errors during periodic checks do not redirect; only confirmed session expiry does.
  • auth.js covers login, registration with five security questions, and a four-step password reset with both security answer and recovery code paths. Input is sanitized before use. CSRF token is required for every submission. Backend errors are mapped to user-friendly messages before display.
  • devel.js is only loaded dynamically when the server confirms DEVEL mode. It injects its own CSS, provides a floating tools panel with disable, SRI regeneration, session info, and console clear actions, and registers keyboard shortcuts. It is never present in production.
  • view.js intercepts all sidebar link clicks, fetches view HTML from the Perl backend, and inserts it into #main-content. On each navigation the entire main element is replaced to eliminate zombie listeners and stale references from the previous view. Embedded scripts in view HTML are re-executed via DOM element replacement.
  • powermgmt.js wires the logout, restart, and poweroff buttons. Restart requires two confirmations. Poweroff requires a typed confirmation. Power actions silently destroy the session, clear the CSRF token, disable all buttons, and redirect to the maintenance page before the system goes down.
  • All static scripts on both pages carry SHA-384 SRI integrity hashes and crossorigin="anonymous". Any file change requires a hash update before deployment.
