Foreword

TNWAF is the web application firewall and HTTP router. Every request to the system enters through router.pl, which loads TNWAF and calls route_request(). TNWAF validates the request, checks the rate limit, routes to the correct handler, serves static files with server-side SRI tamper detection, and emits the full security header set. This document covers the routing table, each serve function, the SRI tamper detection mechanism, the CSP policy, rate limiting, and the DEVEL production guard. SRI hash values in TNWAF.pm are written at install time by TN_SUBSTITUTE.sh and must not be edited by hand.

TNWAF

Location: /var/www/htdocs/tn/data/lib/TNWAF.pm
Version: 1.0.2
Entry point: TNWAF::route_request()

Request Flow

Every incoming HTTP request follows this path:

  1. router.pl receives the request from slowcgi and calls TNWAF::route_request().
  2. URI validation: checks length limit (2048 bytes) and pattern blocklist.
  3. Method validation: allows GET, POST, HEAD, OPTIONS only.
  4. Rate limit check: per-IP request counting against a 60-second sliding window.
  5. Access log write.
  6. Route dispatch to the appropriate serve function or CGI proxy.

If any step fails, error_response() is called and the function returns 0. If routing succeeds, serve_file() or proxy_to_cgi() handles the rest and the process exits.

CGI Proxy

proxy_to_cgi($script, $path_info) resolves the script path as BASE_DIR/cgi-bin/SCRIPT.pl, verifies the file exists and is executable, sets $ENV{PATH_INFO}, and calls exec($script_path). The current router.pl process is replaced by the CGI script. After exec, the CGI script is a fresh process and must set up its own pledge and unveil.

router.pl itself currently does not implement pledge or unveil. This is a known gap. The WAF calls system(logger) in log_security() and exec() in proxy_to_cgi(), both of which require proc exec in the pledge promises. A future router.pl pledge block should use pledge("stdio rpath wpath cpath flock proc exec") with unveil covering the BASE_DIR tree, waf log directories, /tmp, /usr/bin/logger, cgi-bin scripts, and Perl library paths.

Serve Functions

serve_html

Serves BASE_DIR/view.html with CSP headers included (include_csp = 1). This is the main SPA entry point.

serve_static_html

Validates the filename against /^([\w.-]+\.html)$/, then serves the file from BASE_DIR with CSP headers.

serve_asset

Routes to BASE_DIR/assets/TYPE/FILE. Validates both the type component (matched by the routing regex to js, css, fonts, img, or images) and the file path against /^([\w./-]+)$/. Rejects any path containing ... Content type is determined by file extension via get_content_type(). No CSP headers for assets (they are sub-resources, not documents).

serve_data

Routes to BASE_DIR/data/FILE. Applies a denylist of top-level subdirectory names that must not be served:

text 
Blocked: keys  config  lib  lib-bak  scripts  session  run  queue

The db/ directory is partially blocked: any path matching db/NAME.db, db/NAME.db-wal, or db/NAME.db-shm is blocked (SQLite database files). Subdirectories of db/ such as db/GeoIP/ and db/pf/ are served normally because they contain UI-readable lookup data, not database files.

serve_doc_fragment

Routes to BASE_DIR/docs/FRAGMENT.html. The filename is validated against /^([\w-]+\.html)$/. Serves with CSP headers. These are the HTML fragment files loaded by the documentation SPA.

serve_root_file

Serves favicon, apple touch icon, manifest.json, and sw.js from BASE_DIR root. Validates the filename against /^([\w.-]+)$/. No CSP headers for these files.

Server-Side SRI Tamper Detection

Before serving any JavaScript file, serve_file() computes the SHA-384 of the file content and compares it against the known-good hash in %CONFIG{sri_hashes}. If the hashes do not match, the file is refused and an HTTP 500 is returned. The browser never receives the compromised file regardless of what the HTML integrity= attribute says.

perl 
my $actual   = 'sha384-' . encode_base64(sha384($content), '');
my $expected = $CONFIG{sri_hashes}{$relative_path};

unless ($actual eq $expected) {
    log_security('SRI_TAMPER', "Hash mismatch for $relative_path ...");
    return error_response(500, 'Internal Server Error');
}

The sri_hashes table in %CONFIG maps relative asset paths (e.g. /assets/js/auth.js) to their expected sha384- hash strings. The table is populated at install time by TN_SUBSTITUTE.sh, which computes the SHA-384 of every asset file, writes the hashes into the sri_hashes block in TNWAF.pm, and updates all integrity= attributes in the HTML files in a single pass. No manual hash generation step is required after deployment; re-run TN_SUBSTITUTE.sh after any asset change.

When the serve function returns an unexpected 500 on a JS asset, the first thing to check is whether the asset file was modified after the last TN_SUBSTITUTE.sh run. Check the WAF security log at data/logs/waf/security.log for SRI_TAMPER events.

Security Headers

print_headers($content_type, $include_csp, $file_path) emits the security header set for all file responses. It is the TNWAF counterpart to TNSecurity::print_secure_headers(), which covers JSON API responses.

Standard headers emitted on every response:

X-Frame-Options: DENY
Belt-and-suspenders alongside CSP frame-ancestors 'none'.
X-Content-Type-Options: nosniff
Prevents MIME sniffing.
X-XSS-Protection: 1; mode=block
Legacy XSS auditor signal for older browsers.
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: geolocation=(), microphone=(), camera=(), payment=()
Strict-Transport-Security: max-age=31536000; includeSubDomains
Effective over HTTPS only. Harmless over HTTP.
Cache-Control: no-cache, no-store, must-revalidate, private

When $include_csp = 1 (HTML documents only), the full Content-Security-Policy header is added via build_csp().

When serving a JS file that has a known SRI hash, an X-Content-Digest header is added containing the expected hash. This is informational only -- it is not enforced by browsers.

Content Security Policy

The CSP is assembled from the %CONFIG{csp} hash by build_csp(), which sorts directive names and joins them with ; . The policy applied to all HTML documents:

text 
default-src 'self';
script-src 'self';
style-src 'self';
img-src 'self' data:;
font-src 'self';
connect-src 'self';
object-src 'none';
form-action 'self';
base-uri 'self';
frame-ancestors 'none';
upgrade-insecure-requests

Key policy decisions:

script-src 'self'
No unsafe-inline or unsafe-eval. All JavaScript is served as external files. Server-side tamper detection in serve_file() ensures only files whose SHA-384 matches the sri_hashes table are served; hashes are written into that table by TN_SUBSTITUTE.sh at install time.
img-src 'self' data:
The data: allowance is required for inline SVG data URIs generated by Tailwind/Flowbite CSS (custom selects, checkboxes, form elements).
object-src 'none'
Blocks Flash and all plugin content entirely.
frame-ancestors 'none'
Prevents the application being embedded in an iframe on any origin.
upgrade-insecure-requests
Instructs browsers to upgrade HTTP sub-resource requests to HTTPS. Effective when HTTPS is in use; harmless otherwise.

Rate Limiting

TNWAF maintains two in-memory hashes: %RATE_LIMITS (per-IP request counters) and %IP_BLOCKS (blocked IPs with expiry epochs). These are process-local and reset on every request under the CGI model -- slowcgi spawns a fresh process for each request. The rate limiting is therefore effective only within a single CGI process lifetime, which is a single request.

In the slowcgi CGI model, each request is a separate process. The in-memory rate limit counters in TNWAF do not persist between requests. For persistent per-IP rate limiting, see the rate_limits table in auth.db, which is used by TNAuth for password reset throttling. The TNWAF rate limiter provides per-request pattern validation and basic flood detection within a single connection window only.

Rate limit parameters are read from security.conf [rate_limit]: MAX_REQUESTS_PER_MINUTE (floor 10) and LOCKOUT_DURATION (floor 60 seconds).

Request Validation

validate_request($uri) applies two checks:

URI length
Rejected if longer than 2048 bytes.
Pattern blocklist
The following patterns trigger an immediate block and a security log entry: .. (path traversal), <>'" (injection characters), union.*select (SQL injection), script.*src (XSS), javascript: (protocol injection), data:text/html (data URI XSS).

Logging

Three log files are written under BASE_DIR/data/logs/waf/:

waf/access.log
One line per request: timestamp, IP, method, URI, User-Agent.
waf/security.log
Security events: rate limit blocks, pattern blocks, SRI tamper detections. Also written to syslog via logger -t tnwaf -p security.warning.
waf/error.log
HTTP error responses (4xx, 5xx). Also mirrored to data/logs/httpd/httpd_error.log for the httpd error log viewer in the UI.

Log directories are created automatically if absent. Log writes are not flock-protected in TNWAF (unlike TNSecurity's security.log). Under the CGI model each request is a single process with a single log write, so interleaving is unlikely, but it is possible under very high concurrent load.

DEVEL Production Guard

router.pl checks DEVEL mode status before calling TNWAF::route_request(). If DEVEL mode is active and the server is bound to a non-loopback address (SERVER_ADDR or HTTP_HOST is not 127.0.0.1, localhost, or ::1), router.pl returns HTTP 503 and refuses to route the request:

perl 
if (TNConfig::is_devel_mode()) {
    my $addr = $ENV{SERVER_ADDR} // $ENV{HTTP_HOST} // '';
    $addr =~ s/:\d+$//;

    my $is_loopback = ($addr eq '127.0.0.1' || $addr eq 'localhost'
                    || $addr eq '::1'        || $addr =~ /^127\./);

    unless ($is_loopback) {
        # return 503, log, exit
    }
}

This guard catches the most common deployment accident: enabling DEVEL mode for local testing and then forgetting to disable it before making the system accessible on the LAN interface. Any request arriving on the LAN IP (192.168.122.25 in this documentation's reference configuration) while DEVEL is active will receive a 503.

The production guard is a safety net, not a substitute for correct configuration. Always verify that DEVEL = 0 in security.conf before any deployment. The 503 response body explicitly states that the system is in development mode, which is information disclosure. Disable DEVEL mode before exposing the system to any untrusted network.

Summary

  • Every request enters through router.pl, which calls TNWAF::route_request(). TNWAF validates, rate-checks, logs, then routes to a serve function or exec's a CGI script.
  • Server-side SRI tamper detection in serve_file() rejects any JS asset whose SHA-384 does not match the sri_hashes table in %CONFIG. A tampered file returns 500 before the browser sees it. Hashes in the table are written by TN_SUBSTITUTE.sh at install time.
  • The CSP applied to all HTML documents has no unsafe-inline or unsafe-eval. All script is served as external files; serve_file() enforces tamper detection server-side. data: is allowed only in img-src for Tailwind/Flowbite SVG data URIs.
  • TNWAF in-memory rate limiting is process-local and does not persist across requests under the CGI model. It provides per-request flood detection only.
  • router.pl refuses to route any request to a non-loopback address while DEVEL mode is active, returning 503.
  • router.pl does not currently implement pledge or unveil. This is a known gap. A full pledge block requires proc exec in addition to the standard file I/O promises, because TNWAF calls both exec() and system(logger).
