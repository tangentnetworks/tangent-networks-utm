Foreword

TNSecurityCheck is the one-line security gate that every backend CGI script calls at the top of its handler. A single security_check($level) call enforces all active protection layers appropriate to the endpoint and returns a session hashref on success or exits with an error response on failure. This document covers the three protection levels, each validation layer, the DEVEL mode bypass, and the pledge requirements that each calling CGI must satisfy.

TNSecurityCheck

Location: /var/www/htdocs/tn/data/lib/TNSecurityCheck.pm
Version: 2.0.0
Exports: security_check (auto-exported on use)

Usage

Every backend CGI script calls security_check() as its first action after loading modules. The call either returns a session hashref that the script can use immediately, or it exits -- it never returns on failure.

perl 
use TNSecurityCheck;

# Public endpoint (origin check only, no authentication)
my $session = security_check('standard');

# Authenticated endpoint (session + POST + CSRF)
my $session = security_check('protected');

# Admin-only endpoint (session + POST + CSRF + admin role)
my $session = security_check('restricted');

# $session hashref fields:
#   user_id   -- undef for standard level
#   username  -- 'anonymous' for standard level
#   role      -- 'public' for standard level, 'user' or 'admin' otherwise
#   devel_mode -- 1 if DEVEL mode is active

Protection Levels

standard

Suitable for public endpoints that serve data to any authenticated browser session, or to unauthenticated browsers that pass origin validation. Runs two checks: origin validation and download protection. Does not check for a session cookie. Returns an anonymous session hashref (role = 'public').

protected

Suitable for endpoints that require a logged-in user. Runs all six check layers in order. Returns the authenticated user's session hashref on success.

restricted

Identical to protected but adds a role check in layer 6: the session role must be 'admin'. Suitable for admin-only endpoints. Returns the admin session hashref on success.

Check Layers

The six layers are applied in a fixed order. Any failure exits immediately with an error response -- later layers are not reached.

Layer 1: Session Validation

Reads the tn_session cookie from $ENV{HTTP_COOKIE}, extracts the signed value with a regex that limits the character set to [a-zA-Z0-9_\-=.], calls TNSecurity::verify_session_id() to verify the HMAC signature and extract the raw session ID, then calls TNAuth::validate_session() to check the database.

Returns the session hashref from TNAuth on success. Returns undef (triggering a 401) on any failure: no cookie header, cookie not found, signature invalid, session not in database, session expired.

Only runs for protected and restricted levels.

Layer 2: Origin Validation

Checks HTTP_ORIGIN and HTTP_REFERER against HTTP_HOST.

  • If both Origin and Referer are absent, the request is allowed (same-origin requests from most browsers do not send these headers for non-CORS requests).
  • If Origin is present: allowed if it matches http(s)://HTTP_HOST.
  • If Referer is present: allowed if it starts with http(s)://HTTP_HOST/.

Runs for all three levels.

Layer 3: Download Protection

Rejects requests with Accept headers that suggest a download tool or an attempt to retrieve raw script source:

  • application/octet-stream
  • text/x-perl
  • application/x-perl
  • text/plain.*download

Returns HTTP 550 on a match. Runs for all three levels.

Layer 4: Method Validation

Requires REQUEST_METHOD = POST. Returns 405 for any other method.

Runs for protected and restricted levels only.

Layer 5: CSRF Validation

Reads the JSON POST body, extracts csrf_token, and calls TNSecurity::validate_csrf_token(). Returns 403 on failure.

Runs for protected and restricted levels only.

Layer 6: Role Check

Verifies that $session->{role} eq 'admin'. Returns 403 for any other role.

Runs for restricted level only.

DEVEL Mode Bypass

When TNConfig::is_devel_mode() returns true, security_check() returns immediately with a synthetic admin session without running any check layer:

perl 
{
    user_id   => 'devel',
    username  => 'developer',
    role      => 'admin',
    devel_mode => 1,
}

This bypass applies regardless of the level requested. A request to a restricted endpoint in DEVEL mode is granted admin access without a cookie, without a CSRF token, and without any database lookup.

The DEVEL bypass is comprehensive. All six check layers are skipped simultaneously. DEVEL mode must be disabled before any deployment that handles real traffic. The production guard in router.pl (which refuses to serve non-loopback traffic while DEVEL is active) provides a second line of defence, but the config must be correct for it to trigger.

Error Responses

_error_response($status, $message, $log_message) logs the event via TNSecurity::log_security_event('warning', ...), prints the HTTP status line, calls TNSecurity::print_secure_headers() for the full security header set, prints a JSON error body, and calls exit. It does not return.

Error codes returned by each layer:

401
Session absent, invalid, or expired (layer 1).
403
Origin mismatch (layer 2), CSRF token invalid (layer 5), insufficient role (layer 6).
405
Request method is not POST (layer 4).
550
Download attempt blocked (layer 3). This is a non-standard code used internally; it does not appear in standard HTTP status registries.

Pledge Requirements

TNSecurityCheck calls TNAuth::validate_session (database read), TNSecurity::verify_session_id (key file read), and TNSecurity::log_security_event (log file write). Any CGI script that calls security_check() must therefore have the following pledge promises active before the call:

text 
stdio rpath wpath cpath flock

And the following paths must be unveiled before the pledge lock is applied:

text 
data/db        rwc   (SQLite database)
data/keys      r     (HMAC and session key files)
data/logs      rwc   (security.log and waf logs)
/dev/urandom   r     (salt generation)
Perl lib paths r     (module loading)

Pledge and unveil are the calling CGI script's responsibility. TNSecurityCheck does not issue any pledge or unveil calls itself. A backend CGI script that calls security_check() without first setting up pledge and unveil will function correctly, but runs without the kernel- level sandbox protection. All CGI scripts should implement their own pledge block modelled on the one in control.pl. See the backend CGI documents for script-specific pledge blocks.

All probe-and-unveil path existence checks must happen before the first unveil() call. Once any unveil is issued, the kernel begins restricting filesystem visibility. A -d test on an un-unveiled path will return false even if the directory exists. Resolve all conditional paths while the filesystem is unrestricted, then issue unveil calls in a single pass.

Comparison with control.pl

The authentication CGI script control.pl does not use TNSecurityCheck. It implements its own session and CSRF validation directly via require_session() and require_csrf_token(). This is intentional: control.pl handles the unauthenticated endpoints (login, registration, password reset) where TNSecurityCheck's session requirement would block legitimate use.

All other CGI scripts (the backend handlers for dashboard, firewall, content filter, DNS, flow accounting, system, and integrity endpoints) use TNSecurityCheck. They call security_check('protected') or security_check('restricted') depending on whether admin access is required.

Summary

  • Three protection levels: standard (origin + download only), protected (session + origin + POST + CSRF), restricted (all of protected plus admin role check).
  • All six check layers run in fixed order. Any failure exits immediately with a JSON error response -- the function never returns on failure.
  • DEVEL mode bypasses all layers and returns a synthetic admin session unconditionally. The router.pl production guard is the only barrier against DEVEL mode on a live system.
  • Pledge and unveil are the calling CGI's responsibility. TNSecurityCheck issues no pledge calls. Required promises: stdio rpath wpath cpath flock. Required unveil paths: data/db (rwc), data/keys (r), data/logs (rwc), /dev/urandom (r), Perl libs (r).
  • control.pl uses its own session and CSRF validation directly. All other CGI scripts use TNSecurityCheck.
