Foreword

This document covers the boot orchestration layer of Tangent Networks: specifically /etc/rc.local, the hook injected into /etc/rc that runs tangent_logrotate.sh before any daemon starts, and the log rotation framework itself. These three components together define how the system comes up, in what order services start, what happens when something fails, and how logs are managed across reboots and sysupgrades. Read this document when the dashboard is dark after a reboot, when a service is missing from the Services panel, or when you need to understand why a particular service starts when it does.

The log rotation framework is covered in passing references here. A dedicated document will cover tangent_logrotate.sh and the associated scripts in full detail.

System Orchestration

The Two Boot Files

OpenBSD's standard boot sequence ends with /etc/rc, which starts the base system daemons via start_daemon. After start_daemon completes, /etc/rc calls /etc/rc.local if it exists. Everything Tangent starts — Snort, ClamAV, pmacct, SSLproxy, e2guardian, all 17 WebUI runners — starts from /etc/rc.local.

One additional hook is injected into /etc/rc itself, placed after reorder_libs and before the first start_daemon call. This hook runs tangent_logrotate.sh before any daemon starts, including before the base system daemons, so that log files are rotated into a clean state before anything begins writing to them. The position is deliberate: reorder_libs must complete first because it makes system binaries functional, and the hook must run before start_daemon because daemons begin writing logs immediately.

This hook is self-healing. tangent_logrotate.sh checks for its own presence in /etc/rc on every invocation. If a sysupgrade has replaced /etc/rc and removed the hook, the script detects this, backs up the new /etc/rc, and reinjects the hook automatically. The backup is written to /usr/local/share/tangent/rc.backup.<timestamp>.

rc.local Structure

/etc/rc.local opens with set -euo pipefail. This means any unhandled command failure causes the script to exit immediately, any reference to an unset variable is a fatal error, and any failure in a pipeline propagates correctly. The consequence is that a bug or missing file early in the script will cause rc.local to exit before reaching the WebUI runners at the end. The dashboard will be dark. The first diagnostic step is always:

sh 
tail -50 /var/www/htdocs/tn/data/logs/bootlog/rc.local.log

The very first thing rc.local does after setting up the log directory is redirect all stdout and stderr to rc.local.log with exec > "$RCLOGFILE" 2>&1. Every subsequent log message, every service output, and every error goes to that file. It is a complete record of the boot session.

The second thing rc.local does is truncate all PID files under data/ to zero bytes. This clears stale PID state from the previous boot session. A non-empty PID file after boot always means the service started successfully in the current session.

The third thing is housekeeping in /var/www/tmp: dated control log files (control-YYYY-MM-DD.log) are deleted and all other log files in that directory are truncated to zero bytes.

The log() Function

rc.local defines a log() function used throughout the file. It takes four arguments: service name, log level, message, and an optional target log file (defaults to rc.local.log). Most service sections pass their own dedicated log file as the fourth argument, so service-specific messages appear in both rc.local.log (via the exec redirect) and in the service's own log file simultaneously.

Boot Sequence

The following is the exact order in which rc.local starts services. Order matters: ClamAV must be running before e2guardian can scan, pmacct MFS paths must exist before pmacctd writes to them, and the WebUI runners must start last because they depend on everything else being up.

1. PID file truncation and /var/www/tmp housekeeping
All data/**/*.pid files truncated to zero bytes. Dated control log files deleted. Other tmp logs truncated.
2. Snort IDS
Started on %%INT_IF%% with config /etc/snort/snort.conf. Runs as _snort:_snort. Logs via logger -t snort_ids. rc.local waits up to 15 seconds for snort_%%INT_IF%%.pid to appear before declaring success or failure. PID file permissions are fixed to 0644 after write.
3. Snort IPS
Started in divert-packet mode (-Q) with config /etc/snort/snortinline.conf. Logs via logger -t snort_ips. rc.local waits up to 10 seconds for snort_.pid.
4. snortsentry
Started with -f /etc/snort/snortsentry.conf. rc.local waits up to 5 seconds for its PID file. snortsentry reads data/logs/snort/alert.log, the alert file shared by both Snort instances, and manages the <snort_block> PF table.
5. e2guardian
Web content filter. Creates data/tmp/e2guardian/ if missing, sets ownership to _e2guardian:_clamav, then starts. e2guardian requires ClamAV's socket, but ClamAV starts after it. e2guardian's own startup handles the socket wait internally.
6. collectd
Removes any stale collectd.sock from the previous session before starting, so collectd creates a fresh socket. rc.local sleeps 2 seconds after starting collectd to ensure the socket exists before the dashboard stats runner first queries it.
7. p3scan
Transparent POP3/POP3S scanning proxy. Creates data/run/p3scan/, sets ownership to _p3scan:wheel, removes any stale PID file, then starts.
8. ClamAV (clamd)
The most time-consuming boot step. rc.local starts clamd and then waits in a polling loop for up to 90 seconds for clamd.socket to appear. Progress dots are logged every 10 seconds. This wait is intentional: freshclam only starts after the socket is confirmed, and e2guardian's scan path depends on the socket. On cold hardware with large signature databases this wait is normal and expected. It is not a hang.
9. pmacct — MFS instances
Two pmacctd instances start immediately: ext_if_json_mfs and int_if_json_mfs. Both write to the MFS paths (data/pipes/pmacct/). The MFS files are pre-created with correct permissions before pmacctd starts. A background permission fixer loop runs every 8 seconds to ensure newly-written files remain 0644.
10. pmacct — disk archive instance (delayed)
The third pmacctd instance (ext_if_json_log) starts in a background subshell that calculates the time to the next quarter-hour boundary and sleeps until then. This delay is up to 15 minutes. The instance writes quarter-hour JSON files to data/network/pmacct/ext/. Its PID file will be empty until it starts. At the same boot run, rc.local also merges yesterday's quarter-hour files into a single YYYY-MM-DD.json and purges files older than 7 days.
11. sockd (Dante SOCKS proxy)
Started with PID file at data/run/sockd/sockd.pid.
12. spamd
Spam deferral daemon. Flags: -L -d -x -u _spamdaemon. PID file at data/run/spamd/spamd.pid.
13. smtp-gated
Transparent SMTP/SMTPS/submission scanning proxy.
14. SSLproxy
Transparent SSL/TLS interception proxy. Listens on lo:8081 (HTTP) and lo:8443 (HTTPS).
15. imspector
Instant messaging transparent proxy. Creates /tmp/imspector/ with _imspector:_imspector ownership before starting.
16. pf_mirror_sync.sh
Synchronises PF assets for the WebUI. Runs synchronously (no background &) before the WebUI runners start.
17. tcpdump on pflog1
Starts tcpdump reading from pflog1 and writing to data/logs/pf/pflog1.log (MFS). The MFS directory and file are created if missing and permissions set to www:wheel 0644. Any stale tcpdump process is killed before starting a new one. If pflog1 does not exist, this step logs an error and continues; the PF log panel in the dashboard will be dark.
18. process_monitor.pl (delayed)
Scheduled in a background subshell to run at 1 minute past the next 15-minute boundary, up to 16 minutes after boot. It writes a plain-text service status report to data/logs/bootlog/services.log. The delay avoids running while services are still initialising.
19. start_service() — 16 WebUI runners
The start_service() function is defined here and then called 16 times in sequence. See the WebUI Runners section below.

start_service()

The start_service() function defined in rc.local is the standard launcher for all WebUI runners. It takes three arguments: the script path, a display name used in log messages, and an optional PID file basename (defaults to the display name). All PID files are written to data/run/webui/.

The function validates that the script exists and is executable before doing anything. It then ensures the PID directory exists with the correct ownership (www:www 0750) using install -d, which is idempotent and safe on every boot. It checks for a live existing process via the PID file; if the process is already running it logs and returns without starting a duplicate. If the PID file is stale it removes it and proceeds. It launches the script with stdin, stdout, and stderr all redirected to /dev/null, backgrounds it, waits 200ms, and then verifies the process is still alive before writing its PID to the file. If the process has already exited by then, it logs a failure.

The 200ms verification window is intentional. Runner scripts that fail immediately due to a missing dependency would otherwise produce a PID file for a process that no longer exists. The verification catches this class of failure at boot time rather than silently producing a stale PID.

tangent_logrotate.sh

tangent_logrotate.sh is a deterministic, self-healing log rotation framework. It runs in two contexts: injected into /etc/rc before any daemon starts at boot, and every 10 minutes via cron (with a full daily run at 01:00). It is written in ksh with set -eu.

Self-healing rc integration

The first thing the script does on every invocation is check for its own hook in /etc/rc. It looks for both the marker comment and the script path. If either is missing, it backs up the current /etc/rc to /usr/local/share/tangent/rc.backup.<timestamp>, strips any incomplete Tangent block using awk, injects a clean hook block immediately before the first start_daemon line, and replaces /etc/rc atomically using mv -f. The patch target is the first start_daemon invocation, which on a standard OpenBSD system is the networking daemon block. This positions the log rotation hook after reorder_libs (which makes binaries functional) and before any daemon that might write logs.

Concurrency protection

After the rc check the script acquires a PID lock at data/run/tangent_logrotate.pid. If a previous instance is still running it exits silently. Stale PID files (where the process is gone) are removed and the lock is taken. The PID file is cleaned up via trap on any exit.

Triple verification

The script uses three independent checks to determine whether rotation has already happened today, preventing duplicate rotation if the script is called multiple times in the same calendar day.

First it checks the stamp file at data/logs/bootlog/.rotation_stamp. If the stamp contains today's date, the script exits immediately. Second, if the stamp is absent or stale, it looks for at least two of three sample archived files that would only exist if rotation had already run today (yesterday's doas log, yesterday's messages log, and /var/log/authlog.0). If two or more are found, the stamp file is updated and the script exits. Third, if neither check passes, rotation proceeds.

Rotation procedure

Rotation covers three categories of log files. System logs are /var/log/authlog, /var/log/maillog, /var/log/secure, and /var/cron/log. Custom system logs are data/logs/system/messages and data/logs/system/daemon. Application logs cover all service logs under data/logs/, 26 files in total including all WAF logs, all proxy logs, and all security logs.

For each file the script skips it if it does not exist or is zero bytes. For non-empty files it copies the current log to a .0 temporary file, truncates the original to zero bytes (so the running daemon continues writing to the same inode without interruption), and then calls logrotate_rename.sh to rename the .0 file to the date-stamped archive format (name_YYYY-MM-DD.log).

After rotating httpd logs, the script sends USR1 to httpd via pkill -USR1. This causes httpd to reopen its log file descriptors. Because TNWAF.pm writes httpd logs directly rather than through syslog, this signal is required for the rotation to take effect cleanly.

Purge cycle

On every run, before the rotation check, the script scans all date-stamped archive files under data/logs/ and deletes any whose embedded date is older than the retention window (default 7 days). The comparison is a string comparison on the YYYY-MM-DD portion of the filename, which is lexicographically equivalent to a date comparison.

Metadata file

After a successful rotation the script writes a machine-readable metadata file to data/logs/.rotation_meta containing the last rotation date, total archive count, total archive size in KB, lists of skipped empty and missing files, and the configured retention period. This file is read by the dashboard log management panel.

Atomic success marking

The stamp file is only written if the rotation completed with zero errors. If any file failed to copy or truncate, the stamp file is not updated and the script exits with status 1. The next cron invocation will retry the full rotation. This ensures a partial rotation does not silently mark the day as complete.

Boot Warnings

Critical — softraid crypto and unattended reboots. If the root filesystem is on a softraid crypto volume, the passphrase is entered at the boot loader before the kernel starts. A WebUI-initiated reboot will halt at the boot loader passphrase prompt with no network, no dashboard, and no remote recovery. Encrypted swap (OpenBSD default) is not the same as softraid crypto and reinitialises automatically. Do not confuse the two.

Critical — set -euo pipefail in rc.local. Any unhandled command failure before the start_service block causes rc.local to exit early. All WebUI runners fail to start. The system is up but the dashboard is dark. First diagnostic step: tail -50 /var/www/htdocs/tn/data/logs/bootlog/rc.local.log

MFS paths are empty after every reboot. data/logs/pf/ (128MB) and data/pipes/pmacct/ (64MB) are swap-backed memory filesystems. Their contents are lost on every reboot including WebUI-initiated reboots. rc.local pre-creates the required files before starting tcpdump and pmacctd. If the MFS mount entries are missing from /etc/fstab, these paths become regular disk directories with no size limit; tcpdump and pmacctd will still write to them but the disk will eventually fill.

ClamAV startup delay — up to 90 seconds. rc.local waits for clamd.socket before proceeding to the pmacct section. On cold hardware with a large signature database this wait is normal. The boot log will show periodic dot entries every 10 seconds during the wait. This is not a hang.

pmacct ext_if_json_log starts up to 15 minutes after boot. The disk archive pmacctd instance waits for the next quarter-hour boundary. An empty PID file at data/run/pmacct/pmacct-ext_if_json_log.pid immediately after boot is correct and expected.

process_monitor.pl starts up to 16 minutes after boot. It is delayed to 1 minute past the next 15-minute boundary. The Services panel may show stale or empty data for up to 16 minutes after boot.

pflog1 requires /etc/hostname.pflog1. If this file is missing or does not contain up, pflig1 never comes up. tcpdump will fail to start on it. The PF log panel in the dashboard will be dark. This is not logged as a fatal error; rc.local continues past it.

tangent_logrotate.sh self-heals /etc/rc after sysupgrade. The rc integration check runs on every invocation. After a sysupgrade that overwrites /etc/rc, the next run of tangent_logrotate.sh (either from cron or the next boot) will detect the missing hook and reinject it automatically. The pre-upgrade /etc/rc is backed up before any modification.

WebUI runners do not receive a clean shutdown signal. Tangent daemons are started directly by rc.local, not via pkg_scripts. They are not stopped by the rc shutdown path and receive no SIGTERM on halt or reboot; init kills them. For file-writing loop daemons this is safe in practice. The PID files are truncated at the start of the next boot regardless.

BPF buffer is a critical silent dependency. sysctl.conf sets net.bpf.bufsize=2097152 (2MB). The default of 32KB causes silent packet drops under load in both pflog1 and pmacct. If sysctl.conf is overwritten by a sysupgrade, tcpdump and pmacctd will appear to work normally but will drop packets silently. Verify with sysctl net.bpf.bufsize.

Diagnostics

sh 
# Full boot log — start here when anything is wrong after reboot
tail -100 /var/www/htdocs/tn/data/logs/bootlog/rc.local.log

# Check rc integration hook is present

# Check rotation stamp
cat /var/www/htdocs/tn/data/logs/bootlog/.rotation_stamp

# Check rotation metadata
cat /var/www/htdocs/tn/data/logs/.rotation_meta

# Verify BPF buffer

# Check pflog1 interface exists

# Check hostname.pflog1 contents
cat /etc/hostname.pflog1

# Force log rotation now
/usr/local/sbin/tangent_logrotate.sh

# Re-run all WebUI runners after rc.local failure
# (safe to call start_service inline — idempotent if already running)
sh /etc/rc.local

Summary

  • rc.local is the single orchestration point for everything Tangent starts. It runs with set -euo pipefail: any early failure stops all subsequent services. The boot log at data/logs/bootlog/rc.local.log is always the first diagnostic step.
  • The log rotation hook in /etc/rc runs before any daemon starts, positioned after reorder_libs and before the first start_daemon call. It is self-healing and will reinject itself after a sysupgrade.
  • ClamAV startup takes up to 90 seconds. pmacct's disk archive instance starts up to 15 minutes after boot. process_monitor.pl starts up to 16 minutes after boot. All three delays are by design.
  • start_service() guards against duplicate starts, stale PID files, and immediate-exit failures. It is idempotent: calling rc.local again on a running system is safe.
  • Two MFS filesystems are empty after every reboot. Three sysctl and login.conf values are silent hard dependencies. All five are checked and set by install.sh but can be lost after a sysupgrade.
